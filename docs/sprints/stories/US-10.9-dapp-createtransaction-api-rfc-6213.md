## Status refresh — 2026-07-15

> Synced from GitHub Projects board #2 ("SubWallet.App – Development"): issue #4713 is **In Review** there, so this story moves `backlog` → `review` (sprint `sprint-2026-W28`). Only status/sprint changed; Goal, AC and reasoning below are untouched. The board is the live source for workflow state.

## Goal

Expose the Polkadot-JS **`createTransaction`** dApp API (RFC #6213) on the Substrate
inject interface, so a dApp can hand the wallet a transaction *call* and let the
wallet build, present, sign, and submit it — rather than constructing the full
extrinsic itself. This raises SubWallet to the newest Polkadot dApp-interaction
standard and lets dApps delegate extrinsic assembly to the wallet.

## Background

Today a Substrate dApp builds the whole extrinsic and asks the wallet to
`signPayload` (US-10.2). [Polkadot-JS RFC #6213](https://github.com/polkadot-js/extension)
proposes a higher-level `createTransaction` injected method: the dApp supplies the
call (and intent) and the *wallet* assembles, fee-estimates, displays, signs, and
submits the transaction — a cleaner contract that moves extrinsic-construction
responsibility into the wallet where the account/fee context lives. This is the one
**forward-looking** capability in EPIC-10: PRD marks [FR-102](../../PRD.md#functional-requirements) as
`📋 planned`, so this story is authored forward, not retroactively.

`createTransaction` is added to the Substrate inject API (US-10.2) and gated by the
same per-origin authorization (US-10.7). The build/sign/submit step routes through
`RequestService` (AD-21) for approval and then into the transaction lifecycle owned
by [EPIC-8](../epics/EPIC-8.md) — this story exposes and wires the dApp-facing API surface;
it does not reimplement transaction submission.

Materializes [FR-102](../../PRD.md#functional-requirements). **Forward-looking / planned** — not yet shipped.
Tracked by [#4711](https://github.com/Koniverse/SubWallet-Extension/issues/4711) — Support
new function `createTransaction` for dApp interfaces, and
[#4713](https://github.com/Koniverse/SubWallet-Extension/issues/4713) — [Research] Review
Polkadot-JS API RFC #6213 (Deprecate `signPayload` → `createTransaction`).

## Acceptance criteria

- [ ] **AC-1** — **Given** the Substrate inject API, **When** a dApp inspects it,
  **Then** an authorized dApp can call `createTransaction(...)` per the RFC #6213
  shape (call + intent), and the method is absent / rejected for unauthorized origins.
- [ ] **AC-2** — **Given** an authorized dApp calling `createTransaction`, **When**
  the wallet receives the call, **Then** it assembles the extrinsic, estimates the
  fee, and presents a human-readable confirmation in the approval popup.
- [ ] **AC-3** — **Given** the user approves, **When** signing proceeds, **Then** the
  transaction is signed in the background keyring and submitted through the
  transaction lifecycle (EPIC-8), returning the result/tx hash to the dApp.
- [ ] **AC-4** — **Given** an unsupported call, a fee/validation failure, or a user
  rejection, **When** it resolves, **Then** `createTransaction` rejects with a clear,
  typed error and nothing is signed or submitted.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — the `createTransaction` approval enqueues through `RequestService` / the Substrate handler behind the shared `PopupHandler`.
- Submission is owned by [EPIC-8](../epics/EPIC-8.md) (transaction lifecycle); this story exposes the dApp-facing API and routes into it, it does not reimplement submit/broadcast.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — signing runs in the background keyring.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-10.2](US-10.2-substrate-inject-api-injectedweb3.md) — adds `createTransaction` to the same `injectedWeb3` surface.
- Builds on [US-10.7](US-10.7-dapp-authorization-ui-per-origin.md) — only authorized origins on granted accounts may call it.
- Required by: future Substrate dApps that adopt RFC #6213 instead of building extrinsics client-side.
- Depends on EPIC-8 transaction lifecycle for the submit step.

### What we explicitly did NOT do

- No replacement of `signPayload` (US-10.2) — `createTransaction` is additive; existing dApps that build their own extrinsics keep working. Trigger to revisit: RFC #6213 finalizes and ecosystem dApps migrate.
- No EVM equivalent — this RFC is Polkadot-JS-specific; EVM tx building stays `eth_sendTransaction` (US-10.1).

### Sizing

Sized 3: one new injected method on the existing Substrate surface, wiring into
existing fee + request + transaction engines — a forward API addition, not a new
ecosystem integration. (Forward-looking; reassess once RFC #6213 stabilizes.)
