## Goal

A user can stake TAO into a Bittensor subnet using the native dTAO / alpha-token
model — delegating to a subnet validator and holding the subnet's alpha token —
and unstake and track rewards, so that Bittensor subnet staking is available
in-wallet through the native Subtensor path.

## Background

Bittensor's dTAO model stakes TAO into a *subnet*, where stake is represented as
the subnet's alpha token and on-chain swap runs via the `swapStakeLimit` pallet.
Per [AD-15](../../ARCHITECTURE.md#architecture-decisions) SubWallet uses the
**native Subtensor (Substrate) path, not the Subtensor-EVM**, because subnet
staking and reward-claim semantics cannot be expressed over the EVM bridge. The
engine models this through the Tao subclass of the native handler tree
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)). This story is the dTAO
staking *screen*: subnet/validator selection, stake (TAO→alpha), unstake
(alpha→TAO) and position tracking, reusing the earning-list surface from
[US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md).

Materializes [FR-119](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** a TAO account, **When** the user stakes into a subnet
  by delegating to a subnet validator, **Then** the native dTAO extrinsic submits
  via the Subtensor path and the subnet position (alpha holding, subnet, APY)
  appears in the earning list.
- [x] **AC-2** — **Given** a subnet position, **When** the user unstakes
  (alpha→TAO) or moves stake, **Then** the corresponding native call submits and
  the position state updates from the subject.
- [x] **AC-3** — **Given** the flow, **When** any dTAO action is constructed,
  **Then** it uses the native Subtensor pallet path (AD-15), never the
  Subtensor-EVM bridge.
- [x] **AC-4** — **Given** a stake amount that exceeds the subnet's
  `swapStakeLimit` slippage bound, **When** the user submits, **Then** the flow
  blocks with a clear limit/slippage error and submits nothing.

## Dev notes

### Architecture constraints

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — Bittensor uses the native Subtensor path with the dTAO / alpha-token model and on-chain swap via `swapStakeLimit`; never the Subtensor-EVM.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — dTAO staking is the Tao subclass of the native handler tree; this screen consumes its subjects.
- This story does NOT introduce new AD entries.

distinct native-Subtensor integration (subnet/validator targeting, TAO↔alpha

one external-system integration, sizing at 5.
