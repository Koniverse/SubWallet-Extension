## Goal

Users can discover and enter NFT-mint campaigns directly from the wallet: an active
mint campaign is displayed on the relevant in-app surfaces with its details and a
clear entry point to mint, so the team can run NFT drops to engaged users
release-free. When no campaign is active or the config is unavailable, nothing is
shown and the surface keeps working.

## Background

NFT-mint campaigns are remotely configured content delivered through the same
static-data / cache-proxy substrate as the other campaign surfaces
([AD-23](../../ARCHITECTURE.md#architecture-decisions),
[AD-25](../../ARCHITECTURE.md#architecture-decisions)). This story owns the
*display* of an NFT-mint campaign (artwork, details, mint CTA) and the *entry point*
to mint, reusing the remote-content mock and graceful-degradation pattern from
[US-19.4](US-19.4-in-app-campaign-banners.md). The mint transaction itself routes
through the existing transaction/NFT flows; this story surfaces the campaign and
hands off to mint.

Materializes [FR-159](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** an active NFT-mint campaign in remote config, **When** the
  user opens the hosting surface, **Then** the campaign is displayed with its
  artwork/details and a mint entry point.
- [ ] **AC-2** — **Given** a displayed mint campaign, **When** the user taps mint,
  **Then** they are routed to the configured mint flow (the mint transaction is not
  re-implemented here).
- [ ] **AC-3** — **Given** no active campaign or a failed config fetch, **When** the
  surface renders, **Then** nothing is shown and the surface keeps working — the
  failure never breaks the host screen.
- [ ] **AC-4** — **Given** a campaign outside its active mint window, **When** the
  surface renders, **Then** the campaign is not offered as mintable.

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions), [AD-25](../../ARCHITECTURE.md#architecture-decisions) — campaign config served release-free through static-data / cache-proxy; do not hardcode mint campaigns in the bundle.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-19.4](US-19.4-in-app-campaign-banners.md) — reuses the remote-content fetch + mock + graceful-degradation pattern.
- Hands off to the existing transaction/NFT flow for the actual mint — does not re-implement it.

### What we explicitly did NOT do

- No mint-transaction re-implementation — the mint routes through the existing transaction/NFT flows; this story owns display + entry only.

### Points justification

3 pts — §3a-bis "1 internal integration": display an NFT-mint campaign from the
existing cache-proxy substrate (AD-23/AD-25) and route into the existing mint flow.
It rides established delivery + transaction layers, so 3 not 5.
