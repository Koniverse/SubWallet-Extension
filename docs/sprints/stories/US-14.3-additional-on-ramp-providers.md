## Goal

Broaden the on-ramp's provider and payment-rail coverage so more users — across
more regions, tokens, and payment methods — can fund the wallet. This adds
MoonPay and Ramp.Network as on-ramp providers and brings native Apple Pay /
Google Pay rails into the buy flow. Each addition plugs into the existing
adapter contract, so the buy surface gains coverage without gaining complexity.

## Background

The on-ramp aggregator established in
[US-14.1](US-14.1-fiat-on-ramp-buy-crypto-with-card.md) was built so that a new
provider is a **new adapter, not a new buy-UI branch**: a
`CreateBuyOrderFunction` keyed by `SupportService`, surfaced through the
`buyServiceInfos` / `buyTokenConfigs` static-data roster
([AD-23](../../ARCHITECTURE.md#architecture-decisions)). This story exercises that
contract by adding MoonPay and Ramp.Network and wiring native Apple Pay / Google
Pay as payment methods. Where a provider requires a secret credential to mint its
order URL, the URL is built behind the SubWallet backend / Services SDK
([AD-19](../../ARCHITECTURE.md#architecture-decisions),
[AD-24](../../ARCHITECTURE.md#architecture-decisions), NFR-16 / NFR-20), exactly
as Transak does; provider publishable keys may live client-side.

Materializes [FR-140](../../PRD.md#functional-requirements). This story is **forward / planned** — the
capability is not yet shipped. It is sequenced after US-14.1 because it depends on
the adapter contract and static-data roster that story establishes. Tracked by
[#2046](https://github.com/Koniverse/SubWallet-Extension/issues/2046) — Integrate
MoonPay fiat on-ramp feature,
[#2666](https://github.com/Koniverse/SubWallet-Extension/issues/2666) — Integrate
Ramp.Network fiat on-ramp solution, and
[#4129](https://github.com/Koniverse/SubWallet-Extension/issues/4129) — Integrate
additional payment methods (Apple pay, Google Pay).

## Acceptance criteria

- [ ] **AC-1** — **Given** the existing buy surface, **When** MoonPay and
  Ramp.Network are added, **Then** each ships as a `CreateBuyOrderFunction`
  adapter registered under `SupportService` with **no** per-provider `if` branch
  added to the buy screen's dispatch.
- [ ] **AC-2** — **Given** a provider that requires a secret credential to build
  its order URL, **When** that adapter is added, **Then** the URL is minted
  behind the backend / Services SDK and **no** secret credential is added to the
  shipped bundle (provider publishable keys excepted).
- [ ] **AC-3** — **Given** the new providers and tokens, **When** their
  availability is configured, **Then** it is driven by `buyServiceInfos` /
  `buyTokenConfigs` static data (release-free), including per-provider
  `isSuspended`.
- [ ] **AC-4** — **Given** native Apple Pay / Google Pay are offered by a
  provider, **When** the user selects that payment method, **Then** the order URL
  / provider flow carries the native-pay rail and falls back gracefully where the
  device or provider does not support it (unhappy path: native pay unavailable).
- [ ] **AC-5** — **Given** a newly added provider does not service the user's
  region or rejects KYC, **When** the user reaches its hosted flow, **Then** the
  rejection is handled by the provider and the user returns to the wallet cleanly
  (unhappy path: provider unavailable in region / KYC rejection).

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) / [AD-19](../../ARCHITECTURE.md#architecture-decisions) — providers needing a secret credential mint their order URL behind the backend / Services SDK; no secret in the bundle (NFR-16).
- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — new providers/tokens are configured via the static-data roster, not hard-coded.
- This story does NOT introduce new AD entries — it extends the existing adapter contract.

### Cross-story dependencies

- Builds on [US-14.1](US-14.1-fiat-on-ramp-buy-crypto-with-card.md) — adds adapters into the `CreateBuyOrderFunction` map and `SupportService` roster, and the static-data-driven availability set up there. No change to the buy-screen dispatch.

### What we explicitly did NOT do

- No new buy-UI screen or per-provider branch — the whole point is that providers are adapters. Trigger to revisit: a provider that cannot be expressed as a URL-builder hand-off (e.g. an embedded in-app SDK).
- No off-ramp expansion — off-ramp stays Transak-only ([US-14.2](US-14.2-fiat-off-ramp-sell-crypto-for-fiat.md)); this story is on-ramp coverage (FR-140).
