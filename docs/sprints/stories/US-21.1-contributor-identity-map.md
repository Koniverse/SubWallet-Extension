## Goal

Produce the canonical contributor map that resolves every identity in `git shortlog -s` to exactly one GitHub login, so the history backfill (US-21.2) can populate the `assignee` frontmatter field with real RULE-15-compliant logins.

## Background

The project's git history contains ~1,700 commits from dozens of identities — real names, GitHub usernames, email addresses, and bots mixed together. The contributor map is a prerequisite for the history backfill because every story's `assignee` field must hold a GitHub login (RULE-15), not a `git user.name`. Without this map, the backfill would leave `assignee` empty or fill it with non-compliant values.

The approach is:

1. Collect all identities from `git shortlog -s` across all branches.
2. Cross-reference each identity against the GitHub API (commit authors, team membership).
3. Merge duplicates (one person under multiple emails or names).
4. Classify each entry: human (with GitHub login), bot (e.g. `dependabot[bot]`), or orphan (unresolvable).
5. Write the canonical map to `docs/notes/contributor-map.md`.

## Acceptance criteria

- [ ] **AC-1** — **Given** the repo's full git history, **When** `git shortlog -s` is run across all branches, **Then** every identity is collected **And** the list deduplicated.
- [ ] **AC-2** — **Given** a list of raw identities, **When** each is cross-referenced against the GitHub API (`gh api` or `GET /search/users`), **Then** a GitHub login is resolved for every human identity.
- [ ] **AC-3** — **When** bots are detected (e.g. `dependabot[bot]`, `github-actions[bot]`), **Then** they are classified separately with a `(bot)` annotation.
- [ ] **AC-4** — **When** an identity cannot be resolved (no GitHub match, no email match), **Then** it is flagged as `orphan` in a separate section for manual resolution.
- [ ] **AC-5** — **When** the map is complete, **Then** `docs/notes/contributor-map.md` is created with the canonical table of `git identity → GitHub login` mappings.
- [ ] **AC-6** — **When** the map is checked with `koni-docs validate`, **Then** no new errors are introduced.

## Dev notes — Architecture constraints

- The map is a **one-time manual effort** with a maintenance pattern: new contributors add themselves to the map as part of their first PR.
- The map must survive branch resets — it lives in `docs/notes/contributor-map.md`, committed to `subwallet-dev`.
