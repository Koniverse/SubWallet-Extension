## Goal

Produce the canonical contributor map that resolves every identity in `git shortlog -s` to exactly one GitHub login, so the history backfill (US-21.2) can populate the `assignee` frontmatter field with real RULE-15-compliant logins.

## Background

The project's git history contains 20,125 commits from 123 identities — real names, GitHub usernames, email addresses, and bots mixed together. The contributor map is a prerequisite for the history backfill because every story's `assignee` field must hold a GitHub login (RULE-15), not a `git user.name`. Without this map, the backfill would leave `assignee` empty or fill it with non-compliant values.

The approach is:

1. Collect all identities from `git shortlog -s` across all branches.
2. Cross-reference each identity against the GitHub API (commit authors, team membership).
3. Merge duplicates (one person under multiple emails or names).
4. Classify each entry: human (with GitHub login), bot (e.g. `dependabot[bot]`), or orphan (unresolvable).
5. Write the canonical map to `docs/notes/contributor-map.md`.

## Acceptance criteria

- [x] **AC-1** — **Given** the repo's full git history, **When** `git shortlog -s` is run across all branches, **Then** every identity is collected **And** the list deduplicated.
- [x] **AC-2** — **Given** a list of raw identities, **When** each is cross-referenced against the GitHub API (`gh api` or `GET /search/users`), **Then** a GitHub login is resolved for every human identity.

## Implementation notes

123 identities over 104 unique emails and 20,125 commits resolved to **72 humans,
6 bots, 1 orphan**. The three commit counts sum exactly to 20,125.

Two resolution rules turned out to matter more than the story anticipated:

- **Resolve noreply addresses through the numeric id, not the embedded login.**
  `<id>+<login>@users.noreply.github.com` freezes the login as it was when the
  commit landed. Three accounts had since been renamed — `NamPhamc99` → `nulllpc`
  split 2,871 commits across two apparent people until the id was used.
- **Read `author.login` off a real commit, not `GET /commits?author=<email>`.**
  That endpoint only searches the **default branch**, so it reported branch-only
  contributors as unattributable. Switching to `GET /commits/{sha}` moved 14 emails
  out of `orphan`.

**Name matching is not a valid rule.** `lw` (a git name of `lw-cdm`) is a real
GitHub account belonging to Luca Wehrstedt; `bluedot` is an Organization, not
`bluezdot`; `namph` is Nam Phung, not `nulllpc`. Resolving those names to their
same-named accounts would have misattributed 1,033 commits to strangers.

Four identities had contradictory evidence and were confirmed by the repo owner:
the `quangdm@*` / `dominhquangdev1998@` cluster → `Quangdm-cdm`, `anhnt@*` →
`anhntk54`, `nampc@foobla.com` → `nulllpc`, `duntpa2@gmail.com` → `frenkie-ng`.

The single orphan is `CustomBlink` (1 commit); its GitHub account (id `62216396`)
has been deleted, so no login exists to map to.

Status is `review`, not `done`: the `done` gate needs `version_shipped`, and this
change ships in no product release yet.
