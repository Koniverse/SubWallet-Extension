## Goal

A user can connect SubWallet to a dApp that pairs over **WalletConnect** — by
scanning a QR or pasting a `wc:` URI — and approve a session that spans Substrate
and/or EVM chains. This reaches the large class of dApps (and mobile/desktop dApps)
that integrate WalletConnect rather than relying on an injected browser provider.

## Background

WalletConnect v2 is a relay-based session protocol: a dApp shows a pairing URI, the
wallet pairs, the two negotiate a *session* over required/optional namespaces
(`polkadot`, `eip155`), and thereafter the dApp sends signing requests over the
relay that the wallet approves and answers. SubWallet uses
`@walletconnect/sign-client` in the background. Unlike injection, WC works without
a content script on the dApp page, which is why it is the path for mobile dApps and
sites that do not inject.

WC connect and per-request approvals flow through `RequestService`: AD-21 names
dedicated WalletConnect connect/unsupported handlers alongside the per-ecosystem
ones, all behind the shared `PopupHandler`. Session-scoped chains and accounts are
gated by the same per-origin authorization model (US-10.7). Signing for an approved
WC request routes to the matching ecosystem signer (Substrate or EVM) in the
background keyring (AD-04).

Materializes [FR-96](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a dApp pairing URI (QR or pasted `wc:` string), **When**
  the user submits it, **Then** a session proposal is shown listing the requested
  Substrate / EVM namespaces and accounts for approval.
- [x] **AC-2** — **Given** an approved session, **When** the dApp sends a sign
  request (Substrate or EVM), **Then** it is routed to the WalletConnect request
  handler, surfaced for approval, signed in the background keyring, and answered over
  the relay.
- [x] **AC-3** — **Given** an active session, **When** the user (or the dApp)
  disconnects, **Then** the session is removed, its authorization revoked, and no
  further requests from it are accepted.
- [x] **AC-4** — **Given** a session proposal requesting a chain/method SubWallet
  does not support, **When** it is evaluated, **Then** the unsupported handler
  surfaces a clear error and the proposal is not silently approved.
- [x] **AC-5** — **Given** a user rejection or relay timeout, **When** the proposal
  or request resolves, **Then** it rejects cleanly, no session is created/left
  half-open, and the dApp receives a standard WC error.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — WalletConnect connect + unsupported handlers behind the shared `PopupHandler`, routing approved requests to the per-ecosystem signer.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — WC signing runs in the background keyring; the relay never sees key bytes.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-10.7](US-10.7-dapp-authorization-ui-per-origin.md) — WC sessions are authorized per origin and surfaced in the access manager.

- Sibling [US-10.11](US-10.11-walletconnect-session-and-dashboard-hardening.md) — the WalletConnect-dashboard hardening (stale/duplicate sessions) lands there.

### Sizing

Sized 5: one external system integration (the WalletConnect relay + sign-client)
spanning two namespaces, with session lifecycle (pair / approve / sign / disconnect
/ expire) and an unsupported-request path.

> **"Connection loss" is filed twice, verbatim** — #1935 and #2052, both *"Re-check for
> WalletConnect connection loss"*. A relay session that the wallet cannot see the state of will drop
> silently, and the same symptom keeps returning; the durable answer is the session reconciliation
> in [US-10.11](US-10.11-walletconnect-session-and-dashboard-hardening.md).
>
> **The namespace is the recurring integration cost.** #2119 (update the WC namespace), #3870
> (allow the Polkadot namespace to use an EVM address), #4598 (handle networks the WC system does not
> support). Every chain WalletConnect adds is a new namespace entry the wallet has to speak.
>
> **#4330 keeps its own row rather than being an umbrella.** *"Bug when SubWallet connects to
> Remix"* has one child (#4377) **and** its own CHANGELOG line, so it is a delivery that spawned a
> sub-task, not an umbrella ([AGENTS.md](../../../AGENTS.md) rule 10) — unlike #4438, which has six children and no
> line of its own.
