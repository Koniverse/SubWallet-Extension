## Goal

Users in more language markets can read the wallet natively: the bundled-locale set
is extended beyond the Round-1 four to additional languages (German, French, and
others), reusing the same English-canonical loader and fallback the first round
established — so each new market is a content addition, not new infrastructure.

## Status

> **📋 backlog — nothing here has shipped.** All 3 rows below are **open on the tracker** and no
> acceptance criterion is ticked. FR-154 is `📋 planned` in the PRD; this is the forward story of
> the epic.

## Background

[US-19.2](US-19.2-multi-language-i18n-round-1.md) shipped the i18n infrastructure —
the language picker, the bundled-locale loader, the English-fallback behavior, and
the key-parity harness — for the first four languages (VI/ZH/JA/RU). This story is
the **forward** continuation: add the next set of bundled locales (DE/FR/…) on top
of that established baseline, governed by the same English-canonical rule
([NFR-13](../../PRD.md#non-functional-requirements)). No new mechanism is needed —
each language is a new `locales/{lng}/translation.json` plus a picker entry.

Materializes [FR-154](../../PRD.md#functional-requirements). This story is **forward** — FR-154 is
`📋 planned`; it is not yet shipped. Tracked by
[#4154](https://github.com/Koniverse/SubWallet-Extension/issues/4154) — Support
adding some new languages on the wallet, with per-language requests
[#4126](https://github.com/Koniverse/SubWallet-Extension/issues/4126) — New language
support: German, French and
[#4528](https://github.com/Koniverse/SubWallet-Extension/issues/4528) — Spanish
Language Support.

## Acceptance criteria

- [ ] **AC-1** — **Given** the additional bundled locales (German, French, …),
  **When** the user selects one from the language picker, **Then** the UI re-renders
  in that language from its bundled `locales/{lng}/translation.json`.
- [ ] **AC-2** — Each new locale's `translation.json` keys are a subset of the
  English source (no new locale introduces a key absent from English), verified by
  the same key-parity harness from US-19.2.
- [ ] **AC-3** — **Given** a newly added locale with an incomplete translation,
  **When** an untranslated string renders, **Then** it falls back to the English
  value (never the raw key), reusing the US-19.2 fallback behavior.

## Dev notes

### Architecture constraints

- Same i18n governance as US-19.2: English-canonical, bundled per locale ([NFR-13](../../PRD.md#non-functional-requirements)). No new AD entry.
- Reuses the loader, picker and fallback from US-19.2 — this story adds content + picker entries, not new infrastructure.

### Cross-story dependencies

- Builds on [US-19.2](US-19.2-multi-language-i18n-round-1.md) — consumes its bundled-locale loader, language picker, English-fallback behavior, and key-parity harness; adds locales on top.

### What we explicitly did NOT do

- No new i18n infrastructure — this is purely additive content + picker entries on the US-19.2 baseline.
- No online hot-update — still EPIC-1's FR-4.

### Points justification

3 pts — §3a-bis "multi-doc bundle": several additional locale files (a content
bundle) layered on the already-built US-19.2 infrastructure. No new system or
external integration, so 3 rather than the 5 the first round needed.
