## Goal

A user can stake on additional networks beyond the relay/parachain set —
directly from the wallet, so that staking coverage expands as new networks are
onboarded. This story scopes the **planned** next batch (Enjin, Phala, xx
Network and further Substrate chains).

## Background

This is a *coverage-expansion* story: each additional network is onboarded as a
new staking handler on the existing native-staking handler tree
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)) — each rides the same
handler seam introduced in
[US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) (and reuses
the native / collator screen patterns from
[US-12.3](US-12.3-collator-parachain-staking.md)) but each has its own pallet
quirks, minimums and target lists (served from the static-data cache,
[AD-23](../../ARCHITECTURE.md#architecture-decisions)). The story bundles several
network onboardings rather than one, so it is sized as a batch.

> **Cross-reference (not this story's scope)** — Mythos, Tanssi, Amplitude and
> Energy native staking already ship under **FR-114 /
> [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md)** (the
> native-nomination handler tree) — they are **not** part of this FR. Their
> handlers exist in code under
> `services/earning-service/handlers/native-staking/` (`mythos.ts`, `tanssi.ts`,
> `amplitude.ts`, `energy.ts`), each instantiated in `earning-service/service.ts`,
> as evidence they are shipped *elsewhere* (FR-114). This story (FR-125) scopes
> the **planned** Enjin / Phala / xx batch only.

**Planned / forward (not yet shipped)** — Enjin, Phala, xx Network and further

## Acceptance criteria

- [ ] **AC-1** — **Given** an account on a newly supported network (Enjin / Phala /
  xx / …), **When** the user stakes, **Then** the stake submits through that
  network's handler and the position appears in the earning list with the correct
  per-network APY and unbonding semantics.
- [ ] **AC-2** — **Given** an active position on an added network, **When** the user
  manages it (bond-more / unbond / claim / withdraw as the network supports),
  **Then** each action submits the correct per-network extrinsic and the position
  updates from the subject.
- [ ] **AC-3** — **Given** a stake amount below a network's specific minimum or
  violating a network-specific rule, **When** the user submits, **Then** the flow
  blocks with that network's validation error and submits nothing.

## Dev notes

### Architecture constraints

- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — each network is a new staking handler subclass on the existing tree; per-network quirks live in the subclass, not in shared logic or the screen.
- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — each network's validator/collator target list is served from the static-data cache.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) and [US-12.3](US-12.3-collator-parachain-staking.md) — reuses the native / collator staking screen patterns and the pool-handler subject fixture.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — adds handler subclasses against the engine seam.

### Points justification

8 pts — §3a-bis "content batch ≥ 3 items": this bundles the **planned** staking
onboarding for three-plus distinct networks (Enjin, Phala, xx, …), each with its
own pallet quirks, minimums and target lists — a batch of ≥3 handler integrations
rather than one. Batch of ≥3 sizes at 8; split per-network if any one proves
non-standard. (Mythos, Tanssi, Amplitude and Energy native staking ship under
FR-114 / US-12.1, not this story, and add no points here.)
