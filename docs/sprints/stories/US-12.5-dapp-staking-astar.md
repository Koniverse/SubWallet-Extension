## Goal

A user can stake ASTR to dApps registered in Astar's dApp-staking program, move
stake between dApps, and claim staker rewards, so that Astar's dApp-staking yield
is available in-wallet with the program's period/era and tier semantics handled
for the user.

## Background

Astar dApp staking is a distinct yield program: stakers lock ASTR against
*registered dApps* (not validators), with its own period/era lifecycle, tiered
rewards and claim windows. The engine models it through the Astar subclass of the
native handler tree ([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)), which encodes the
dApp-staking pallet calls and reward-claim semantics. This story is the
dApp-staking *screen*: dApp selection, stake/unstake-and-withdraw, move-stake and
the (sometimes multi-era) claim flow, reusing the earning-list surface from
[US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md).

Materializes [FR-118](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [ ] **AC-1** — **Given** an Astar account, **When** the user stakes to a
  registered dApp at or above the program minimum, **Then** the stake submits
  through the Astar handler and the dApp-staking position (staked dApp, period,
  claimable rewards) appears in the earning list.
- [ ] **AC-2** — **Given** an active dApp-staking position, **When** the user
  unstakes-and-withdraws, moves stake to another dApp, or claims staker rewards
  (including across multiple eras), **Then** the corresponding pallet calls submit
  and the position state updates from the subject.
- [ ] **AC-3** — **Given** an unregistered or capacity-limited dApp, or a stake
  below the program minimum, **When** the user tries to submit, **Then** the flow
  blocks with a clear validation error and submits nothing.

nomination variant. One distinct program integration sizes at 5.
