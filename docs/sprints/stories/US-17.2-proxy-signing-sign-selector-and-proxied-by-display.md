## Goal

When a transaction needs to be dispatched on behalf of a proxied account, a
**Sign Selector** lets the user pick which controlling (proxy) account actually
signs, and the **proxied-by** provenance — who signed, on whose behalf — is shown
through the confirmation screen and into history. This is the *signing-time* half
of proxy: the proxy account signs with its own key, but the extrinsic executes
for the proxied account, and that two-account relationship stays visible end to
end.

## Background

Proxy signing surfaces as a **Sign Selector popup on the existing transaction
flows** ([AD-16](../../ARCHITECTURE.md#architecture-decisions)) rather than a
separate proxy-initiated transaction builder. When the sender is a proxied
account, the wallet wraps the chosen call as `proxy.proxy(...)` and presents the
Sign Selector so the user picks which authorised controlling account signs it.
The signature is produced by that proxy account's key — which stays in the
background keyring ([AD-04](../../ARCHITECTURE.md#architecture-decisions)) — while
the extrinsic executes on behalf of the proxied account. The popup and approve /
reject plumbing are the *existing* RequestService per-ecosystem handler surface
([AD-21](../../ARCHITECTURE.md#architecture-decisions),
[US-2.7](US-2.7-requestservice-approval-queue.md)); this story plugs the Sign
Selector into it rather than building a new approval surface.

FR-150 is one functional requirement split into two stories by capability. The
sibling ([US-17.1](US-17.1-proxy-types-and-authority-management.md)) owns the
authority **model** — adding a proxy, choosing its named type, managing the
relationship. This story (US-17.2) owns the signing-time **behaviour** — the Sign
Selector that picks the signing account at dispatch time, the authority/type
check that the chosen proxy actually holds the required authority for the call,
and the proxied-by display that keeps both accounts visible. It depends on
US-17.1 because there is no authority to select or enforce until the model exists.

The key cross-cutting invariant this story enforces: a proxied transaction is
**signed by the proxy account's key but executes on behalf of the proxied
account**, and that proxied-by provenance must always be visible — in the Sign
Selector, on the confirmation, and in the resulting history entry. A proxied
transaction that renders as an ordinary self-signed one is a defect.

Materializes [FR-150](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [ ] **AC-1** — **Given** a transaction whose sender is a proxied account,
  **When** the user signs, **Then** a Sign Selector lets them choose an
  authorised controlling (proxy) account, the call is dispatched as
  `proxy.proxy(...)` signed by that account's key, and it executes on behalf of
  the proxied account.
- [ ] **AC-2** — **Given** a proxied transaction at the Sign Selector,
  confirmation, and in history, **When** the user views it, **Then** the
  proxied-by provenance is shown — both the signing proxy account and the
  proxied (on-behalf-of) account — never collapsed into a single self-signed
  sender.
- [ ] **AC-3** — **Given** a chosen controlling account that lacks the proxy
  authority or whose proxy type does not permit this call (e.g. a NonTransfer
  proxy used for a balance transfer), **When** the user attempts to sign,
  **Then** the wallet rejects it before submit with a clear authority error.
- [ ] **AC-4** — **Given** a proxy relationship that has been revoked,
  **When** the user attempts to sign on behalf of the proxied account with the
  now-revoked proxy, **Then** the Sign Selector no longer offers it (or signing
  fails with a clear error) and nothing is submitted.

## Dev notes

### Architecture constraints

- [AD-16](../../ARCHITECTURE.md#architecture-decisions) — proxy signing is a Sign
  Selector popup on the *existing* transaction flows, not a parallel
  proxy-initiated transaction surface; the supported types are the named pallet
  types only.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) — the proxy account's key
  signs in the background keyring; proxy introduces no new custody path.
- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — the Sign Selector rides
  the per-ecosystem RequestService handler surface (popup / approve / reject),
  whose primary implementation is [US-2.7](US-2.7-requestservice-approval-queue.md).
- This story introduces no new AD entries.

### Cross-story dependencies

- Builds on [US-17.1](US-17.1-proxy-types-and-authority-management.md) — enforces
  the proxy relationship and named-type authority that story establishes.
- Builds on / relates to [US-2.7](US-2.7-requestservice-approval-queue.md) — the
  Sign Selector plugs into the existing RequestService sign queue and
  per-ecosystem handlers rather than building a new approval flow.

### What we explicitly did NOT do

- No parallel proxy-initiated transaction builder — proxy signing is surfaced on
  the existing flows via the Sign Selector (AD-16). Trigger to revisit: a need
  for proxy-only flows the standard transaction surface cannot express.
- No EVM / hardware-wallet proxy signing — Substrate software accounts only
  (AD-16); the proxy key signs in the background keyring (AD-04).
