## Goal

A security-conscious user can sign with additional cold-storage devices — Trezor,
Tangem, D'Cent, and Keystone 3 Pro — so that the same off-extension signing
guarantee already delivered for Ledger and the QR devices extends to their device
of choice, with their private key never entering the extension.

## Background

This is the roadmap story for [FR-149](../../PRD.md#functional-requirements): broaden hardware-wallet
coverage beyond the shipped Ledger ([US-16.1](US-16.1-ledger-hardware-wallet-signing.md))
and QR ([US-16.2](US-16.2-qr-signing-keystone-and-polkadot-vault.md)) devices to
Trezor, Tangem, D'Cent, and Keystone 3 Pro. It is **planned / forward-looking** —
not yet shipped — and inherits the offline branch of the keyring boundary
([AD-04](../../ARCHITECTURE.md#architecture-decisions), [NFR-14](../../PRD.md#non-functional-requirements)): the
wallet builds and validates the payload, the device signs, the wallet submits.
Tracked by [#4175](https://github.com/Koniverse/SubWallet-Extension/issues/4175) —
Support cold wallets, with per-device tracking issues
[#1387](https://github.com/Koniverse/SubWallet-Extension/issues/1387) — Integrate
with Trezor, [#1829](https://github.com/Koniverse/SubWallet-Extension/issues/1829) —
Integrate Tangem Cold Wallet,
[#4128](https://github.com/Koniverse/SubWallet-Extension/issues/4128) — Integrate
D'Cent, and [#2273](https://github.com/Koniverse/SubWallet-Extension/issues/2273) —
Support for Keystone 3 Pro.

Per the EPIC-16 cross-cutting **transport-adapter invariant**, a new device is a
new transport adapter on the shared signing surface (USB / QR / NFC), **not new
signing logic**. Each device family here maps to one of the transports the pilot
already established — D'Cent and Tangem add **NFC**, Trezor adds another **USB**
adapter alongside Ledger, and **Keystone 3 Pro** extends the existing **QR** path
from US-16.2. The work is therefore device-/transport-specific framing and
detection on top of the signing-request surface, never a second way to produce a
signature.

The devices are bundled into one roadmap story because each is a comparatively
small adapter on an already-proven surface, but together they span multiple
distinct transports (USB, NFC, QR) and multiple third-party device integrations —
which is what sets the size (see Dev notes). When this story is scheduled it is a
candidate to split per-transport (e.g. NFC family vs. the Keystone 3 Pro QR
extension) if any one device proves heavier than the batch estimate.

## Acceptance criteria

- [ ] **AC-1** — **Given** a connected Trezor over USB, **When** the user signs a
  transaction, **Then** it is signed on-device via the Trezor transport adapter and
  submitted; no private key enters the extension (AD-04, NFR-14).
- [ ] **AC-2** — **Given** a Tangem card or a D'Cent device over NFC, **When** the
  user signs a transaction, **Then** it is signed on-device via the NFC transport
  adapter and submitted; no private key enters the extension (AD-04, NFR-14).
- [ ] **AC-3** — **Given** a Keystone 3 Pro account, **When** the user signs a
  transaction, **Then** the existing QR display→scan→submit round trip
  ([US-16.2](US-16.2-qr-signing-keystone-and-polkadot-vault.md)) completes fully
  offline, with any Keystone 3 Pro-specific QR framing handled.
- [ ] **AC-4** — **Given** every added device, **When** signing completes, **Then**
  it reuses the shared signing-request surface from US-16.1 / US-16.2 — the only
  new code is the transport adapter, with no new signing path (EPIC-16 invariant).
- [ ] **AC-5** — **Given** an unsupported firmware version or a device that cannot
  be detected on its transport, **When** the user attempts to sign, **Then** the
  wallet surfaces a clear error and submits nothing.

## Dev notes

### Architecture constraints

- [AD-04](../../ARCHITECTURE.md#architecture-decisions) — signing delegates to the
  device; the keyring never holds any of these device keys. Each new device is a
  transport adapter on the existing off-extension signing surface, introducing no
  new place a key could enter the extension.
- This story introduces no new AD entries.

### Points justification (§3a-bis)

- **8 points (~1 week, multi-system integration).** The story bundles **four
  distinct third-party device integrations** spanning **three transports** —
  Trezor (USB), Tangem + D'Cent (NFC), Keystone 3 Pro (QR). Each is individually a
  small adapter (the signing surface already exists), but together they are a
  multi-system integration batch, which the calibration scale places at 8, not 5.
  It deliberately stays under the 13-point split threshold; if scheduling reveals a
  single device (e.g. the first NFC integration) is heavier than budgeted, split
  per-transport at that point per the §3a-bis splitting rule.

### Cross-story dependencies

- **Builds on** [US-16.1](US-16.1-ledger-hardware-wallet-signing.md) — reuses the
  USB/WebHID connection + signing-request surface; Trezor is a sibling USB adapter.
- **Builds on** [US-16.2](US-16.2-qr-signing-keystone-and-polkadot-vault.md) —
  Keystone 3 Pro extends the QR display→scan→submit round trip; only device-specific
  QR framing is new.
- **Sibling:** both pilot stories establish the transport-adapter surface this story
  consumes; coordinate review where the shared signing-request surface is touched.
