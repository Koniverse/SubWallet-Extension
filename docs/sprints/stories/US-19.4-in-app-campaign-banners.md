## Goal

The team can reach users inside the wallet without shipping a release: in-app
marketing campaign banners are rendered from remotely configured content on the
relevant surfaces, so growth and announcement messaging can be pushed and updated
release-free. When the content is unavailable, the host screen shows no banner and
keeps working — a campaign never breaks the wallet.

## Background

Campaign banners are remotely configured content, not bundled assets. They are
fetched through SubWallet's static-data / cache-proxy substrate
([AD-23](../../ARCHITECTURE.md#architecture-decisions),
[AD-25](../../ARCHITECTURE.md#architecture-decisions)) so the team can push, update
or retire a campaign without an extension release. This story owns the *display +
fetch wiring* of those banners and, critically, the graceful-degradation path: a
fetch failure or empty config must render no banner rather than a blank or broken
screen.

Materializes [FR-157](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** a remotely configured active campaign, **When** the user
  opens a surface that hosts a banner slot, **Then** the campaign banner is rendered
  from the fetched config (image/copy/CTA) without an extension release.
- [ ] **AC-2** — **Given** a banner with a call-to-action, **When** the user taps it,
  **Then** they are routed to the configured destination (deep-link / external link)
  defined in the campaign config.
- [ ] **AC-3** — **Given** the campaign config fetch fails or returns empty, **When**
  the host surface renders, **Then** no banner is shown and the surface works
  normally — the failure never blanks or breaks the host screen.
- [ ] **AC-4** — **Given** a campaign whose active window has ended (expired), **When**
  the surface renders, **Then** the expired banner is not shown.

## Tasks

- [ ] **TASK-19.4.1** — Fetch campaign-banner config from the cache-proxy substrate (AC: 1)
  - [ ] Wire the banner slot to the static-data / `api-cache` campaign endpoint (AD-23, AD-25).
- [ ] **TASK-19.4.2** — Render the banner + CTA routing (AC: 1, 2)
  - [ ] Render image/copy/CTA; route the CTA to the configured deep-link / external link.
- [ ] **TASK-19.4.3** — Graceful-degradation path (AC: 3)
  - [ ] On fetch failure / empty config, render no banner and leave the host surface intact.
  - [ ] Set up the **remote-content mock** (populated + empty + failure) reused by US-19.5/19.6/19.7.
- [ ] **TASK-19.4.4** — Honor the campaign active window (AC: 4)
  - [ ] Hide banners outside their configured active window.

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — campaign config is static-ish remote data served release-free; consume it, do not hardcode campaigns in the bundle.
- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — campaign content is fetched through the cache-proxy layer with a fallback, never a keyed upstream in the bundle.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Required by [US-19.5](US-19.5-airdrop-quest-mission-display.md), [US-19.6](US-19.6-nft-mint-campaigns.md), [US-19.7](US-19.7-mission-pools-reward-program.md) — they reuse the remote-content mock (populated / empty / failure) set up here.
- Sibling [US-19.8](US-19.8-in-app-notification-center.md) — campaign alerts also surface in the notification center; coordinate the campaign-config shape.

### Points justification

3 pts — §3a-bis "1 internal integration": render banners from the existing
cache-proxy substrate (AD-23/AD-25) plus the degradation path. It rides an existing
delivery layer rather than integrating a new external system, so 3 not 5.
