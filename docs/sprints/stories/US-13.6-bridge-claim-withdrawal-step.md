## Goal

For asynchronous bridges that do not auto-deliver (Avail, Polygon), a user can
finalize the transfer with an explicit **claim / withdrawal** step on the
destination chain — and at every point can see that in-transit funds are
accounted for, never lost — so that a cross-chain transfer reliably completes
even when settlement spans a relay/proving window.

## Background

Some bridges in this epic settle in two phases: a source-chain deposit, then —
after a relay, proving, or challenge window — a destination-chain claim the user
must submit to release the funds. FR-131 names Avail and Polygon as the bridges
requiring this step. This story owns the shared **claim/withdrawal primitive**
and the in-transit → claim-ready → claimed state machine that
[US-13.3](US-13.3-avail-bridge.md) and
[US-13.4](US-13.4-polygon-unified-bridge.md) consume; it is the single place that
guarantees the epic-wide invariant that **in-transit funds are never reported as
lost**.

The claim transaction itself is built via the delegate-don't-build posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)) and its readiness/state is
tracked through the Services SDK backend
([AD-24](../../ARCHITECTURE.md#architecture-decisions)). The story's hardest case
is the **stalled** path: a deposit that left the source chain but whose claim is
not yet provable, or a claim that fails — the user must always see the funds as
pending/recoverable with clear guidance, never as a vanished balance.

Materializes [FR-131](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** an async bridge deposit that has reached the
  claim-ready state, **When** the user opens the pending bridge, **Then** a claim
  action is offered with the claimable amount and destination chain shown, and the
  claim is built via AD-18 (not hand-rolled).
- [x] **AC-2** — **Given** a claim-ready transfer, **When** the user submits the
  claim, **Then** the destination-chain claim transaction is signed, submitted,
  tracked, and the destination balance updates on inclusion.
- [x] **AC-3** — **Given** a deposit whose relay/proving window has not completed,
  **When** the user views it, **Then** it is shown as in-transit / claim-pending
  with an ETA and an explicit "funds are accounted for" state — never failed and
  never a blank/lost balance.
- [x] **AC-4** — **Given** a claim submission that fails (e.g. underpriced gas or
  a not-yet-provable root), **When** it errors, **Then** the transfer stays in
  claim-pending and is re-claimable — the user is never left with a transfer that
  reports done while funds are still escrowed.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — the destination claim transaction is built via the delegated bridge/XCM construction, not hand-rolled.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — claim readiness / bridge state is tracked through the Services SDK backend.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) — uses the bridge-hop tracking surface.
- Required by [US-13.3](US-13.3-avail-bridge.md), [US-13.4](US-13.4-polygon-unified-bridge.md) — both async bridges consume this claim step and the async-claim state machine owned here.

### What we explicitly did NOT do

- No claim step for the auto-delivering bridges (Snowbridge, Across) — they settle without a user claim; this primitive applies only to claim-requiring bridges (Avail, Polygon). Trigger to revisit: a new bridge whose settlement needs an explicit destination claim.

### Points justification

5 pts — §3a-bis sits between "1 internal integration" (3) and a multi-system
base (8). This is more than a single internal integration: it owns the
destination-chain claim transaction, the in-transit → claim-ready → claimed state
machine, AND the funds-never-lost recovery surface that two external-bridge
stories depend on. It is a production deliverable with a real failure-mode
contract, so it sizes at 5, not 3.
