## Goal

A user can stake into the Trusted Stake alpha-index integration — a partner-curated
alpha-index product — directly from the wallet, so that holders can access the
Trusted Stake strategy without leaving SubWallet.

## Background

Trusted Stake is a third-party alpha-index staking product integrated as an
earning option. Like the auto-rebalancing index
([US-12.11](US-12.11-alpha-index-auto-rebalancing-delegate-staking.md)) it gives
diversified subnet exposure, but the index composition and curation are owned by
the Trusted Stake partner; SubWallet integrates its stake/unstake path. Staking
runs the native Subtensor path per
[AD-15](../../ARCHITECTURE.md#architecture-decisions) and the position is modeled
on the handler tree ([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)). Because the index
definition and any partner API are external, this story carries a partner external
dependency.

Materializes [FR-125](../../PRD.md#functional-requirements). 📋 **Planned / forward** — not yet shipped;
this story scopes the planned integration. Tracked by
[#4879](https://github.com/Koniverse/SubWallet-Extension/issues/4879) —
Integration: Trusted Stake (Alpha Index Staking).

## Acceptance criteria

- [ ] **AC-1** — **Given** the Trusted Stake product, **When** the user stakes an
  amount, **Then** the stake submits via the native path per the Trusted Stake
  index definition and a Trusted Stake position appears in the earning list.
- [ ] **AC-2** — **Given** a Trusted Stake position, **When** the user unstakes,
  **Then** the position is unwound and the aggregate alpha/TAO is returned.
- [ ] **AC-3** — **Given** the Trusted Stake index definition is unavailable from
  the partner source, **When** the user opens the flow, **Then** the option is
  shown as temporarily unavailable rather than staking against stale/guessed
  weights.

## Dev notes

### Architecture constraints

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — native Subtensor path for the staking actions.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — Trusted Stake is a handler-backed position; the screen consumes its subjects.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.6](US-12.6-bittensor-dtao-subnet-staking.md) — reuses the dTAO native-staking foundation.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the (planned) handler subjects.
- Sibling [US-12.11](US-12.11-alpha-index-auto-rebalancing-delegate-staking.md) — both are alpha-index products; share the index-position UI shape.

### Points justification

5 pts — §3a-bis "1 external-system integration": integrating a single
partner-curated index product (stake/unstake against the Trusted Stake definition)
on the existing native path. One external partner integration sizes at 5; the
partner dependency is surfaced in `external_deps`.
