## Goal

A user can bridge assets between Avail and Ethereum directly in the wallet —
seeing the fee and ETA before signing and finalizing arrival with a claim step
on the destination — so that Avail-ecosystem value can move to and from Ethereum
without an external bridge UI.

## Background

The Avail bridge is an external Avail ↔ Ethereum bridge that hangs off the
bridge-hop interface from [US-13.1](US-13.1-xcm-parachain-transfers.md) and the
provider-handler fixture set up by
[US-13.2](US-13.2-snowbridge-eth-asset-hub.md). It is an **asynchronous** bridge:
the source-chain submission and the destination-chain settlement are separated by
a proving/relay window, and arrival requires an explicit claim step on the
destination chain — that claim step is owned by
[US-13.6](US-13.6-bridge-claim-withdrawal-step.md), which this story consumes.
Bridge construction follows the delegate posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the provider is reached
through the SubWallet proxy ([AD-19](../../ARCHITECTURE.md#architecture-decisions),
NFR-16).

Materializes [FR-128](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** an Avail-bridge-supported asset, **When** the user
  requests a bridge between Avail and Ethereum, **Then** the wallet shows the fee,
  estimated destination amount, and ETA before any signature.
- [x] **AC-2** — **Given** a confirmed Avail bridge, **When** the user signs the
  source-chain transaction, **Then** it is submitted and tracked, and the bridge
  enters the in-transit / claim-pending state.
- [x] **AC-3** — **Given** a bridge whose relay/proving window has not completed,
  **When** the user returns, **Then** it is shown as in-transit (not failed) and
  funds are never reported as lost; once provable, the destination claim becomes
  available (handed to US-13.6).
- [x] **AC-4** — **Given** the Avail bridge provider, **When** the wallet calls
  it, **Then** the call goes through the SubWallet proxy and no provider key is in
  the bundle (AD-19 / NFR-16).

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction delegated; the Avail handler consumes the provider.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — provider proxied; no key in the bundle (NFR-16).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) (bridge-hop interface) and [US-13.2](US-13.2-snowbridge-eth-asset-hub.md) (provider-handler fixture).
- Required by / consumes [US-13.6](US-13.6-bridge-claim-withdrawal-step.md) — the destination-chain claim step that finalizes this async bridge.
- Sibling [US-13.4](US-13.4-polygon-unified-bridge.md) — also async-with-claim; shares the async-claim state machine.

### Points justification

5 pts — §3a-bis "1 external system integration": one external bridge provider
(Avail) behind the existing interface. The async-claim handoff reuses the state
machine owned by US-13.6 rather than building it here, so it stays a
single-provider integration at 5.
