## Goal

A user can delegate stake to collators on parachains that run a
collator-staking model — Moonbeam, Moonriver, Astar, Amplitude, Pendulum, Krest,
Energy Web X, Mythos, Tanssi and compatible chains — and manage the delegation,
so that parachain-native staking is available in-wallet alongside relay-chain
nomination.

## Status

> **✅ done — shipped in 0.5.3.** All acceptance criteria are ticked and the 16 rows below are
> settled: 15 delivered, 1 closed without shipping.

## Background

Collator/parachain staking differs from relay nomination: a holder *delegates* to
a collator and the chain rewards delegations, with per-chain variations in
minimums, bond/unbond semantics and reward scheduling. The engine models these
through the parachain subclasses of `BaseNativeStakingPoolHandler`
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)). This story is the
delegation *screen* across the supported parachains; it reuses the earning-list
and pool-handler subject fixture from
[US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) and adds the
delegate / bond-more / schedule-and-execute-unbond / claim flows.

Materializes [FR-116](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** an account on a supported collator-staking parachain,
  **When** the user delegates to a collator at or above the chain minimum,
  **Then** the delegation submits through the parachain handler and the position
  appears in the earning list.
- [x] **AC-2** — **Given** an active delegation, **When** the user bonds more,
  schedules/executes an unbond, or claims rewards, **Then** the corresponding
  per-chain extrinsic submits and the position state updates from the subject.
- [x] **AC-3** — **Given** a delegation below the chain's collator minimum or to a
  collator at capacity, **When** the user tries to submit, **Then** the flow
  blocks with a clear validation error and submits nothing.
