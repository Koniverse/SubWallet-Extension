## Goal

Stop the web app from redoing account-add work on every open: today, opening the web
app re-injects accounts as brand-new ones, re-triggering account-add side effects
(auto-activating chains and tokens) every time. This story caches/reconciles injected
accounts so that work runs once, not on each open. It ships no new FR; its value is
that opening the web app is cheap and idempotent rather than re-running setup the user
already paid for.

## Background

This is the **injected-account caching slice** of EPIC-10's bug / iteration
(hardening) cluster — it owns no FR. When SubWallet injects accounts into the web app,
each open treats the injected accounts as brand-new, re-firing the account-add side
effects — auto-activating chains and tokens — that should only run the first time an
account appears. The result is redundant work and unwanted re-activation on every
open. The fix is to cache/reconcile injected accounts against what the wallet already
knows, so an account that was injected before is recognized as existing and the
add-side-effects run once
([#2286](https://github.com/Koniverse/SubWallet-Extension/issues/2286)).

This shares the lifecycle principle owned by EPIC-20 — *avoid redundant work on open*
— applied here to the injected-account surface. It cross-references EPIC-20's
core-structure / app-lifecycle refactor
([US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md)) for the general
"don't redo init on every open" principle, but the concrete fix stays in EPIC-10
because the bug lives on the inject surface (the injected-account bridge), which this
epic owns.

This routes through the same injected-account surface the feature stories built —
behind `RequestService`
([AD-21](../../ARCHITECTURE.md#architecture-decisions)) — so this story adds a
cache/reconcile guard and regression coverage rather than a new connector. It is
**ongoing**: the cache/reconcile fix lands here and the story stays open to absorb
follow-up injected-account churn issues.

## Acceptance criteria

- [ ] **AC-1** — **Given** accounts already injected on a previous open, **When** the
  web app is opened again, **Then** the injected accounts are reconciled against the
  cached set and recognized as existing — they are not treated as brand-new.
- [ ] **AC-2** — **Given** an injected account that already exists, **When** the web
  app opens, **Then** the account-add side effects (auto-activating chains/tokens) do
  NOT re-run for it — the work runs once, not on each open.
- [ ] **AC-3** — **Given** a genuinely new account injected for the first time,
  **When** the web app opens, **Then** the account-add side effects DO run for that
  account exactly once (the cache must not suppress first-time setup).
- [ ] **AC-4** — **Given** the injected-account caching fix, **When** it lands,
  **Then** it is paired with a regression test or guard tied to #2286 so the churn
  cannot silently reappear.

## Tasks

- [ ] **TASK-10.12.1** — Cache injected accounts and reconcile against the cached set on web-app open (AC: 1)
- [ ] **TASK-10.12.2** — Gate account-add side effects (chain/token auto-activation) on first appearance only, so they do not re-run on each open (AC: 2)

### What we explicitly did NOT do

- No new connector or ecosystem — this is hardening of the existing injected-account flow only.
- No change to the app-lifecycle init phasing itself — that is EPIC-20's ([US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md)); this story only stops the inject surface from redoing add-side-effects on open.
- No change to the chain/token activation logic itself — it gates *when* it runs (first appearance), not *what* it does.

### Points justification

3 pts — §3a-bis "2 days / 1 internal integration": a single, well-scoped fix on one
surface (cache + reconcile injected accounts; gate add-side-effects on first
appearance) anchored to one issue (#2286), plus a regression test. Comparable to the
other EPIC-10 hardening slices; carries the extra care of not suppressing legitimate
first-time setup (AC-3), which keeps it at 3 rather than 2.
