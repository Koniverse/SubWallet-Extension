## Goal

Keep the wallet's XCM and external-bridge routes working — and keep their funds
recoverable — as the ecosystem moves underneath them: parachain runtime upgrades
and ParaSpell API version bumps, plus the recurring real-world failure modes of
the asynchronous bridges (a claim that won't go through, an error message that
hides the cause, a fee re-query that never fires). This is the epic's **bridge
reliability hardening cluster**: one story whose single logic is *keep the bridge
working end to end* — both the construction seam (ParaSpell / runtime drift) and
the finalization seam (claim, error surfacing, fee re-query) that decides whether
a user actually receives their bridged tokens. It owns no FR; it defends the
already-shipped XCM base ([US-13.1](US-13.1-xcm-parachain-transfers.md)) and the
claim/withdrawal step ([US-13.6](US-13.6-bridge-claim-withdrawal-step.md)) against
the reliability issues observed in the field.

## Background

This story is re-grounded in **real, reported bridge/XCM reliability issues** on
SubWallet-Extension. It bundles them into one hardening anchor because they share
a single failing logic — the cross-chain hop, once built, must reliably *finish*:
the destination claim must succeed, errors must be legible, and stale fees must be
re-queried before submit. They are not separate features; they are the durability
of the same bridge engine.

The issues group into three facets:

**ParaSpell delegation & version drift.** The bridge-xcm epic delegates XCM
construction to ParaSpell ([AD-18](../../ARCHITECTURE.md#architecture-decisions))
so the wallet inherits ecosystem tracking instead of hand-maintaining XCM logic.
That delegation has a cost surface: ParaSpell ships breaking API version changes,
and parachains push runtime upgrades that move fee parameters and route shapes.
The approach itself needs periodic review, and individual providers reached via
that layer ship critical updates that must be tracked.
- [#4405](https://github.com/Koniverse/SubWallet-Extension/issues/4405) — Review and evaluate ParaSpell and the approach to handle XCM transfer.
- [#4424](https://github.com/Koniverse/SubWallet-Extension/issues/4424) — Critical Across update (a provider behind the bridge layer shipped a breaking update that must be absorbed without breaking the route).

**Bridge claim / withdrawal reliability.** The asynchronous bridges (Avail,
Polygon) finalize with a destination-chain claim step ([US-13.6](US-13.6-bridge-claim-withdrawal-step.md)).
The most-reported failures are claims that error or cannot complete after the
source-chain leg has succeeded — the exact case where the epic's invariant
("in-transit funds are never reported as lost") is under the most pressure.
- [#4108](https://github.com/Koniverse/SubWallet-Extension/issues/4108) — Recheck claim AVAIL token after bridge.
- [#4636](https://github.com/Koniverse/SubWallet-Extension/issues/4636) — [WebApp] Error when claiming token on Ethereum Sepolia after bridging AVAIL.
- [#4756](https://github.com/Koniverse/SubWallet-Extension/issues/4756) — Unable to claim token from cross-chain transfer Polygon zkEVM → Ethereum.
- [#4255](https://github.com/Koniverse/SubWallet-Extension/issues/4255) — AvailSpace — fix some bugs (claim/withdrawal-path defects on the Avail surface).

**Error handling.** When a bridge does fail, the failure must be legible and
actionable rather than an opaque dead end — otherwise the user cannot tell an
in-transit/claim-pending state from a true failure.
- [#4798](https://github.com/Koniverse/SubWallet-Extension/issues/4798) — Improve Across bridge error handling.

**Fee re-query.** EVM-side Snowbridge submissions need the fee re-queried at
submit time; a stale quote causes a wrong-fee submission or a silent failure.
- [#4622](https://github.com/Koniverse/SubWallet-Extension/issues/4622) — [Snowbridge] Improve re-query fee when submitting Snowbridge tx from EVM.

This is a defensive cluster that regression-guards
[US-13.1](US-13.1-xcm-parachain-transfers.md) and
[US-13.6](US-13.6-bridge-claim-withdrawal-step.md) rather than shipping a new
user feature; it owns no FR. Route quotes / multi-chain assembly that the fee
re-query rides on flow through the Services SDK backend
([AD-24](../../ARCHITECTURE.md#architecture-decisions)).

## Acceptance criteria

- [ ] **AC-1** — **Given** a ParaSpell version bump or a parachain runtime
  upgrade that changes a route's fee/shape (the #4405 evaluation surface),
  **When** the dependency or runtime moves, **Then** the core XCM routes still
  build + fee-estimate, and a route that previously built but no longer does is
  surfaced before release rather than failing silently in the field.
- [ ] **AC-2** — **Given** a provider critical update behind the bridge layer
  (#4424 Across), **When** the update is absorbed, **Then** the affected route
  keeps quoting and submitting under the new provider behavior, and any breaking
  change is contained to that route without taking down the rest of the bridge
  surface.
- [ ] **AC-3** — **Given** an asynchronous bridge whose source leg has succeeded
  (Avail #4108/#4636/#4255, Polygon zkEVM→Ethereum #4756), **When** the user runs
  the destination-chain claim, **Then** the claim completes and the bridged tokens
  arrive — the previously reported claim failures no longer reproduce.
- [ ] **AC-4** *(unhappy path)* — **Given** a bridge or claim that genuinely fails
  (#4798 Across), **When** the failure occurs, **Then** the user sees a clear,
  actionable error distinguishing a true failure from an in-transit / claim-pending
  state — funds in transit are never presented as lost (epic invariant).
- [ ] **AC-5** — **Given** a Snowbridge transaction submitted from EVM (#4622),
  **When** the user reaches submit after the quote may have gone stale, **Then** the
  fee is re-queried at submit time and the transaction is submitted with the current
  fee — never a stale-fee or silent-failure submission.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — ParaSpell is the delegated XCM/bridge builder; this story hardens the seam where ParaSpell version drift (v4 → v5 → v1 API) and provider critical updates meet live routes, without reimplementing XCM in-house.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — bridge quote / fee assembly rides the Services SDK backend; the Snowbridge fee re-query (#4622) operates through that path, not an ad-hoc on-device fee computation.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) — hardens the XCM base, the ParaSpell delegation, and the per-route toggle it established; reuses the per-route-toggle harness.
- Builds on [US-13.6](US-13.6-bridge-claim-withdrawal-step.md) — fixes the destination-chain claim/withdrawal step it owns (AVAIL, Polygon zkEVM); reuses the async-claim state-machine fixture.
- Protects every external-bridge story (US-13.2–13.5, US-13.7) indirectly: error containment keeps one broken route from taking down the shared bridge surface.

### What we explicitly did NOT do

- No in-house XCM reimplementation as a fallback to ParaSpell — that would violate AD-18. The posture is detect-fix-and-contain (route smoke check + claim-path fixes + per-route error containment), not build-our-own. Trigger to revisit: ParaSpell ceasing to track the ecosystem (the #4405 evaluation outcome).
- No new bridge provider — this story repairs and hardens existing routes (Across, Avail, Polygon, Snowbridge); new providers stay in their own stories (US-13.8–13.10).

### Points justification

5 pts — §3a-bis. This is no longer a single-concern hardening story (which would
calibrate at 3); it spans **multiple providers' reliability** across two seams —
the ParaSpell/runtime construction seam (#4405, #4424) and the
claim/error/fee-re-query finalization seam across Avail, Polygon zkEVM, Across,
and Snowbridge (#4108/#4636/#4756/#4255/#4798/#4622). That is broader than the
"1 internal integration" (3 pt) band and crosses into multi-provider reliability
work, but it adds **no new external system** — it repairs already-built routes —
so it sizes at 5, below the multi-system-integration (8 pt) band. Still one story
because all eight issues share one logic: keep the already-built bridge working
end to end.
