## Goal

Make the wallet's heavy list screens render fast and stay smooth on large
datasets, so scrolling and opening them feels instant rather than janky. The
target surfaces are the long, item-dense lists a user hits constantly — NFT,
Receive Token, the customization network list, Select Token, and Select Network.
When this lands, list size stops being a performance cliff on these screens.

## Background

This is the list/render facet of the umbrella performance program
([#4197](https://github.com/Koniverse/SubWallet-Extension/issues/4197)). The
concrete defect is
**[#2245](https://github.com/Koniverse/SubWallet-Extension/issues/2245)**: list
performance is poor on the NFT, Receive Token, customization-network, Select Token,
and Select Network screens — these render large item sets synchronously and jank
when the list grows. Issue
**[#2549](https://github.com/Koniverse/SubWallet-Extension/issues/2549)** is the
generic "optimize extension UI performance" umbrella (no body of its own) and is
referenced here as the broad UI-perf parent that #2245 is a concrete instance of.

The fix is render-path optimization: virtualize the heavy lists, paginate where the
dataset is unbounded, and avoid synchronous full-list renders so the work scales
with what's on screen, not with the total item count. Where these screens read
slow-changing config (e.g. network/token lists), they should read from the
static-data cache ([AD-23](../../ARCHITECTURE.md#architecture-decisions)) rather
than recomputing per render. This story owns no FR (cf. the FR-less
defending-story pattern of
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)); it defends list-render cost — which the PRD does **not** state (NFR-17 is about `extension-base` background logic running across hosts, not render performance; recorded as a PRD gap in US-21.2). Its one real anchor is NFR-21: list screens read slow-changing config from the static-data cache rather than re-computing per render
(cross-platform portability — the same render path runs across surfaces) on the
heavy list screens specifically.

## Acceptance criteria

- [ ] **AC-1** — **Given** a large dataset on the NFT / Receive Token / customization-network / Select Token / Select Network screen, **When** the screen opens and is scrolled, **Then** it renders responsively (virtualized: only visible rows mounted), with no synchronous full-list render (#2245).
- [ ] **AC-2** — **Given** an unbounded list (e.g. many networks/tokens), **When** it loads, **Then** it paginates or windows the data so render cost scales with what is on screen, not the total item count.
- [ ] **AC-3** — **Given** these screens read slow-changing network/token config, **When** they render, **Then** they read from the static-data cache (AD-23), not a live per-render recompute/RPC sweep.
- [ ] **AC-4** — **Given** an empty or filtered-to-empty list, **When** it renders, **Then** it shows the empty state immediately without first rendering and discarding the full list.

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — slow-changing list config (networks, tokens, priority lists) is served from the static-data cache via `fetchStaticData`; these screens read the cache rather than recomputing per render.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Sibling [US-20.6](US-20.6-webapp-and-web-runner-performance.md) — both own render performance and share the **render-cost benchmark** harness; US-20.5 targets the extension's selection/collection screens, US-20.6 the WebApp/web-runner web surfaces.

### Performance budget

- Heavy lists virtualized + paginated; no synchronous full-list render on the five target screens.
- Render cost scales with on-screen items, not total item count.
- Measured by the render-cost benchmark (TASK-20.5.5).
- Story PR description must explicitly confirm this budget is met.

### What we explicitly did NOT do

- No change to *what* these lists contain — NFT content stays with [EPIC-9](../epics/EPIC-9.md), token/network lists with their owning epics; this story only optimizes the render path.
- No WebApp/web-runner work — that web-surface render concern is [US-20.6](US-20.6-webapp-and-web-runner-performance.md), kept separate (different surface, different runtime).

### Points justification

5 pts — §3a-bis "production deliverable": virtualization + pagination + static-cache
reads across five concrete screens plus a render benchmark. Bounded, single-surface
render optimization — a 5, below the multi-system 8. P2: real jank but not a
blocking freeze.

## Implementation notes

**Audited 2026-07-13 (US-21.2 follow-up). Zero work is attributable to this story — but
two of its four ACs are *already true* through pre-existing code, so "backlog" needs
this footnote to be honest.**

- **AC-1 virtualization — definitively not met, and there is a trap here.** No
  virtualization library is used anywhere: no `react-window` / `react-virtualized` /
  `virtuoso` / `rc-virtual-list` in any workspace `package.json` (they exist only as
  transitive deps inside `@subwallet/react-ui`). The list stack is `SwList` →
  `flatlist-react` `PlainList`. **`renderOnScroll={true}` looks like windowing and is
  not**: `ScrollRenderer` does `list.slice(0, index)` with a monotonically growing
  index and **never unmounts scrolled-past rows** — progressive append, not
  virtualization. It is also enabled on only 1 of the 5 target screens (NFT); the other
  four render every row synchronously via `dataList.map(...)`.
- **Dead-code trap:** `hooks/common/useLazyList.ts` (`TOKENS_PER_PAGE = 10`) looks
  exactly like the pagination primitive AC-2 wants. It has **zero consumers**. Do not
  mis-credit it.
- **AC-2 bounded render cost — not met** (only the NFT screens bound the *initial*
  paint; the mounted-row count still grows to the full list on scroll).
- **AC-3 static-data cache — already true.** `ChainService` serves slow-changing config
  via `fetchStaticData` (AD-23) and the screens read it from the redux store, not per
  render. Pre-existing, unclaimed.
- **AC-4 fast empty state — already true**, and it is *library* behaviour:
  `PlainList` short-circuits on an empty list before any row render.

Its cited issues `#2245`, `#2549` and umbrella `#4197` have **zero commits**. The
headline demand (real virtualization across five screens + a render-cost benchmark) is
entirely unstarted.
