## Goal

Backfill the project's commit and version history into the doc layers so that
every `done` story has `version_shipped`, every CHANGELOG entry carries a real
commit SHA, version-progression entries exist for every intermediate version,
and VERSION is bumped to the current release (1.3.83) to reflect the current state. Downstream epics
and the `koni-docs validate` gate depend on this history being complete.

## Background

At story start the docs tree had 175 stories across 21 epics, a canonical
`docs/CHANGELOG.md` covering all 302 releases through 1.3.82 (converted from the
root file with real SHAs), and a VERSION file still at `1.3.79`. The gaps:

- 172 story files have empty `version_shipped` and `commit` — they are retroactive
  stubs awaiting version reconciliation.
- The three `done` stories carried **wrong** metadata (versions the commits are not
  contained in; one pointed at the skipped 1.3.81 release while unreleased).
- The VERSION file and PRD said 1.3.79 while the product shipped through 1.3.82.
- Epic statuses in PRD.md may be stale relative to actual story statuses.

This story closes all those gaps. It depends on US-21.1 for the `assignee` mapping
and feeds into US-21.3 for the final conformance check.

## Dev notes — Architecture constraints

- RULE-16: `version_shipped` is bare semver — never `v`-prefixed. `0.3.1` ✅ — `v0.3.1` ❌.
- RULE-1: VERSION + CHANGELOG in same commit. Every time a version is bumped, the CHANGELOG entry goes in the same commit.
- RULE-2: CHANGELOG commit SHA mandatory, never `pending`.
- Use `git tag --list` to find version tags, then `git log --oneline tag1..tag2` to find commits per version.
- Tags alone are not proof: use `git merge-base --is-ancestor <sha> v<X.Y.Z>` to confirm containment, and treat 1.3.81 as never released (skipped due to a release issue) — nothing ships in 1.3.81.

## Verification commands

```bash
# Check all SHAs are resolved
npx koni-docs backfill-commits --dry-run

# Check the full validate passes
npx koni-docs validate

# Verify VERSION file
cat VERSION

# Check no story has "pending" in version_shipped
grep -r "version_shipped:" docs/sprints/stories/ | grep -i "pending" || echo "✅ No pending versions"

# Every done story's commits are contained in the right anchor for its version SPACE.
# Anchor rules: extension → tag v<ver>; if the tag is missing (v0.2.1, v1.0.1, …) →
# the release commit from docs/CHANGELOG.md; if that is missing too (the six earliest
# Koni releases) → the earliest existing tag >= ver. webapp → the
# "[CI Skip] release/stable <ver>" commit on origin/webapp (that lineage has no tags).
for f in docs/sprints/stories/*.md; do
  [ "$(grep -m1 '^status:' "$f" | awk '{print $2}')" = done ] || continue
  v=$(grep -m1 '^version_shipped:' "$f" | awk '{print $2}')
  space=$(grep -m1 '^version_space:' "$f" | awk '{print $2}')   # empty = extension
  if [ "$space" = webapp ]; then
    anchor=$(git log --format='%H %s' origin/webapp --grep="^\[CI Skip\] release/stable $v\$" | head -1 | cut -d' ' -f1)
  elif git rev-parse -q --verify "v$v" >/dev/null; then
    anchor="v$v"
  else
    anchor=$(awk -v v="$v" '$0 ~ "^## \\["v"\\]" {f=1} f && /^\*\*Commit\*\*:/ {print $2; exit}' docs/CHANGELOG.md)
    [ -n "$anchor" ] || anchor=$(git tag --list 'v*' --sort=v:refname | awk -v v="v$v" '$1 >= v {print; exit}')
  fi
  for sha in $(grep -m1 '^commit:' "$f" | sed 's/^commit://; s/,/ /g'); do
    git merge-base --is-ancestor "$sha" "$anchor" || echo "MISMATCH: $f ($sha not in $anchor)"
  done
done
```

Closing it needed two things the earlier passes had wrong:

**1. The web app is not another repo — it is another *lineage of this one*, with a
colliding version series.** `webapp` / `webapp-dev` build the web app; `master` /
`subwallet-dev` build the extension. `origin/master` is not an ancestor of
`origin/webapp`. The web app is **untagged** — its releases are
`[CI Skip] release/stable X.Y.Z` commits — and it keeps its own CHANGELOG on that
branch. **Extension v1.3.56 ≠ web-app 1.3.56.** So `version_shipped` alone is
ambiguous, and stories now declare the space (default = extension):

```yaml
version_shipped: 1.3.56
version_space: webapp
```

[US-14.2](US-14.2-fiat-off-ramp-sell-crypto-for-fiat.md) (Transak off-ramp) is the
first user: `done`, **web-app 1.3.56**, anchor = release commit `c1301536b2`. Written
up in [AGENTS.md](../../../AGENTS.md) §7 so nobody re-derives it. (The extension port
of the off-ramp exists on `koni/dev/issue-3843` — unmerged, in no tag.)

**2. A "shipped" FR can stop being shipped.** The PRD had no way to say so, so two
capabilities that are *gone from the product* read `✅ shipped`. Added
`⏸️ withdrawn` to the status key — *was in the product, since removed; the story
keeps whatever status its delivery earned*:

- **FR-120** Interlay lending — shipped 1.1.36, **removed in 1.2.12** ("Remove
  Interlay lending", #3226). Its story stays `done`: it really was delivered.
- **FR-157** NFT-mint campaigns — ran on the web-app Earning Dashboard as a CD
  deploy during the Oct–Nov 2023 campaign; it is **in no numbered release of either
  product** and the surface has since been deleted. Story stays `backlog`.

[US-19.6](US-19.6-nft-mint-campaigns.md)'s old note claimed "no candidate commit is
contained in any extension tag" — **that was false**, and worth remembering: the mint
commits *are* ancestors of every tag from v1.1.36, so a naive `git tag --contains`

injects `NFT_MINTING_HOST`. **Containment is necessary, not sufficient — when
