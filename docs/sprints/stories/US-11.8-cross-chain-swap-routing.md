## Goal

A user can swap between tokens that don't live on the same chain ‚Äî the wallet
automatically composes a swap with a bridge (Swap‚ÜíBridge or Bridge‚ÜíSwap) and
guides them through the multi-step route ‚Äî so that they reach pairs no single DEX
lists.

## Background

This is the routing layer that sits **above** the per-provider handlers (e.g.
[US-11.2](US-11.2-in-wallet-swap-via-hydration-dex.md) and the other provider
stories US-11.3‚Äì11.11). When source and destination are on different chains, the
SwapService composes a route that chains a same-chain swap with a bridge step (XCM
or an external bridge, delegated to ParaSpell per
[AD-18](../../ARCHITECTURE.md#architecture-decisions)) and tracks each hop to
completion. It is one FR but a large capability ‚Äî and it has distinct failure
semantics: a half-completed multi-hop must be **resumable**, never silently lost
between hops. That is why it is a separate, larger story (8 pts ‚Äî a multi-system
integration across the swap and bridge engines) and not folded into a provider story.

Materializes [FR-110](../../PRD.md#functional-requirements). **Retroactive** ‚Äî already shipped.

## Acceptance criteria

- [x] **AC-1** ‚Äî **Given** a cross-chain pair, **When** the user requests a swap,
  **Then** the wallet returns a multi-step route (swap + bridge) showing each
  step's direction, fee and estimated time before any signature.
- [x] **AC-2** ‚Äî **Given** a confirmed route, **When** the user executes it, **Then**
  each hop is submitted and tracked in order, and the UI shows per-step status
  (pending / done / failed).
- [x] **AC-3** ‚Äî **Given** a route that stalls after the first hop, **When** the
  user returns, **Then** the partial route is recoverable / resumable and clearly
  surfaced ‚Äî funds are never reported as lost.
- [x] **AC-4** ‚Äî **Given** routing, **When** a step needs a bridge, **Then** it
  reuses the bridge engine (AD-18 / EPIC-13), not a bespoke per-pair path.
