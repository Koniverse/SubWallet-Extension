## Goal

A user can create and manage a native Substrate multisig account by choosing a set
of signatories and a threshold — and the multisig address is derived **off-chain,
deterministically**, with no account-creation transaction and no server — so that a
group gains shared M-of-N custody instantly and reproducibly. This is the account
that [US-18.2](US-18.2-pending-transaction-detection-and-approval.md) approves
transactions for.

## Background

This is the **account model** pillar of [EPIC-18](../epics/EPIC-18.md). On native
Polkadot/Substrate chains, a multisig address is a pure function of its **signatory
set + threshold** ([AD-17](../../ARCHITECTURE.md#architecture-decisions)): the same
inputs always derive the same address, computed entirely client-side. There is no
"create multisig" extrinsic to pay for and wait on — the account simply *exists* the
moment the signatories and threshold are chosen. That is the property this story
delivers and protects.

The multisig address holds **no private key of its own**
([AD-04](../../ARCHITECTURE.md#architecture-decisions)); authority is the signatory
set, each of whom signs with their own keyring or hardware key. This story owns
creating the multisig, listing/managing it, and surfacing its signatories and
threshold — it does **not** own collecting approvals (that is
[US-18.2](US-18.2-pending-transaction-detection-and-approval.md)) or building/submitting
the underlying transaction ([EPIC-8](../epics/EPIC-8.md)). It is also distinct from the
proxy model ([EPIC-17](../epics/EPIC-17.md)): a multisig is M-of-N shared authority,
not a delegated single-signer authority.

Materializes [FR-149](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** a chosen set of signatories and a threshold M-of-N,
  **When** the user creates the multisig, **Then** the multisig address is derived
  off-chain (client-side, deterministically) with **no on-chain creation transaction**
  and no server call (AD-17).
- [x] **AC-2** — **Given** the same signatory set and the same threshold re-entered on
  a fresh install, **When** the multisig is recreated, **Then** the **identical**
  address is reproduced (deterministic derivation, no server).
- [x] **AC-3** — **Given** a created multisig, **When** the user opens account
  management, **Then** the multisig is listed with its signatory set and threshold,
  and is usable as a normal account context (view balance, initiate a transaction).
- [x] **AC-4** — **Given** an invalid configuration (threshold greater than the number
  of signatories, or a duplicate/zero signatory set), **When** the user attempts to
  create, **Then** creation is rejected with a clear validation error and no
  malformed multisig is added.

## Dev notes

### Architecture constraints

- [AD-17](../../ARCHITECTURE.md#architecture-decisions) — the multisig address is
  derived off-chain from the signatory set + threshold; creation needs **no on-chain
  transaction** and no backend service.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) — the multisig address holds
  no key of its own; the keyring gains no "multisig key", and each signatory signs
  with its own key. This story introduces no new AD entries.

### Cross-story dependencies

- Required by [US-18.2](US-18.2-pending-transaction-detection-and-approval.md) — that
  story detects and approves pending transactions *for* the multisig account this
  story creates.
- Relates to [EPIC-3](../epics/EPIC-3.md) — signatories are ordinary (single-key or
  hardware) accounts; this story composes them, it does not re-implement key custody.

### What we explicitly did NOT do

- No on-chain "create multisig" transaction — Phase-1 multisig is off-chain by design
  (AD-17); on-chain activation only happens implicitly when the first transaction
  executes.
- No auto-detection of already-activated multisig accounts — deferred to
  [US-18.3](US-18.3-auto-detection-indexer-history-and-optimization.md) (Phase 2).
