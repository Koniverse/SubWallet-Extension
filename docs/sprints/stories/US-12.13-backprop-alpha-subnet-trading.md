## Goal

A user can access the Backprop alpha-subnet trading integration from the wallet —
an active subnet-trading strategy over Bittensor alpha — so that holders can put
alpha into the Backprop strategy as an earning option alongside passive subnet
staking.

## Background

Backprop is a third-party alpha-subnet *trading* strategy (as opposed to passive
hold-and-earn): it actively moves alpha across subnets to capture yield. SubWallet
integrates its entry/exit path as an earning option. Staking/position moves run
the native Subtensor path per
[AD-15](../../ARCHITECTURE.md#architecture-decisions); the position is modeled on
the handler tree ([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)). Because the trading logic
and any partner endpoint are external, this story carries a partner external
dependency.

Materializes [FR-126](../../PRD.md#functional-requirements). 📋 **Planned / forward** — not yet shipped;
this story scopes the planned integration. Tracked by
[#4880](https://github.com/Koniverse/SubWallet-Extension/issues/4880) —
Integration: Backprop (Alpha Subnet Trading), under the
[#4837](https://github.com/Koniverse/SubWallet-Extension/issues/4837) — Integrate
Alpha Ecosystem DeFi Projects (Liquid Stake Alpha Bridge, Trusted Stake, Backprop)
umbrella.

## Acceptance criteria

- [ ] **AC-1** — **Given** the Backprop product, **When** the user enters the
  strategy with an amount, **Then** the entry submits via the native path and a
  Backprop position appears in the earning list.
- [ ] **AC-2** — **Given** a Backprop position, **When** the user exits, **Then**
  the position is closed and the proceeds are returned to the user's account.
- [ ] **AC-3** — **Given** the Backprop strategy/endpoint is unavailable, **When**
  the user opens the flow, **Then** the option is shown as temporarily unavailable
  rather than entering on stale strategy state.

## Dev notes

### Architecture constraints

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — native Subtensor path for the alpha position moves.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — Backprop is a handler-backed position; the screen consumes its subjects.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.6](US-12.6-bittensor-dtao-subnet-staking.md) — reuses the dTAO native-staking foundation.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the (planned) handler subjects.

### What we explicitly did NOT do

- No in-wallet subnet *AMM swap* — TAO↔alpha and cross-subnet alpha swap is owned by [EPIC-11](../epics/EPIC-11.md) (FR-109). Backprop consumes those primitives as a strategy; it does not reimplement the swap.

### Points justification

5 pts — §3a-bis "1 external-system integration": integrating a single
partner-owned trading strategy (enter/exit, position tracking) on the existing
native path. One external partner integration sizes at 5; the partner dependency
is surfaced in `external_deps`.
