## Goal

Multi-chain aggregate data — balance, fee, swap, XCM, NFT — is fetched from the SubWallet
Services SDK backend instead of being computed on-device by fanning out to every chain, so
the extension stops paying per-chain RPC cost for data a backend can aggregate once.

## Background

Carved out of [US-20.2](US-20.2-api-call-optimization.md) on 2026-07-14 — see
[CONTEXT D100](../../CONTEXT.md). This is
**[#4465](https://github.com/Koniverse/SubWallet-Extension/issues/4465)**, shipped
**1.3.52**: the in-repo `packages/subwallet-api-sdk` was **deleted** and aggregated data
routed through the external `@subwallet-monorepos/subwallet-services-sdk` across **18 call
sites**, with per-service error handling (swap, Cardano, Across bridge).

**This is what actually satisfies [NFR-20](../../PRD.md#non-functional-requirements) and
[AD-24](../../ARCHITECTURE.md#architecture-decisions)** — the requirement had been on the
books with no story materializing it. Its CHANGELOG bullet, *"Apply sw version when request
to Backend"*, badly undersells what landed: a release note is a claim, not evidence
([LESSONS §64](../../LESSONS.md)).

**Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** aggregate multi-chain data (balance, fee, swap, XCM, NFT),
  **When** it is requested, **Then** it is fetched through the Services SDK backend rather
  than computed on-device from a per-chain fan-out (AD-24, NFR-20).
- [x] **AC-2** — **Given** the in-repo `packages/subwallet-api-sdk`, **When** the migration
  lands, **Then** it is **deleted** and all 18 call sites consume
  `@subwallet-monorepos/subwallet-services-sdk` — there is one SDK, not two.
- [x] **AC-3** — **Given** an SDK-backed service fails (swap, Cardano, Across bridge),
  **When** the error surfaces, **Then** it carries a service-specific message the caller can
  act on, not a bare rejection.

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — aggregate balance / fee / swap / XCM / NFT data is served by the Services SDK backend, not computed on-device. **This story is what makes AD-24 true in the code.**
- This story does NOT introduce new AD entries.

### What shipped is narrower than NFR-20 claims

**"No direct per-chain fan-out" is not fully true.** `SubscanService` still calls
`https://{chain}.api.subscan.io` directly, as does the bridge-indexer path. NFR-20 is
satisfied for the *aggregate* surfaces the SDK covers — **not app-wide**. State it here
rather than letting the NFR imply a property the tree does not have.

### Cross-story dependencies

- Sibling [US-20.8](US-20.8-api-request-strategy-v2.md) — the other half of the shipped request economy: the client-side strategy for the indexer traffic that did **not** move to the SDK.
- Forward work: [US-20.2](US-20.2-api-call-optimization.md) (`backlog`) — in-flight dedup, an app-wide cap, and the unbounded notification loop, none of which the SDK migration addresses.

### Points justification

5 pts — §3a-bis "one external-system integration": moving 18 call sites onto an external
backend SDK and retiring the in-repo one, with per-service error semantics.
