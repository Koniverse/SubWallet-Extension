## Goal

Keep the dependency tree auditable and the shipped bytes trustworthy. The Yarn 3
lockfile must be committed and integrity-checked, every core-wallet dependency
must resolve from the public npm registry (no private/untrusted registries), there
must be no untrusted or duplicate dependencies, and CI must fail closed on a
drifted lockfile or a flagged advisory. When this holds, a reviewer can trust that
what ships is exactly what the lockfile pins — no silent registry swap, no
unaudited transitive dep.

## Background

This is the supply-chain facet of the umbrella performance/quality program; the
umbrella tracker is [#4197](https://github.com/Koniverse/SubWallet-Extension/issues/4197).
Unlike the other EPIC-20 stories, this one has **no dedicated GitHub issue anchor**
— it is a standing policy story that materializes NFR-19 ("Dependency auditability:
Yarn 3 lock file committed; all packages published to npm; no private registry
dependencies for core wallet logic"). It rides the Yarn 3 monorepo boundaries
([AD-05](../../ARCHITECTURE.md#architecture-decisions)) and Webpack bundle-splitting
([AD-06](../../ARCHITECTURE.md#architecture-decisions)) owned by [EPIC-1](EPIC-1.md);
it anchors no new AD of its own.

Some of this posture is **already in place** (a committed `yarn.lock`, an
npm-published dependency set) — so those parts are recorded **retroactively**, and
the forward work is the *enforcement*: turning the posture into a CI gate that fails
on regression rather than relying on convention. This story owns no FR (cf. the
FR-less defending-story pattern of
[US-12.15](US-12.15-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)); it defends NFR-19.

## Acceptance criteria

- [ ] **AC-1** — **Given** the repository, **When** CI runs, **Then** it fails if `yarn.lock` is missing, out-of-sync with `package.json`, or modified by an install (lockfile integrity / `--immutable`) (NFR-19). *(Committed lockfile: retroactive; the CI gate: forward.)*
- [ ] **AC-2** — **Given** the dependency set, **When** it is resolved, **Then** every core-wallet dependency comes from the public npm registry — no private/untrusted registry sources (NFR-19). *(npm-only set: retroactive; the assertion: forward.)*
- [ ] **AC-3** — **Given** the dependency tree, **When** it is audited, **Then** there are no untrusted dependencies and duplicate dependencies are flagged/deduped, asserted in CI.
- [ ] **AC-4** — **Given** a known-vulnerable advisory in the tree, **When** CI runs the audit gate, **Then** it fails on a flagged advisory (above an agreed severity threshold).

## Tasks

- [ ] **TASK-20.7.1** — CI lockfile-integrity gate: `yarn install --immutable` (or equivalent) fails on a drifted/out-of-sync `yarn.lock` (AC: 1)

## Dev notes

### Architecture constraints

- [AD-05](../../ARCHITECTURE.md#architecture-decisions) — the Yarn 3 monorepo package boundaries are the substrate this hygiene rides on; the lockfile pins the whole workspace.
- [AD-06](../../ARCHITECTURE.md#architecture-decisions) — Webpack bundle-splitting determines the shipped bytes; an unaudited dep silently changes them, which is why the gate matters.
- This story does NOT introduce new AD entries — it enforces NFR-19 over the existing AD-05/AD-06 substrate.

### Cross-story dependencies

- No code-artifact dependency on other EPIC-20 stories — this is a CI/policy story. It is the epic's supply-chain invariant and is referenced from the EPIC-20 cross-cutting invariants.

### Performance budget

- Not a runtime budget — the "budget" here is CI: `yarn.lock` clean, npm-only registry, audit gate green. (Listed in the epic's performance/invariants table as the lockfile-integrity invariant.)
- Story PR description must confirm the CI gate is wired and fails closed.

### What we explicitly did NOT do

- No dependency-version *upgrades* as part of this story — it is about *auditability and gating*, not bumping versions; a flagged advisory triggers a separate remediation.
- No change to the Yarn 3 / Webpack substrate (AD-05, AD-06) — owned by [EPIC-1](EPIC-1.md); this story only adds the enforcement gate on top.

### Retroactive vs forward

The committed `yarn.lock` and the npm-only dependency set are **already in place**
(retroactive — record, do not re-implement). The forward work is the **CI
enforcement**: the immutable-install gate, the registry assertion, the
duplicate/untrusted check, and the advisory audit gate that fail closed on
regression.

### Points justification

3 pts — §3a-bis "multi-doc bundle OR 1 internal integration": wiring a set of CI
gates (immutable install, registry assertion, dedupe/audit) over an existing,
mostly-in-place posture. Smaller than the 5-pt runtime-optimization stories because
much of the posture is retroactive and the work is CI configuration, not
application-runtime change. P2/P3: important hygiene, not a user-facing defect.
