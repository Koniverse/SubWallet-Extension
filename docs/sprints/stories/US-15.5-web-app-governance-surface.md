## Goal

A user of SubWallet's standalone web app gets the **full OpenGov governance
experience** — referenda browse + detail, conviction voting (vote / revote /
unvote), locked-token detail + unlock, and delegation + tracks — inside the
browser app, not only the extension popup. Governance reaches parity across
surfaces so a user who lives in the web app never has to open the extension to
participate in chain governance.

## Background

EPIC-6 ships the standalone web app and its feature-parity guarantee
([FR-64](../../PRD.md#functional-requirements), owned by [EPIC-6](../epics/EPIC-6.md)); this story brings
the *governance surface* into that shell. Per
[NFR-17](../../PRD.md#non-functional-requirements), the same
`@subwallet/extension-base` background logic runs in the extension service
worker, the web app, and the mobile web-runner — so the web-app governance
surface **re-renders** the OpenGov flow against the same RxJS subjects and
background signing path, it does not re-implement it.

This is the **planned / forward** Phase-2 capstone (FR-143, `📋 planned`): it
brings the whole of US-15.1 (referenda + voting), US-15.2 (locked-token + unlock),
and US-15.4 (delegation + tracks) to the web app. Referenda/track data is
aggregated through the Services SDK ([AD-24](../../ARCHITECTURE.md#architecture-decisions));
market/metadata and chain-list data the surface needs is fetched through
SubWallet's cache/CDN proxies ([AD-25](../../ARCHITECTURE.md#architecture-decisions))
rather than keyed upstreams; vote/unlock/delegate extrinsics are signed in the
web-app background context and surfaced via the typed `pri()` bus
([AD-03](../../ARCHITECTURE.md#architecture-decisions)). Because it depends on the
full extension governance flow, this story is the **last** in the epic to land.

Materializes [FR-143](../../PRD.md#functional-requirements). This story is **📋 planned** (forward) — not
yet shipped. Tracked by
[#4677](https://github.com/Koniverse/SubWallet-Extension/issues/4677) —
[OpenGov][Webapp] Support Governance feature.

## Acceptance criteria

- [ ] **AC-1** — **Given** the standalone web app, **When** a user opens
  governance, **Then** they can browse OpenGov referenda (list + detail) and cast
  / revote / unvote a conviction vote, with the same behaviour as the extension
  (US-15.1).
- [ ] **AC-2** — **Given** a conviction-vote lock in the web app, **When** the
  user opens locked-token detail, **Then** they can see the lock and unlock it
  once the period elapses (US-15.2), with the not-yet-unlockable guard.
- [ ] **AC-3** — **Given** a multi-track OpenGov chain in the web app, **When**
  the user delegates / undelegates a track, **Then** the Phase-2 delegation +
  tracks flow works as in the extension (US-15.4).
- [ ] **AC-4** — **Given** the web-app governance surface, **When** it renders,
  **Then** it consumes the same `extension-base` governance subjects and signing
  path as the extension (NFR-17) — no governance logic is forked into the web app.
- [ ] **AC-5** — **Given** a web-app session where the user is not yet
  unlocked/authenticated or has no governance-enabled chain selected, **When**
  they open governance, **Then** an explicit gated / empty state is shown rather
  than a blank or errored surface.

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — referenda/track/delegation data is aggregated through the Services SDK backend; the web-app surface consumes the same feed as the extension.
- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — market/metadata/chain-list data is fetched through SubWallet's cache/CDN proxies (`api-cache` / `static-data`), never a keyed upstream in the web-app bundle.
- [AD-03](../../ARCHITECTURE.md#architecture-decisions) — vote/unlock/delegate extrinsics are signed in the web-app background context and surfaced via `pri()`; no key material in the web-app UI.
- This story does NOT introduce new AD entries. It renders governance into the web-app shell owned by [EPIC-6](../epics/EPIC-6.md) (FR-64); it does not build the shell.

### Cross-story dependencies

- Builds on [US-15.1](US-15.1-opengov-referenda-and-conviction-voting.md), [US-15.2](US-15.2-locked-token-detail-and-unlock-flow.md), [US-15.4](US-15.4-opengov-delegation-and-governance-tracks.md) — re-renders their components and signing path in the web app; this is why it lands last.
- Builds on [EPIC-6](../epics/EPIC-6.md) — renders into the standalone web-app shell / parity layer (FR-64); EPIC-6 owns the shell, this story owns the governance surface within it.

### What we explicitly did NOT do

- No forked web-app governance logic — the surface re-renders the shared `extension-base` flow (NFR-17). Trigger to revisit: a web-app-only governance affordance with no extension equivalent.
- No new governance capability beyond US-15.1/15.2/15.4 — this story is a surface port, not a feature expansion.
