## Goal

A user can bridge assets between Polygon and Ethereum directly in the wallet via
the Polygon Unified Bridge — seeing the fee and ETA before signing and finalizing
arrival with a claim step on the destination — so that Polygon-ecosystem value
can move to and from Ethereum without an external bridge UI.

## Background

The Polygon Unified Bridge is an external Polygon ↔ Ethereum bridge that hangs
off the bridge-hop interface from [US-13.1](US-13.1-xcm-parachain-transfers.md)
and the provider-handler fixture from
[US-13.2](US-13.2-snowbridge-eth-asset-hub.md). Like the Avail bridge it is
**asynchronous** with a destination-chain claim/withdrawal step — that step is
owned by [US-13.6](US-13.6-bridge-claim-withdrawal-step.md), which this story
consumes (FR-131 names Avail and Polygon as the bridges that require the claim
step). Bridge construction follows the delegate posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the provider is reached
through the SubWallet proxy ([AD-19](../../ARCHITECTURE.md#architecture-decisions),
NFR-16).

Materializes [FR-129](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a Polygon-Unified-Bridge-supported asset, **When** the
  user requests a bridge between Polygon and Ethereum, **Then** the wallet shows
  the fee, estimated destination amount, and ETA before any signature.
- [x] **AC-2** — **Given** a confirmed bridge, **When** the user signs the
  source-chain transaction, **Then** it is submitted and tracked, and the bridge
  enters the in-transit / claim-pending state.
- [x] **AC-3** — **Given** a bridge whose withdrawal window has not completed,
  **When** the user returns, **Then** it is shown as in-transit (not failed),
  funds are never reported as lost, and once withdrawable the destination claim
  becomes available (handed to US-13.6).
- [x] **AC-4** — **Given** the Polygon Unified Bridge provider, **When** the
  wallet calls it, **Then** the call goes through the SubWallet proxy and no
  provider key is in the bundle (AD-19 / NFR-16).

## Tasks

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction delegated; the Polygon handler consumes the provider.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — provider proxied; no key in the bundle (NFR-16).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) (bridge-hop interface) and [US-13.2](US-13.2-snowbridge-eth-asset-hub.md) (provider-handler fixture).
- Required by / consumes [US-13.6](US-13.6-bridge-claim-withdrawal-step.md) — the destination-chain claim/withdrawal step that finalizes this async bridge.
- Sibling [US-13.3](US-13.3-avail-bridge.md) — also async-with-claim; shares the async-claim state machine.

### Points justification

5 pts — §3a-bis "1 external system integration": one external bridge provider
(Polygon Unified Bridge) behind the existing interface. The async withdrawal
handoff reuses the US-13.6 state machine, keeping it a single-provider
integration at 5.
