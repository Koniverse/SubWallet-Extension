## Goal

A power user can set up delegated authority on a Substrate account: add a proxy
account, choose which named Polkadot proxy-pallet type it holds (Any,
NonTransfer, Governance, Staking, …), and see and manage that delegated-authority
relationship. This is the *model* half of proxy — it establishes who may sign for
whom and under which type, so that the signing-time story
([US-17.2](US-17.2-proxy-signing-sign-selector-and-proxied-by-display.md)) has a
well-defined authority to enforce.

## Background

Proxy is the Polkadot `proxy` pallet's delegated-authority model: an account can
authorise another account to dispatch certain calls on its behalf, scoped to a
named proxy *type* that limits which calls are allowed (Any permits everything;
NonTransfer excludes balance transfers; Governance and Staking scope to those
domains; and so on). This story owns the account/authority **model**: adding a
proxy to a Substrate account, choosing the proxy type, and managing (viewing,
removing) the relationship. Per [AD-16](../../ARCHITECTURE.md#architecture-decisions)
the wallet supports **only the named pallet types** (no custom proxy types) and
**only Substrate software accounts** (no EVM-solo, no Ledger-EVM).

FR-150 is one functional requirement deliberately split into two stories by
capability. This story (US-17.1) is the **authority model** — the data and
management surface that says "account A is a Governance proxy for account B". The
sibling ([US-17.2](US-17.2-proxy-signing-sign-selector-and-proxied-by-display.md))
is the **signing-time behaviour** — the Sign Selector that, at the moment a
transaction is signed, picks which controlling account signs on behalf of the
proxied account and surfaces the proxied-by provenance. Splitting model from
signing keeps each story independently testable: this one can be verified by
authority CRUD against the pallet without exercising a live signing flow.

Materializes [FR-150](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [ ] **AC-1** — **Given** a Substrate account, **When** the user adds a proxy and
  picks a named proxy type (Any / NonTransfer / Governance / Staking / …),
  **Then** the delegated-authority relationship is established on-chain and shown
  in the wallet as "<proxy account> is a <type> proxy for <proxied account>".
- [ ] **AC-2** — Only the named Polkadot proxy-pallet types are offered; no custom
  proxy type can be entered, and only Substrate software accounts are eligible
  (EVM-solo and Ledger-EVM accounts are not offered the proxy action) — per
  [AD-16](../../ARCHITECTURE.md#architecture-decisions).
- [ ] **AC-3** — **Given** an existing proxy relationship, **When** the user views
  the account, **Then** all of its proxy relationships (as proxy and as proxied)
  are listed with their types, and a relationship can be removed.
- [ ] **AC-4** — **Given** an attempt to add a proxy on an unsupported account
  type or with an unrecognised proxy type, **When** the user submits, **Then**
  the wallet rejects it with a clear error and establishes nothing on-chain.

## Dev notes

### Architecture constraints

- [AD-16](../../ARCHITECTURE.md#architecture-decisions) — proxy is scoped to the
  *named* Polkadot proxy-pallet types (no custom types) and to Substrate
  accounts only; this story implements the type set and the eligibility gate.
- This story introduces no new AD entries; it materializes the model half of
  AD-16.

### Cross-story dependencies

- Required by [US-17.2](US-17.2-proxy-signing-sign-selector-and-proxied-by-display.md)
  — the Sign Selector enforces the authority/type relationship this story
  establishes; without the model there is no authority to check at signing time.
- Relates to the RequestService approval queue
  ([US-2.7](US-2.7-requestservice-approval-queue.md)) — the `addProxy` /
  `removeProxy` extrinsics are themselves signed through the existing sign queue.

### What we explicitly did NOT do

- No custom / arbitrary proxy types — only the named pallet types per AD-16.
  Trigger to revisit: a pallet upgrade that introduces a new named type.
- No EVM or hardware-wallet proxy accounts — Substrate software accounts only.
  Hardware-device signing is [EPIC-16](../epics/EPIC-16.md)'s transport.
