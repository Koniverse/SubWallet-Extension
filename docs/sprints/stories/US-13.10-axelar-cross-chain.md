## Goal

A user can bridge assets via Axelar directly in the wallet — seeing the fee and
ETA before signing — so that the broad set of chains Axelar connects becomes
reachable cross-chain without an external tool.

## Background

This is a **planned** (forward) external bridge provider. Axelar is a
general-message-passing cross-chain network; integrating it adds one more provider
handler behind the bridge-hop interface from
[US-13.1](US-13.1-xcm-parachain-transfers.md) and the provider-handler fixture
from [US-13.2](US-13.2-snowbridge-eth-asset-hub.md). Bridge construction follows
the delegate posture ([AD-18](../../ARCHITECTURE.md#architecture-decisions)) and
the provider is reached through the SubWallet proxy
([AD-19](../../ARCHITECTURE.md#architecture-decisions), NFR-16).

Materializes [FR-135](../../PRD.md#functional-requirements), which is `📋 planned` in the PRD — this story
is **forward / planned**, not retroactive. `commit` / `version_shipped` are set
when the work lands. _Speculative — no tracked roadmap issue found; inferred from
the PRD roadmap._ (The only Axelar issue on record is a swap-area request, not a
bridge integration, so it is not cited here.)

## Acceptance criteria

- [ ] **AC-1** — **Given** an Axelar-supported asset and route, **When** the user
  requests a bridge, **Then** the wallet shows the fee, estimated destination
  amount, and ETA before any signature.
- [ ] **AC-2** — **Given** a confirmed Axelar transfer, **When** the user signs,
  **Then** the source-chain transaction is submitted and tracked, and the
  destination balance updates on settlement.
- [ ] **AC-3** — **Given** an Axelar transfer in transit (or that stalls), **When**
  the user returns, **Then** it is shown as in-transit with recovery guidance —
  funds are never reported as lost.
- [ ] **AC-4** — **Given** the Axelar provider, **When** the wallet calls it,
  **Then** the call goes through the SubWallet proxy and no provider key is in the
  bundle (AD-19 / NFR-16).

- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — provider proxied; no key in the bundle (NFR-16).
