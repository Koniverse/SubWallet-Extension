## Goal

A Bitcoin dApp (Ordinals / inscriptions / BTC-DeFi site) connects to SubWallet,
reads the user's addresses, and asks the wallet to sign messages and PSBTs and to
send transfers. Bitcoin users get the same in-wallet dApp experience the EVM and
Substrate ecosystems already have.

## Background

Bitcoin dApps interact through a wallet provider object (injected as its own
namespace, per [AD-12](../../ARCHITECTURE.md#architecture-decisions)) exposing
`connect`, `getAddresses`, `signMessage`, `signPsbt`, and `sendTransfer`. SubWallet's
Bitcoin model exposes three address types per account (BIP44 Legacy / BIP84 Native
SegWit / BIP86 Taproot, default SegWit) because different BTC counterparties expect
different formats; the connector advertises them so a dApp can pick the one it
needs. Signing is **PSBT-based** (Partially Signed Bitcoin Transactions), the BTC
standard for handing a transaction to a wallet to sign specific inputs.

`connect` is authorization-gated (US-10.7); `signMessage` / `signPsbt` /
`sendTransfer` route to the Bitcoin request handler in `RequestService`
([AD-21](../../ARCHITECTURE.md#architecture-decisions)) behind the shared
`PopupHandler`; signing runs in the background keyring (AD-04). Bitcoin data /
broadcast goes through the indexer/proxy path (AD-19).

Materializes [FR-98](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** a Bitcoin dApp, **When** it calls `connect()` from an
  unauthorized origin, **Then** the request is surfaced for approval and addresses
  are returned only on approval.
- [ ] **AC-2** — **Given** a connected dApp, **When** it calls `getAddresses()`,
  **Then** it receives the authorized account's addresses with their type (Legacy /
  Native SegWit / Taproot).
- [ ] **AC-3** — **Given** a connected dApp, **When** it calls `signMessage` or
  `signPsbt`, **Then** the request routes to the Bitcoin request handler, is
  surfaced for approval, and is signed in the background keyring (PSBT inputs signed
  per the request).
- [ ] **AC-4** — **Given** a connected dApp, **When** it calls `sendTransfer`,
  **Then** the transfer is built, surfaced for approval, signed, and broadcast,
  returning a tx id.
- [ ] **AC-5** — **Given** an unauthorized origin or a user rejection, **When** the
  request resolves, **Then** it rejects with a clear error and no address, message
  signature, or PSBT is leaked or partially signed.

## Dev notes

### Architecture constraints

- [AD-12](../../ARCHITECTURE.md#architecture-decisions) — Bitcoin keyring exposes three address types; the BTC dApp provider is a separate inject namespace signing via PSBT.
- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — `signMessage` / `signPsbt` / `sendTransfer` route to the Bitcoin request handler behind the shared `PopupHandler`.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — signing runs in the background keyring.
- This story does NOT introduce new AD entries.

- Sibling [US-10.8](US-10.8-arbitrary-message-signing.md) — `signMessage` is the Bitcoin arm of arbitrary message signing.
