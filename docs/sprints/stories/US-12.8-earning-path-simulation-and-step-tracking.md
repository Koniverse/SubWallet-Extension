## Goal

Before a user commits to a stake that takes several on-chain steps (e.g.
deposit → bond → nominate, or token-approve → supply), the wallet simulates the
full path — showing each step, its fee and order — and then tracks each step's
status to completion, so that multi-step earning flows are legible and a flow
interrupted mid-path is recoverable rather than a mystery balance.

## Background

Many earning flows are not a single extrinsic: a liquid-stake may need an approve
then a mint; a cross-chain stake needs a deposit before the bond; a pool join may
need a create-then-join. The engine exposes the step plan and the per-step
submission through the handler tree
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)), with pool metadata and
fee inputs assembled through the Services SDK
([AD-24](../../ARCHITECTURE.md#architecture-decisions)) and static pool data
([AD-23](../../ARCHITECTURE.md#architecture-decisions)). This story owns the
*path-simulation and step-tracking surface* shared by every multi-step earning
flow — the simulate-then-track state machine and the resumable per-step status
UI. It is a cross-cutting plumbing story, not a per-protocol screen.

Materializes [FR-121](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** an earning action that requires multiple steps, **When**
  the user opens the flow, **Then** the wallet simulates and shows the ordered
  step list with each step's purpose and estimated fee before any signature.
- [x] **AC-2** — **Given** a confirmed multi-step path, **When** the user executes
  it, **Then** each step is submitted in order and the UI shows per-step status
  (pending / done / failed).
- [x] **AC-3** — **Given** a path that stalls or fails after the first step,
  **When** the user returns, **Then** the partial path is recoverable / resumable
  and clearly surfaced — funds are never reported as lost between steps.
- [x] **AC-4** — **Given** a step's simulation cannot be produced (e.g. pool data
  unavailable), **When** the user opens the flow, **Then** the path is shown as
  un-simulatable and execution is blocked rather than proceeding on guessed steps.

## Dev notes

### Architecture constraints

- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — the step plan and per-step submission come from the handler tree; this surface orchestrates them, it does not encode per-protocol steps itself.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — fee/quote inputs flow through the Services SDK backend.
- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — pool metadata used in simulation is read from the static-data cache.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the handler step plan + submission.
- Builds on [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) — the staking flows feed steps into this surface.
- Required by [US-12.9](US-12.9-xcm-deposit-routing-into-earning.md) — the XCM deposit is a step in this path harness.

### Performance budget

- Path simulation is shown before any signature; it must not block the UI thread on a synchronous RPC sweep — simulation inputs come from cached pool data (AD-23) + Services SDK (AD-24).
- Story PR description must confirm no live per-step RPC sweep on path open.

### Points justification

5 pts — §3a-bis "1 internal integration" at cross-cutting scope: a
simulate-then-track state machine spanning the handler tree, fee assembly and a
resumable per-step status UI shared by every multi-step earning flow. Bounded but
load-bearing plumbing — sizes at 5, and it stands up the multi-step path harness
US-12.9 reuses.
