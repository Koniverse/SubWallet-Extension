## Goal

A first-time user is guided through wallet setup with the safety rails that keep
them from losing access: a backup reminder before they leave the seed behind and a
terms-of-service acceptance gate, wrapped around the create/restore flow — so that
the moment a wallet exists, the user has acknowledged backup responsibility and
accepted the terms. This story owns the *onboarding UX wrapper*, not the key
generation underneath it.

## Background

When a user installs the extension they need a standardized first-run experience.
The actual create/restore keyring flow — mnemonic generation, restore-from-seed,
account derivation — is the **keyring/account** layer owned by
[EPIC-3](../epics/EPIC-3.md) and confined to the background service worker
([AD-04](../../ARCHITECTURE.md#architecture-decisions)). This story owns only the
*wrapper* the user walks through around that flow: the backup-reminder gate (so a
freshly created seed is not silently skipped) and the terms-of-service acceptance.
The fiat on-ramp / buy step that once lived in the old onboarding area was split
out to [EPIC-14](../epics/EPIC-14.md) and is not part of this story.

Materializes [FR-152](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a fresh install with no wallet, **When** the user starts
  create or restore, **Then** the onboarding wrapper presents a backup-reminder step
  and a terms-of-service acceptance gate before the wallet becomes usable.
- [x] **AC-2** — **Given** the terms-of-service gate, **When** the user has not
  accepted the terms, **Then** they cannot proceed to the wallet (the continue
  action stays disabled / blocked until acceptance).
- [x] **AC-3** — **Given** the backup-reminder step on the create path, **When** the
  user dismisses or postpones the reminder, **Then** the reminder state is recorded
  so the wallet can re-surface it rather than treating the seed as confirmed-backed-up.
- [x] **AC-4** — **Given** an existing wallet (onboarding already completed), **When**
  the user reopens the extension, **Then** the onboarding wrapper is NOT shown again
  and they land directly on the wallet.

## Dev notes

### Architecture constraints

- [AD-04](../../ARCHITECTURE.md#architecture-decisions) — the keyring is confined to the background; this wrapper must NOT touch private-key material or derivation. It only orchestrates UX gates and records acceptance/reminder flags in settings state.
- This story does NOT introduce new AD entries; onboarding here is a UX layer over EPIC-3's create/restore flow.

### Cross-story dependencies

- Builds on [EPIC-3](../epics/EPIC-3.md) — wraps its create/restore keyring flow; consumes its "wallet exists" signal for first-run detection.
- Sibling [US-19.2](US-19.2-multi-language-i18n-round-1.md) — onboarding strings are part of the bundled-locale set; coordinate so onboarding copy is localized.

### What we explicitly did NOT do

- No key generation / restore / derivation — owned by [EPIC-3](../epics/EPIC-3.md).
- No fiat on-ramp / buy-crypto step — owned by [EPIC-14](../epics/EPIC-14.md); split out of the old onboarding area.

### Points justification

3 pts — §3a-bis "1 internal integration": this is a UX wrapper that gates the
existing EPIC-3 create/restore flow (backup reminder + T&C + first-run detection)
with no external system. It is more than a single template (multi-step flow + state
persistence) but stays internal-integration scope, hence 3 not 5.
