## Goal

Indexer-backed reads stop re-fetching what they just fetched and stop hammering a
provider that is already rate-limiting us: a keyed response cache, cancellable request
groups, a per-window call cap, and an adaptive backoff that halves the call rate when the
provider pushes back.

## Background

Carved out of [US-20.2](US-20.2-api-call-optimization.md) on 2026-07-14. US-20.2's own two
issues (**#4021**, **#4447**) have **zero commits** — but the request-economy work below
*did* ship, under an issue no story cited. A story cannot be `in-progress` for work nobody
is doing while silently containing work already in the product. See
[CONTEXT D100](../../CONTEXT.md).

This is **[#4448](https://github.com/Koniverse/SubWallet-Extension/issues/4448)** (PR
#4470, branch `koni/dev/issue-4448`), shipped **1.3.47** — `strategy/api-request-strategy-v2`:

- an **md5-keyed 60 s response cache** on `ApiHandler`;
- **group request cancellation**;
- a cache for `getMultiChainBalance`;
- reward-history fetches **skipped for addresses with no stake**;
- a **per-window cap** (`callRate = 2` per `intervalCheck = 1000 ms`) with queueing, and an
  **adaptive backoff** — `reduceLimitRate()` halves `callRate` for 30 s on a rate-limit
  error, `maxRetry = 9`.

## Acceptance criteria

- [x] **AC-1** — **Given** an identical request within 60 s, **When** it is issued through
  `ApiHandler`, **Then** it is served from the md5-keyed response cache instead of hitting
  the provider.
- [x] **AC-2** — **Given** a burst of requests through a strategy consumer, **When** they
  are issued, **Then** no more than `callRate` (2) go out per `intervalCheck` (1000 ms) —
  the rest are queued, not fired.
- [x] **AC-3** — **Given** the provider returns a rate-limit error, **When** the strategy
  retries, **Then** retries are bounded (`maxRetry = 9`) and `callRate` is **halved for
  30 s** (`reduceLimitRate()`) — an adaptive backoff, not a retry storm.
- [x] **AC-4** — **Given** an in-flight request group is no longer needed, **When** the
  group is cancelled, **Then** its pending requests are dropped rather than completed.
- [x] **AC-5** — **Given** an address with no stake, **When** the earning surface loads,
  **Then** its reward-history fetch is **skipped entirely**.

## Dev notes

### Architecture constraints

- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — the cache / CDN proxy layer fronting upstream providers. This story is the **client-side** half: the strategy that decides whether a request is issued at all.
- This story does NOT introduce new AD entries.

### What shipped is narrower than the name suggests

Two limits worth stating so nobody reads more into this than shipped:

- **It is a post-hoc response cache, not single-flight.** `cacheMap` is written *after*
  `run()` resolves, so **two identical requests in the same tick both miss and both fire**.
  In-flight de-duplication is forward work — [US-20.2](US-20.2-api-call-optimization.md) AC-1.
- **The cap is per-strategy-consumer, not app-wide.** Its scope is the six consumers
  (Subscan, Runes, Hiro, Bitcoin mainnet/testnet). **Notification, price, SDK and RPC
  traffic bypass it entirely** — which is exactly how the `#4021` notification flood
  survives ([US-20.2](US-20.2-api-call-optimization.md) AC-3).

### Cross-story dependencies

- Sibling [US-20.9](US-20.9-aggregated-data-via-services-sdk.md) — the *other* half of the shipped request economy: aggregate reads routed off-device.
- Forward work on this path: [US-20.2](US-20.2-api-call-optimization.md) (`backlog`) — in-flight dedup, an app-wide cap, and the unbounded notification loop.
- [US-8.6](US-8.6-subscan-api-key-configuration.md) — the user-configurable Subscan API key (#4965, 1.3.75) raises the rate ceiling this backoff halves against.

### Points justification

5 pts — a request-strategy layer (cache keying, cancellation, rate windowing, adaptive
backoff) across the indexer consumers. No user-facing surface, no new external system.
