## Goal

Make EVM dApp connection robust when the page is crowded: SubWallet must coexist
cleanly with other injected wallets (MetaMask, polkadot-js, other EVM extensions)
without fighting over `window.ethereum`, so the dApp's wallet picker keeps working
and a user can always reach SubWallet. This story ships no new FR; its value is that
EIP-6963 discovery and legacy-global contention stay deterministic when several
wallets inject into the same tab.

## Background

This is the **multi-wallet coexistence slice** of EPIC-10's bug / iteration
(hardening) cluster — it owns no FR. When several extensions inject into the same
page, a naive wallet either overwrites `window.ethereum` or gets overwritten,
breaking the dApp's wallet picker. EIP-6963 ([US-10.1](US-10.1-evm-provider-injection-eip-1193-eip-6963.md))
is the standards answer — each wallet announces itself over an event channel
instead of grabbing a single global — and this story hardens that announce/request
handshake against the real-world edges: injection order, late injection (a provider
that appears after the dApp already enumerated), and providers that still grab the
legacy global. The legacy `window.ethereum` path also needs a deterministic,
documented contention behavior so SubWallet never silently hijacks another wallet's
global, while staying reachable via EIP-6963.

This work is anchored in real coexistence reports:
[#3180 — SubWallet wants to connect if using MetaMask](https://github.com/Koniverse/SubWallet-Extension/issues/3180)
(SubWallet's global injection intercepts a connect intended for another wallet) and
[#2605 — Check bug conflict wallet](https://github.com/Koniverse/SubWallet-Extension/issues/2605)
(general multi-extension contention on the injected-provider surface). Both are
the concrete failure class this story's announce/contention guards exist to
eliminate.

All of this routes through the same injected-provider surface the feature story
built — behind `RequestService`
([AD-21](../../ARCHITECTURE.md#architecture-decisions)) — so this story adds guards
and regression coverage rather than a new connector. It is **retroactive/ongoing**:
coexistence fixes have shipped incrementally, and the story stays open to absorb
subsequent multi-wallet dApp-compatibility issues.

## Acceptance criteria

- [ ] **AC-1** — **Given** other EVM wallets injected on the same page, **When** the
  dApp performs EIP-6963 discovery, **Then** SubWallet announces itself correctly
  regardless of injection order and appears in the picker without clobbering or being
  clobbered by other providers.
- [ ] **AC-2** — **Given** a provider that injects late (after the dApp already
  enumerated providers), **When** SubWallet re-announces on request, **Then** the
  dApp still discovers SubWallet and the picker updates rather than missing it.
- [ ] **AC-3** — **Given** a dApp that reads only the legacy `window.ethereum`
  global, **When** multiple wallets contend for it, **Then** SubWallet's behavior is
  deterministic and documented (it does not silently hijack another wallet's global),
  and the user can still reach SubWallet via EIP-6963.
- [ ] **AC-4** — **Given** each coexistence fix, **When** it lands, **Then** it is
  paired with a regression test or guard so the failure class cannot silently
  reappear.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — injected-provider routing stays behind `RequestService`; coexistence hardening adds announce/contention guards, not a parallel injection path.
- This story does NOT introduce new AD entries — it hardens the existing EVM injection surface.

### Cross-story dependencies

- Builds on [US-10.1](US-10.1-evm-provider-injection-eip-1193-eip-6963.md) — hardens its EIP-6963 multi-provider discovery and `window.ethereum` injection.
- Sibling: WalletConnect session/dashboard hardening lives in [US-10.11](US-10.11-walletconnect-session-and-dashboard-hardening.md); injected-account caching lives in [US-10.12](US-10.12-webapp-injected-account-caching.md). This story is EVM-inject coexistence only.

### What we explicitly did NOT do

- No new connector or ecosystem — those are the FR-backed stories (US-10.1..10.6, US-10.9). New capability gets its own FR-backed story.
- No change to the authorization model (US-10.7) — this hardens transport/coexistence, not consent.

### Points justification

3 pts — §3a-bis "2 days / 1 internal integration": a single integration surface
(EVM injected-provider coexistence) with a bounded set of real-world edges
(injection order, late injection, legacy-global contention) plus paired regression
coverage. Narrower than the previously-bundled 5-pt story because the WalletConnect
and injected-account concerns were split into US-10.11 and US-10.12.
