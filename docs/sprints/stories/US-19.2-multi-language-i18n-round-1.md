## Goal

A non-English user can read the wallet in their own language: the UI ships bundled
translations for Vietnamese, Chinese, Japanese and Russian (Round 1), selectable
from a language picker, so the product is usable beyond English-only audiences.
English remains the canonical source — translations are layered on top, and a
missing key falls back to English rather than showing a raw key.

## Status

> **✅ done — shipped in 1.1.11.** All acceptance criteria are ticked, and the 22 rows below are
> settled (21 shipped, 1 closed not-planned). **The table is history, not a work list** — the four
> i18n gaps still open are [US-19.12](US-19.12-i18n-coverage-and-key-hygiene.md)

## Background

The product is authored English-first ([NFR-13](../../PRD.md#non-functional-requirements)):
every user-facing string is written in English, and translations are **bundled per
locale in the extension package**, loaded from `locales/{lng}/translation.json`.
This story delivers the first localization round (VI/ZH/JA/RU): the language picker,
the i18n loader wiring, the bundled locale files, and the English fallback behavior.

This is the *content + UI* half of i18n. The **online i18n hot-update transport** —
pushing new/updated translations at runtime without an extension release — is a
separate mechanism (FR-4) owned by [EPIC-1](../epics/EPIC-1.md)
([US-1.4](../stories/US-1.4-online-i18n-hot-update.md), riding
[AD-25](../../ARCHITECTURE.md#architecture-decisions)); this story ships the
bundled-in-package locales only.

Materializes [FR-153](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** the extension UI, **When** the user opens the language
  picker and selects Vietnamese, Chinese, Japanese or Russian, **Then** the UI
  re-renders in that language from the bundled `locales/{lng}/translation.json`.
- [x] **AC-2** — All user-facing strings are authored English-first (NFR-13); each
  bundled locale's `translation.json` keys are a subset of the English source — no
  locale introduces a key absent from English.
- [x] **AC-3** — **Given** a selected non-English locale missing a particular key,
  **When** that string renders, **Then** it falls back to the English value (never
  the raw key string and never an empty render).
- [x] **AC-4** — **Given** a language selection, **When** the user reopens the
  extension, **Then** the previously chosen language persists.

## Dev notes

### Architecture constraints

- i18n is governed by [NFR-13](../../PRD.md#non-functional-requirements): English-canonical, translations bundled per locale, background error messages internationalized. This story introduces no new AD entry.
- This story ships **bundled** locales only; it does NOT implement the online hot-update transport (FR-4) — that is [EPIC-1](../epics/EPIC-1.md)'s [US-1.4](../stories/US-1.4-online-i18n-hot-update.md) (AD-25).

### Cross-story dependencies

- Required by [US-19.3](US-19.3-additional-ui-languages.md) — that story extends the bundled-locale set and reuses the i18n fallback harness set up here.
- Sibling [US-19.1](US-19.1-wallet-create-restore-onboarding-wrapper.md) — onboarding copy is part of the localized string set.
- Downstream dependent [US-1.4](../stories/US-1.4-online-i18n-hot-update.md) (EPIC-1) — the online hot-update transport later pushes translation updates over the bundled baseline this story establishes.

### What we explicitly did NOT do

- No online i18n hot-update — the runtime remote-translation channel (FR-4) is [EPIC-1](../epics/EPIC-1.md)'s. This story ships in-package locales only.

### Points justification

5 pts — §3a-bis "production deliverable / content batch ≥3 items": four full
translation bundles (VI/ZH/JA/RU) plus the picker + loader + English-fallback +
key-parity harness. Multi-locale content batch with shipping infra sizes at 5.
