## Goal

A Bittensor user can move native TAO between the Subtensor Substrate chain and
the Subtensor-EVM chain, in both directions, directly in the wallet — so that TAO
can be used in EVM contexts and brought back to the native chain without an
external tool.

## Background

Bittensor exposes both a native Substrate (Subtensor) runtime and a Subtensor-EVM
chain. SubWallet's Bittensor integration runs the native pallet path for staking
and swap ([AD-15](../../ARCHITECTURE.md#architecture-decisions)), but moving TAO
between the native chain and the EVM chain is a **bridge** hop — this story owns
the bidirectional native-token bridge between Subtensor (Substrate) and
Subtensor-EVM. It hangs off the bridge-hop interface from
[US-13.1](US-13.1-xcm-parachain-transfers.md) and the provider-handler fixture
from [US-13.2](US-13.2-snowbridge-eth-asset-hub.md), with the bridge construction
following the delegate posture ([AD-18](../../ARCHITECTURE.md#architecture-decisions)).
The two directions have different source-chain transaction shapes (Substrate
extrinsic vs EVM transaction), so the handler must present a single bridge UX over
two submission paths.

Materializes [FR-132](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** TAO on the native Subtensor chain, **When** the user
  bridges to Subtensor-EVM, **Then** the wallet shows the fee, estimated
  destination amount, and ETA before signing, and on submit the TAO arrives on the
  EVM side.
- [x] **AC-2** — **Given** TAO on Subtensor-EVM, **When** the user bridges back to
  the native Subtensor chain, **Then** the reverse direction works symmetrically
  with fee/ETA shown before signing and the TAO arriving on the native side.
- [x] **AC-3** — **Given** either direction, **When** the bridge is in transit,
  **Then** it is tracked through the shared status machine and in-transit funds
  are never reported as lost.
- [x] **AC-4** — **Given** a direction whose source-chain submission fails (e.g.
  EVM tx underpriced), **When** it errors, **Then** the bridge is not marked
  submitted and the source funds remain on the source chain — no half-bridged
  state.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction delegated; the handler consumes it for both directions.
- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — Bittensor integration model: native Subtensor path for staking/swap; this story is the *bridge* between that native chain and Subtensor-EVM, not the native pallet flows.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) (bridge-hop interface) and [US-13.2](US-13.2-snowbridge-eth-asset-hub.md) (provider-handler fixture).
- Sibling [US-13.8](US-13.8-bittensor-alpha-token-bridges.md) — the planned alpha-token bridge family extends the Bittensor bridge surface established here.

### Points justification

5 pts — §3a-bis "1 external system integration": one native-token bridge between
two Bittensor chains behind the existing interface. The extra wrinkle is two
source-chain submission shapes (Substrate + EVM) under one UX, which keeps it a
full single-integration at 5 rather than a 3.
