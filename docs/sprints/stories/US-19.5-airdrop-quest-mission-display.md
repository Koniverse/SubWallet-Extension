## Goal

Users can discover and act on growth incentives inside the wallet: airdrop and
quest missions are listed with their status and progress, and each gives the user a
clear entry point to participate or claim — so engagement campaigns drive on-chain
activity without the user leaving the app. The missions are remotely configured, so
the team can launch new quests release-free.

## Background

Airdrop and quest missions are remotely configured campaign content, aggregated and
served through the SubWallet Services SDK backend
([AD-24](../../ARCHITECTURE.md#architecture-decisions)) — the same backend
aggregation layer that fans balance / NFT / swap data across chains. This story owns
the *display* of missions (list, status, progress) and the *entry point* into a
mission (participate / claim), reusing the remote-content mock and degradation
pattern established by [US-19.4](US-19.4-in-app-campaign-banners.md).

A mission that targets earning/staking deep-links into the EarningService surface
owned by [EPIC-12](../epics/EPIC-12.md) — this story does not re-implement stake or
claim-reward mechanics; it surfaces the mission and routes the user into the
existing flow.

Materializes [FR-158](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** active airdrop/quest missions from the Services SDK, **When**
  the user opens the missions surface, **Then** each mission is listed with its title,
  status and progress.
- [ ] **AC-2** — **Given** a listed mission, **When** the user taps to participate or
  claim, **Then** they are routed to the configured entry point (deep-link into the
  relevant flow, e.g. an earning position in [EPIC-12](../epics/EPIC-12.md), or an
  external quest page) — this story does not re-implement the target flow.
- [ ] **AC-3** — **Given** the mission feed fetch fails or returns empty, **When** the
  surface renders, **Then** an empty/error state is shown gracefully and the rest of
  the app is unaffected — never a blank or crashed screen.
- [ ] **AC-4** — **Given** a mission the user has already completed, **When** the list
  renders, **Then** it is shown as completed (or filtered per config) rather than
  offered again as actionable.

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — mission data is aggregated through the Services SDK backend, not computed on-device; consume it and render.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-19.4](US-19.4-in-app-campaign-banners.md) — reuses the remote-content mock and graceful-degradation pattern.
- Deep-links into [EPIC-12](../epics/EPIC-12.md) (EarningService) for missions that target staking — does not re-implement the earning flow.
- Sibling [US-19.7](US-19.7-mission-pools-reward-program.md) — Mission Pools is a related reward surface; coordinate the mission/reward config shape.

### What we explicitly did NOT do

- No stake/unstake/claim-reward re-implementation — those flows live in [EPIC-12](../epics/EPIC-12.md); this story only surfaces the mission and routes into them.

### Points justification

5 pts — §3a-bis "1 external system integration": consuming the Services SDK mission
feed (AD-24), rendering status/progress, and wiring entry-point deep-links into the
earning surface. External-backend integration plus a non-trivial display sizes at 5.
