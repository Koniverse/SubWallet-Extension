## Goal

A holder of a governance-enabled chain (DOT/KSM and OpenGov parachains) opens the
governance surface, browses the live OpenGov referenda list, drills into a
referendum's detail, and casts a conviction vote — choosing aye/nay, an amount,
and a conviction multiplier — then revotes or unvotes while the referendum is
still open, all without leaving the wallet. This story owns the OpenGov browse +
vote surface; the conviction lock it creates is read and released by
[US-15.2](US-15.2-locked-token-detail-and-unlock-flow.md).

## Background

OpenGov is Polkadot's on-chain referendum system: anyone holding the native token
has voting power, scaled by a conviction multiplier that trades a longer token
lock for more weight. Before this story a SubWallet user with voting power had to
leave for a third-party governance UI (Polkassembly, Subsquare) to participate.

The referenda list, detail, and per-account vote status are **aggregated through
the backend** Services SDK ([AD-24](../../ARCHITECTURE.md#architecture-decisions))
rather than indexed on-device — per-chain referendum indexing across every
OpenGov chain is heavy and rate-limited (the same rationale that drives balance
aggregation). The conviction vote / revote / unvote extrinsics
(`convictionVoting.vote` / `removeVote`) are constructed and signed in the
background and surfaced to the UI only over the typed `pri()` bus
([AD-03](../../ARCHITECTURE.md#architecture-decisions)); the governance UI never
holds key material. The vote-lock balance figure the vote creates is the same
`locked` reservation owned by [EPIC-7](../epics/EPIC-7.md)'s balance model — this
story does not re-derive it.

Materializes [FR-139](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped; `commit` / `version_shipped` are backfilled during version
reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** an account on an OpenGov-enabled chain, **When** the
  user opens the governance surface, **Then** the live referenda list is shown
  (status, track, tally) sourced from the Services-SDK aggregation, and selecting
  one shows its detail.
- [x] **AC-2** — **Given** an open referendum and a sufficient free balance,
  **When** the user casts a conviction vote (aye/nay + amount + conviction
  multiplier), **Then** the vote extrinsic is built and signed in the background,
  the vote is recorded against the referendum, and the chosen amount becomes
  conviction-locked.
- [x] **AC-3** — **Given** an account that has already voted on an open
  referendum, **When** the user revotes (changes side/amount/conviction) or
  unvotes, **Then** the prior vote is replaced or removed via the corresponding
  extrinsic and the vote status updates.
- [x] **AC-4** — **Given** the user attempts to lock more than their available
  free balance, **When** they confirm the vote, **Then** the flow blocks with an
  insufficient-balance error and no extrinsic is submitted.
- [x] **AC-5** — **Given** a referendum that closes (decision period ends) while
  the user is composing or submitting a vote, **When** the submission resolves,
  **Then** the closed state is surfaced and the vote action is disabled rather
  than submitting against a closed referendum.

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — referenda list/detail/track and vote status are aggregated through the Services SDK backend; this surface consumes it rather than indexing per chain on-device.
- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — the conviction-lock/free-balance read rides the lightweight WsProvider path; the full `@polkadot/api` ApiPromise is instantiated only to construct the vote extrinsic.
- [AD-03](../../ARCHITECTURE.md#architecture-decisions) — vote/revote/unvote extrinsics are signed in the background and surfaced to the UI via the typed `pri()` bus; no key material in the governance UI.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Required by [US-15.2](US-15.2-locked-token-detail-and-unlock-flow.md) — that story reads and releases the conviction lock this vote creates.
- Required by [US-15.4](US-15.4-opengov-delegation-and-governance-tracks.md) — Phase-2 delegation/tracks extend this referenda surface and reuse the referenda subject fixture set up here.
- Sibling [US-15.3](US-15.3-governance-v1-democracy-display-only.md) — both render referendum data from the Services SDK; coordinate the shared referenda fixture.
- Consumes the transferable/free figure from [EPIC-7](../epics/EPIC-7.md)'s balance model for the insufficient-balance guard.

### What we explicitly did NOT do

- No on-device per-chain referendum indexer — referenda come through the Services SDK (AD-24). Trigger to revisit: backend coverage gap for a needed OpenGov chain.
- No home-screen rendering of the resulting lock — that figure is owned by [EPIC-7](../epics/EPIC-7.md); this story only creates the lock.
