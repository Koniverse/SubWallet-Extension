## Goal

A Bittensor user can bridge alpha (subnet) tokens out of the native Subtensor
chain ‚Äî to Subtensor-EVM, to Base as xTAO/xAlpha, and via the Liquid Stake Alpha
Bridge ‚Äî directly in the wallet, so that subnet-token value can move into the EVM
and L2 worlds without an external tool.

## Background

This is a **planned** (forward) extension of the Bittensor bridge surface
established by [US-13.7](US-13.7-bittensor-tao-subtensor-evm-bridge.md). Where
US-13.7 bridges the native TAO token, US-13.8 covers the **alpha-token** family:
alpha ‚Üî Subtensor-EVM, xTAO / xAlpha wrapped to Base, and the Liquid Stake Alpha
Bridge. It is a family of related bridges rather than a single provider, each with
its own destination and token-wrapping semantics, all built behind the bridge-hop
interface and the delegate posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the Bittensor integration
model ([AD-15](../../ARCHITECTURE.md#architecture-decisions)) governs the native
alpha-token side.

Materializes [FR-133](../../PRD.md#functional-requirements), which is `üìã planned` in the PRD ‚Äî this story
is **forward / planned**, not retroactive. AC and sizing describe the intended
capability; `commit` / `version_shipped` are set when the work lands. Tracked by
[#4919](https://github.com/Koniverse/SubWallet-Extension/issues/4919) ‚Äî Support
bridge apha token (Bittensor) <-> alpha token (Subtensor EVM), and
[#4903](https://github.com/Koniverse/SubWallet-Extension/issues/4903) ‚Äî Support
bridge xTAO/xAlpha -> Base.

## Acceptance criteria

- [ ] **AC-1** ‚Äî **Given** an alpha (subnet) token on the native Subtensor chain,
  **When** the user bridges it to Subtensor-EVM, **Then** the wallet shows the
  fee, estimated destination amount, and ETA before signing, and the alpha token
  arrives on the EVM side.
- [ ] **AC-2** ‚Äî **Given** TAO/alpha eligible for wrapping, **When** the user
  bridges to Base as xTAO / xAlpha, **Then** the wrapped token arrives on Base with
  the correct destination representation and the fee/ETA shown before signing.
- [ ] **AC-3** ‚Äî **Given** the Liquid Stake Alpha Bridge path, **When** the user
  uses it, **Then** the bridge completes with fee/ETA shown and the resulting
  position/token is reflected in the wallet.
- [ ] **AC-4** ‚Äî **Given** any alpha-bridge direction in transit (or that stalls),
  **When** the user returns, **Then** it is tracked safely and in-transit funds
  are never reported as lost.

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) ‚Äî Bittensor integration model governs the native alpha-token side (dTAO / subnet model).
- This story does NOT introduce new AD entries.
