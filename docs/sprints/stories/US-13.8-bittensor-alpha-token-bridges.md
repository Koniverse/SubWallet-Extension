## Goal

A Bittensor user can bridge alpha (subnet) tokens out of the native Subtensor
chain — to Subtensor-EVM, to Base as xTAO/xAlpha, and via the Liquid Stake Alpha
Bridge — directly in the wallet, so that subnet-token value can move into the EVM
and L2 worlds without an external tool.

## Background

This is a **planned** (forward) extension of the Bittensor bridge surface
established by [US-13.7](US-13.7-bittensor-tao-subtensor-evm-bridge.md). Where
US-13.7 bridges the native TAO token, US-13.8 covers the **alpha-token** family:
alpha ↔ Subtensor-EVM, xTAO / xAlpha wrapped to Base, and the Liquid Stake Alpha
Bridge. It is a family of related bridges rather than a single provider, each with
its own destination and token-wrapping semantics, all built behind the bridge-hop
interface and the delegate posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the Bittensor integration
model ([AD-15](../../ARCHITECTURE.md#architecture-decisions)) governs the native
alpha-token side.

Materializes [FR-135](../../PRD.md#functional-requirements), which is `📋 planned` in the PRD — this story
is **forward / planned**, not retroactive. AC and sizing describe the intended
capability; `commit` / `version_shipped` are set when the work lands. Tracked by
[#4919](https://github.com/Koniverse/SubWallet-Extension/issues/4919) — Support
bridge apha token (Bittensor) <-> alpha token (Subtensor EVM), and
[#4903](https://github.com/Koniverse/SubWallet-Extension/issues/4903) — Support
bridge xTAO/xAlpha -> Base.

## Acceptance criteria

- [ ] **AC-1** — **Given** an alpha (subnet) token on the native Subtensor chain,
  **When** the user bridges it to Subtensor-EVM, **Then** the wallet shows the
  fee, estimated destination amount, and ETA before signing, and the alpha token
  arrives on the EVM side.
- [ ] **AC-2** — **Given** TAO/alpha eligible for wrapping, **When** the user
  bridges to Base as xTAO / xAlpha, **Then** the wrapped token arrives on Base with
  the correct destination representation and the fee/ETA shown before signing.
- [ ] **AC-3** — **Given** the Liquid Stake Alpha Bridge path, **When** the user
  uses it, **Then** the bridge completes with fee/ETA shown and the resulting
  position/token is reflected in the wallet.
- [ ] **AC-4** — **Given** any alpha-bridge direction in transit (or that stalls),
  **When** the user returns, **Then** it is tracked safely and in-transit funds
  are never reported as lost.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — each alpha-token bridge is built via the delegated construction, not hand-rolled.
- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — Bittensor integration model governs the native alpha-token side (dTAO / subnet model).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.7](US-13.7-bittensor-tao-subtensor-evm-bridge.md) — extends the Bittensor bridge surface (native TAO) to the alpha-token family.

### What we explicitly did NOT do

- Forward story — not yet shipped. The three bridge paths (alpha↔EVM, xTAO/xAlpha→Base, Liquid Stake Alpha) are scoped together because they share the Bittensor bridge surface; if any path's external dependency is not ready, ship the available paths and defer the rest rather than blocking the whole story.

### Points justification

8 pts — §3a-bis "multi-asset / content batch ≥3 items" maps here to a **bridge
batch**: three distinct alpha-token bridge paths (alpha↔Subtensor-EVM,
xTAO/xAlpha→Base wrapping, Liquid Stake Alpha Bridge), each with its own
destination and token-wrapping semantics. That is more than one external
integration (5) but below the 13 split threshold — sized at 8 as a multi-bridge
family. If the three paths are delivered independently across releases, consider
splitting per-path at planning time.
