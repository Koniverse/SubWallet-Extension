## Goal

A security-conscious user can sign with a Ledger device over USB — across the
Substrate generic app, per-chain Substrate apps, and the EVM app — so that their
private key never enters the extension.

## Status

> **✅ done — shipped in 0.5.4.** All four acceptance criteria are ticked and the 12 rows below are
> settled (10 delivered, 2 closed not-planned). The chain-by-chain, connect-and-discover,
> per-feature, Generic-app and recovery-tool work that grew out of this capability lives in
> [US-16.4](US-16.4-ledger-network-and-app-coverage.md) …
> [US-16.8](US-16.8-ledger-asset-recovery-tool.md); what is left here is the **signature itself**.

## Background

Ledger is the most involved hardware path: one device, three app surfaces
(Substrate generic, per-chain Substrate, EVM) over USB/WebHID. It is the offline
branch of the keyring boundary ([AD-04](../../ARCHITECTURE.md#architecture-decisions),
[NFR-14](../../PRD.md#non-functional-requirements)) — the wallet builds and validates the payload, the device
signs, the wallet submits. The Generic Ledger app in particular needs
**metadata-hash signing** ([FR-82](../../PRD.md#functional-requirements)) so the device can render a
verified extrinsic. Kept separate from the QR devices
([US-16.2](US-16.2-qr-signing-keystone-and-polkadot-vault.md)) because the USB
transport and multi-app handling are distinct work.

Materializes [FR-144](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** a connected Ledger, **When** the user signs a Substrate
  transaction, **Then** it is signed on-device (generic or per-chain app) and
  submitted; no private key enters the extension (AD-04, NFR-14).
- [x] **AC-2** — **Given** the Ledger EVM app, **When** the user signs an EVM
  transaction, **Then** it is signed and submitted via the EVM app.
- [x] **AC-3** — **Given** the Generic Ledger app requires metadata hash, **When**
  signing, **Then** metadata-hash signing (FR-82) is used so the device shows a
  verified extrinsic.
- [x] **AC-4** — **Given** the user rejects on the device, **When** the rejection
  returns, **Then** the wallet surfaces it cleanly and submits nothing.
