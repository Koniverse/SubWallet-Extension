## Goal

Keep the earning surface *fast* and its data *fresh*. When a user holds
positions across many pools, opening the earning tab must paint quickly, stay
memory-bounded, and never show data for an account that has already been
removed. This story is one half of the earning epic's hardening cluster: it owns
**speed and cache-correctness** of the earning surface (the sibling
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) owns
*correctness* of the reward/APY numbers). It ships no new feature — it defends
the load behaviour and cache freshness of everything the other earning stories
render.

## Background

The earning epic's bug backlog splits into two genuinely different logics, and
this story owns the second: **performance & stale-cache** — a speed/freshness
problem, distinct from whether a reward/APY *figure* is computed correctly (that
correctness logic is the sibling
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md)).

The performance/cache backlog gathers issues that share one failure mode — the
earning surface is slow to load and/or serves stale data — across the read path:

- [#2615](https://github.com/Koniverse/SubWallet-Extension/issues/2615) — review
  earning **performance and cache data**: load is not good *and* the surface
  still shows data for a **removed account** (stale cache not invalidated on
  account removal). (This issue was previously mislabeled as "APY-formula
  accuracy" under the old combined US-12.13; it is a performance + stale-cache
  issue and is correctly homed here.)
- [#2749](https://github.com/Koniverse/SubWallet-Extension/issues/2749) —
  [WebApp] **slow loading** when accessing the Earning option screen.
- [#3328](https://github.com/Koniverse/SubWallet-Extension/issues/3328) —
  [WebApp] **improve Portfolio for Earning, Token** (read-path/list performance
  for the earning + token surfaces).

The fix lives on the read path the engine and platform publish: pool info and
targets are read from the static-data cache
([AD-23](../../ARCHITECTURE.md#architecture-decisions)) rather than a live
per-render RPC sweep, aggregation rides the Services SDK
([AD-24](../../ARCHITECTURE.md#architecture-decisions)) on the lightweight read
path ([AD-07](../../ARCHITECTURE.md#architecture-decisions) — WsProvider-only,
≤ 72 MB regardless of chain count), and cached state is invalidated when its
owning account is removed so a deleted account's earning data never lingers.
This story owns **no FR** — it is the performance/cache half of the earning
hardening cluster (cf.
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md),
[US-11.8](US-11.8-cross-chain-swap-routing.md) pattern of an FR-less defending
story). It defends NFR-11 (memory-bounded read path), NFR-12 (cached cold-start
+ progressive refresh), NFR-20 (Services SDK aggregation) and NFR-21
(static/cache data layer) on the earning surface specifically.

## Acceptance criteria

- [ ] **AC-1** — **Given** a user holding positions across many pools, **When**
  the earning tab opens, **Then** cached positions paint within the cold-start
  budget (≤ 300 ms, NFR-12) and the list stays responsive while fresh data
  streams in ([#2749](https://github.com/Koniverse/SubWallet-Extension/issues/2749)).
- [ ] **AC-2** — **Given** the earning list reads pool info and targets, **When**
  it loads, **Then** it reads from the static-data cache (AD-23), not a live
  per-render RPC sweep, and the read path stays within the memory budget
  (≤ 72 MB, NFR-11 / AD-07)
  ([#3328](https://github.com/Koniverse/SubWallet-Extension/issues/3328)).
- [ ] **AC-3** — **Given** an account that is **removed** while its earning data
  is cached, **When** the earning surface next renders, **Then** that account's
  cached earning/position data is invalidated and no longer shown
  ([#2615](https://github.com/Koniverse/SubWallet-Extension/issues/2615)).
- [ ] **AC-4** — **Given** the WebApp Earning option screen, **When** it is
  accessed, **Then** it loads within the cold-start budget rather than blocking
  on a synchronous per-pool fetch sweep
  ([#2749](https://github.com/Koniverse/SubWallet-Extension/issues/2749),
  [#3328](https://github.com/Koniverse/SubWallet-Extension/issues/3328)).
- [ ] **AC-5** *(unhappy path)* — **Given** one pool's data source is stale or
  unreachable, **When** the earning list refreshes, **Then** the other positions
  still resolve and the degraded pool is shown stale/unavailable rather than
  stalling or blanking the whole list.

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — pool info / targets are read from the static-data cache, not polled live per render.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — earning aggregation rides the Services SDK on the lightweight read path.
- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — earning read uses the lightweight WsProvider connector (≤ 72 MB regardless of chain count), never a full ApiPromise per earning chain just to read positions.
- This story does NOT introduce new AD entries.
- This story does NOT touch reward/APY **formula** correctness (AD-22) — that is the sibling [US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md)'s concern.

### Cross-story dependencies

- **Builds on** [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — verifies load/cache behaviour at the `EarningService` handler seam every earning story consumes.
- **Builds on** [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) — the native-staking story that sets up the static earning-data read path / pool-targets fixture this story stress-tests.
- **Builds on** [US-12.8](US-12.8-earning-path-simulation-and-step-tracking.md) — shares the static earning-data mock (fresh / stale / missing) used for the cache + degraded-source tests.
- **Hardens** [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md)–[US-12.7](US-12.7-lending-interlay.md) — the earning list and position read path those stories render.
- **Sibling** [US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) — the *accuracy* half of the earning hardening cluster; coordinate so a fast cache read does not mask a wrong figure.

### Performance budget

- Earning-tab first paint: cached positions visible ≤ 300 ms on popup open (NFR-12).
- Earning read memory: ≤ 72 MB regardless of pool/chain count (NFR-11, AD-07 lightweight read path).
- Pool data fetch: from static cache, no live per-render RPC sweep (NFR-21, AD-23).
- Removed-account earning data: invalidated on account removal, never served stale.
- Story PR description must explicitly confirm these budgets are met.

### What we explicitly did NOT do

- No change to reward/APY **figures** — figure correctness is the sibling [US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md)'s scope; this story does not re-verify the math, only the speed and freshness of the read path.
- No new earning protocol or pool type — hardening only; new mechanisms are their own stories.
- No new cache layer — reuses the existing static-data cache (AD-23) and Services SDK (AD-24); this story tightens *how the earning surface uses them*, it does not build a new one.

### Points justification

5 pts — §3a-bis "production deliverable / 1 integration surface, 3-4 days". This
is a single-logic hardening cluster: earning-surface **speed + cache freshness**
on the read path (static cache + Services-SDK aggregation + memory budget +
removed-account invalidation), absorbing a focused three-issue cluster (#2615,
#2749, #3328). It does **not** span reward/APY formula accuracy (that is the
other half, [US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md)),
so it sizes below the original combined 8-pt story: the split removes the
per-pool-type accuracy-verification surface, leaving the read-path/cache work,
which calibrates at 5 alongside the US-7.7-class single-concern defending story.
