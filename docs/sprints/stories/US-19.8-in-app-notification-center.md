## Goal

Users have one place inside the wallet to see what's happening: a notification
center that surfaces transaction status, campaign alerts and system messages in a
single feed, with read/unread state and an unread indicator — so a user knows when a
transfer confirmed, a campaign launched, or the team posted a system message without
hunting across screens.

## Background

The notification center is the in-app *feed* that aggregates events from several
sources: transaction-status events the wallet already produces, campaign alerts from
the remotely configured campaign substrate (shared with
[US-19.4](US-19.4-in-app-campaign-banners.md)), and system messages pushed through
the cache-proxy layer ([AD-25](../../ARCHITECTURE.md#architecture-decisions)). This
story owns the *feed* — collecting these event kinds into one list, tracking
read/unread, showing an unread badge, and handling the OS notification-permission
path — not the production of the underlying events (transaction status is produced
by the transaction/history services; campaign content by the campaign stories).

Materializes [FR-161](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** transaction-status events, campaign alerts and system
  messages, **When** the user opens the notification center, **Then** all three kinds
  appear in one chronological feed, each tagged by kind.
- [ ] **AC-2** — **Given** unread notifications, **When** the user views the entry
  point, **Then** an unread indicator/count is shown, and opening an item marks it
  read so the count decrements.
- [ ] **AC-3** — **Given** the user denies (or has not granted) the OS notification
  permission, **When** notifications arrive, **Then** the in-app feed still records
  and shows them — permission denial degrades to in-app-only, it never drops or
  breaks the feed.
- [ ] **AC-4** — **Given** a campaign-alert / system-message fetch fails, **When** the
  center renders, **Then** the still-available kinds (e.g. transaction status) are
  shown and the failed source degrades to empty rather than failing the whole feed.

## Tasks

- [ ] **TASK-19.8.1** — Aggregate the three notification kinds into one feed (AC: 1)
  - [ ] Collect transaction-status events, campaign alerts (shared with [US-19.4](US-19.4-in-app-campaign-banners.md)), and system messages (AD-25) into a single chronological list tagged by kind.
- [ ] **TASK-19.8.2** — Read/unread state + unread indicator (AC: 2)
  - [ ] Persist per-item read state; show an unread badge/count; mark-read on open.
- [ ] **TASK-19.8.3** — OS notification-permission handling (AC: 3)
  - [ ] On permission denied / not-granted, fall back to in-app-only; never drop the feed.
- [ ] **TASK-19.8.4** — Per-source degradation (AC: 4)
  - [ ] A failed campaign/system-message fetch degrades that source to empty; other kinds still render.

## Dev notes

### Architecture constraints

- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — campaign-alert / system-message content is delivered through the cache-proxy layer with a fallback; the feed consumes it, never a keyed upstream in the bundle.
- This story owns the *feed*, not the event producers: transaction-status events come from the existing transaction/history services; campaign content from the campaign stories. No new AD entry.

### Cross-story dependencies

- Builds on [US-19.4](US-19.4-in-app-campaign-banners.md) — reuses the campaign-config shape for campaign alerts.
- Consumes transaction-status events produced by the existing transaction/history flows — this story renders them in the feed, it does not produce them.

### Points justification

5 pts — §3a-bis "multi-system integration": the feed aggregates three distinct event
sources (transaction status, campaign alerts via the campaign substrate, system
messages via AD-25), plus read/unread state, an unread indicator, OS-permission
fallback, and per-source degradation. Aggregating multiple sources into one durable
feed sizes at 5.
