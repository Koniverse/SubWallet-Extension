## Goal

Users can participate in the Mission Pools reward program from inside the wallet:
the program surfaces the available pools, the user's eligibility and the rewards on
offer, with an entry point to join — so a structured, ongoing rewards program drives
retention without leaving the app. The program is remotely configured, so pools and
rewards can be updated release-free.

## Background

Mission Pools is a structured reward program — a richer, ongoing surface than a
single quest. Its pool definitions, eligibility rules and reward data are aggregated
and served through the SubWallet Services SDK backend
([AD-24](../../ARCHITECTURE.md#architecture-decisions)). This story owns the
*display* of the program (pools, eligibility, rewards) and the *entry point* to
join a pool, reusing the remote-content mock and degradation pattern from the
campaign stories.

A pool that targets earning/staking deep-links into the EarningService surface owned
by [EPIC-12](../epics/EPIC-12.md); this story does not re-implement stake/reward
mechanics — it surfaces the program and routes the user into the existing flow.

Materializes [FR-158](../../PRD.md#functional-requirements). This story is **retroactive** — the capability
already ships in the product; `commit` / `version_shipped` are backfilled during
version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** the Mission Pools program config from the Services SDK,
  **When** the user opens the program surface, **Then** the available pools are listed
  with their rewards and the user's eligibility for each.
- [x] **AC-2** — **Given** an eligible pool, **When** the user taps to join, **Then**
  they are routed to the configured entry point (deep-link into the relevant flow,
  e.g. an earning position in [EPIC-12](../epics/EPIC-12.md)) — this story does not
  re-implement the target flow.
- [x] **AC-3** — **Given** a pool the user is not eligible for, **When** the list
  renders, **Then** the ineligibility is shown clearly (with the reason/criteria
  where configured) and the join action is not offered.
- [x] **AC-4** — **Given** the program config fetch fails or returns empty, **When**
  the surface renders, **Then** a graceful empty/error state is shown and the rest of
  the app is unaffected — never a blank or crashed screen.

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — program/pool/reward data is aggregated through the Services SDK backend, not computed on-device; consume it and render.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-19.5](US-19.5-airdrop-quest-mission-display.md) — reuses the Services-SDK fetch pattern and the remote-content mock / degradation pattern.
- Deep-links into [EPIC-12](../epics/EPIC-12.md) (EarningService) for pools that target staking — does not re-implement the earning flow.

### What we explicitly did NOT do
