## Goal

A Polkadot-ecosystem dApp (Polkadot-JS apps, Subsquare, Hydration, any
`@polkadot/extension-dapp` consumer) discovers and enables SubWallet through the
standard `window.injectedWeb3` interface, lists the user's authorized Substrate
accounts, and requests signatures тАФ exactly as it would for the polkadot-js
extension. Substrate dApps work with SubWallet out of the box.

## Status

> **тЬЕ done тАФ shipped in 0.2.1.** All acceptance criteria are ticked and the 3 rows below are
> settled.

## Background

The Polkadot dApp ecosystem standardized on the polkadot-js injection interface:
an extension populates `window.injectedWeb3[<name>]` with `enable()`, and the
enabled object exposes `accounts.get()` / `accounts.subscribe()` and a `signer`
implementing `signPayload` / `signRaw`. SubWallet ships this via
`@subwallet/extension-inject` (the generic injector that populates
`window.injectedWeb3`), so any dApp using `@polkadot/extension-dapp`'s
`web3Enable` / `web3Accounts` / `web3FromSource` finds it without code changes.

`enable(origin)` and every signature request are gated by the per-origin
authorization store (US-10.7) and routed to the Substrate request handler in
`RequestService` ([AD-21](../../ARCHITECTURE.md#architecture-decisions)) behind the
shared `PopupHandler`. Substrate has two signing shapes тАФ `signPayload` (extrinsic
payload) and `signRaw` (arbitrary bytes) тАФ both surfaced through the approval
queue; the actual signing runs in the background keyring (AD-04).

Materializes [FR-95](../../PRD.md#functional-requirements). **Retroactive** тАФ already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** тАФ **Given** any page, **When** it loads, **Then**
  `window.injectedWeb3` contains the SubWallet entry exposing an `enable()` that
  returns an injected object with `accounts` and `signer`.
- [x] **AC-2** тАФ **Given** a dApp calling `web3Enable(...)` from an unauthorized
  origin, **When** it runs, **Then** the enable request is surfaced in the approval
  popup; only on approval does `accounts.get()` return the user-selected accounts.
- [x] **AC-3** тАФ **Given** an enabled dApp, **When** it calls
  `signer.signPayload(payload)` or `signer.signRaw(data)`, **Then** the request is
  routed to the Substrate request handler and surfaced for approval; the signature
  is produced in the background keyring.
- [x] **AC-4** тАФ **Given** an enabled dApp subscribed via `accounts.subscribe`,
  **When** the authorized account set changes, **Then** the dApp receives the
  updated account list.
- [x] **AC-5** тАФ **Given** an unauthorized origin or a user rejection, **When** the
  request resolves, **Then** `enable()` / sign rejects with an error and no account
  list or signature leaks.

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) тАФ `signPayload` / `signRaw` route to the Substrate handler behind the shared `PopupHandler`.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) тАФ signing runs in the background keyring; the injected `signer` carries no key bytes.
- This story does NOT introduce new AD entries.

- Sibling [US-10.8](US-10.8-arbitrary-message-signing.md) тАФ `signRaw` is the Substrate arm of arbitrary message signing.

Sized 5: one external standard surface (the polkadot-js injection interface) тАФ
`injectedWeb3` discovery + `enable` gating + the dual `signPayload`/`signRaw`
signing shapes + account subscription, for one ecosystem.
