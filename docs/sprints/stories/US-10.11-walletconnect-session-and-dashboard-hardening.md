## Goal

Keep the WalletConnect dashboard correct under stale, duplicate, and expired
sessions: WC v2 sessions accumulate, expire on the relay, get duplicated across
re-pairs, or linger after a dApp disconnects, and the session list must reconcile to
the true relay state so a user never sees or acts on a dead session. This story ships
no new FR; its value is that the WalletConnect surface stays trustworthy when
sessions get messy.

## Background

This is the **WalletConnect session slice** of EPIC-10's bug / iteration (hardening)
cluster — it owns no FR. WC v2 sessions
([US-10.3](US-10.3-walletconnect-v2-substrate-and-evm.md)) accumulate over a wallet's
lifetime: they expire on the relay, get duplicated across re-pairs, or linger after a
dApp disconnects. Without reconciliation the dashboard shows sessions that no longer
exist, and a request arriving on a dead session can be silently dropped or
mis-routed. This story reconciles the session list to the live relay state —
expire/dedupe/prune — and rejects requests that arrive on a session the relay has
already terminated, so the failure is explicit rather than silent.

A closely related failure class is **opaque/misleading errors** during connect and
request handling, which this story also hardens. Real reports anchor it:
[#4005 — Error when connecting walletconnect in case the network is not supported](https://github.com/Koniverse/SubWallet-Extension/issues/4005)
(unsupported-chain connect must fail with a clear, actionable message rather than a
confusing one),
[#4995 — EVM transaction from dApp fails with misleading error when chain RPC is unreachable](https://github.com/Koniverse/SubWallet-Extension/issues/4995)
(an unreachable-RPC failure surfaced as a misleading error rather than the true
cause), and
[#3476 — Update message for some error cases when connecting to dApp](https://github.com/Koniverse/SubWallet-Extension/issues/3476)
(clearer messaging across dApp-connect error cases). Together with session
reconciliation, accurate error surfacing is what keeps the WalletConnect surface
trustworthy when sessions and networks get messy.

All of this routes through the same WalletConnect handlers the feature story built —
behind `RequestService`
([AD-21](../../ARCHITECTURE.md#architecture-decisions)) — so this story adds
reconciliation and regression coverage rather than a new connector. It is
**retroactive/ongoing**: session fixes have shipped incrementally, and the story
stays open to absorb subsequent WalletConnect session-state issues.

## Acceptance criteria

- [ ] **AC-1** — **Given** the WalletConnect dashboard, **When** it is opened,
  **Then** the listed sessions reconcile to the live relay state — expired and
  relay-terminated sessions are removed, and no session is shown twice.
- [ ] **AC-2** — **Given** a dApp re-pairs or a session expires, **When** the change
  occurs, **Then** stale/duplicate entries are pruned so the list reflects only live
  sessions.
- [ ] **AC-3** — **Given** a request arriving on a dead/expired session, **When** it
  reaches the WalletConnect handler, **Then** it is rejected rather than silently
  dropped or mis-routed.
- [ ] **AC-4** — **Given** a connect/request error case — an unsupported network on
  connect ([#4005](https://github.com/Koniverse/SubWallet-Extension/issues/4005)),
  an unreachable chain RPC on an EVM dApp transaction
  ([#4995](https://github.com/Koniverse/SubWallet-Extension/issues/4995)), or other
  dApp-connect error cases ([#3476](https://github.com/Koniverse/SubWallet-Extension/issues/3476)),
  **When** it occurs, **Then** the user sees a clear, accurate message that names
  the true cause rather than a misleading or generic error.
- [ ] **AC-5** — **Given** each session/reconciliation or error-messaging fix,
  **When** it lands, **Then** it is paired with a regression test or guard so the
  failure class cannot silently reappear.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — session/request routing stays behind `RequestService` and the WalletConnect handlers; hardening adds reconciliation, not a parallel path.
- This story does NOT introduce new AD entries — it hardens the existing WalletConnect surface.

### Cross-story dependencies

- Builds on [US-10.3](US-10.3-walletconnect-v2-substrate-and-evm.md) — reconciles the WalletConnect session dashboard it created.
- Sibling: EVM-inject coexistence lives in [US-10.10](US-10.10-multi-wallet-coexistence-hardening.md); injected-account caching lives in [US-10.12](US-10.12-webapp-injected-account-caching.md). This story is WalletConnect session state only.

### What we explicitly did NOT do

- No new connector or ecosystem — those are the FR-backed stories (US-10.1..10.6, US-10.9). New capability gets its own FR-backed story.
- No change to the authorization model (US-10.7) — this hardens session lifecycle, not consent.

### Points justification

3 pts — §3a-bis "2 days / 1 internal integration": a single integration surface
(the WalletConnect v2 relay/session lifecycle) with a bounded set of reconciliation
cases (expire, dedupe, prune, dead-session rejection) plus paired regression
coverage. Split out of the previously-bundled 5-pt story so each concern is one
story.
