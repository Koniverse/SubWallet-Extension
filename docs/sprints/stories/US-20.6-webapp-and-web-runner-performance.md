## Goal

Make SubWallet's web surfaces — the standalone WebApp and the mobile web-runner —
perform as smoothly as the extension. In the WebApp, the Dapps and Mission Pools
lists should animate and load incrementally instead of jumping in all at once; in
the web-runner, multiple tabs should share one worker instead of each spawning
its own. When this lands, the web surfaces stop being the slow cousin of the
extension build.

## Background

This is the web-surface facet of the umbrella performance program
([#4197](https://github.com/Koniverse/SubWallet-Extension/issues/4197)). The same
`@subwallet/extension-base` background logic runs in three contexts — the
extension service worker, the standalone WebApp, and the mobile web-runner WebView
(NFR-17) — so the web surfaces have their own performance characteristics worth a
dedicated story. Two concrete issues drive it:

- **[#2248](https://github.com/Koniverse/SubWallet-Extension/issues/2248)** — WebApp: the Dapps and Mission Pools lists are missing animations and render all at once; they should animate and load with pagination instead of a single heavy paint.
- **[#2337](https://github.com/Koniverse/SubWallet-Extension/issues/2337)** — web-runner: each tab spins up its own runner; a shared worker should be used so multiple tabs share one runner instance, improving cross-tab performance and resource use.

These two are grouped into one story (user-approved) because they share a single
concern — performance of the *web surface* — even though they target different
parts of it (WebApp UI vs. web-runner worker model). Where the WebApp reads
slow-changing config (Dapps/Mission-Pools metadata), it should read from the
static-data cache ([AD-23](../../ARCHITECTURE.md#architecture-decisions)). This
story owns no FR (cf. the FR-less defending-story pattern of
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)); it defends NFR-17
(cross-platform portability/performance) on the web surfaces specifically.

## Acceptance criteria

- [ ] **AC-1** — **Given** the WebApp Dapps and Mission Pools lists, **When** they load, **Then** they animate in and load with pagination, not a single all-at-once render (#2248).
- [ ] **AC-2** — **Given** the WebApp list reads slow-changing Dapps/Mission-Pools config, **When** it renders, **Then** it reads from the static-data cache (AD-23), not a live per-render fetch.
- [ ] **AC-3** — **Given** multiple web-runner tabs are open, **When** they run, **Then** they share a single worker instance rather than each spawning its own runner (#2337).
- [ ] **AC-4** — **Given** the shared web-runner worker, **When** one tab closes or one tab crashes, **Then** the remaining tabs keep a working runner (shared worker lifecycle is resilient, no orphaned or doubled runner).

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — WebApp Dapps/Mission-Pools metadata is slow-changing and served from the static-data cache via `fetchStaticData`; the WebApp reads the cache rather than fetching per render.
- This story does NOT introduce new AD entries. The web-runner shared-worker change is a runtime/topology optimization, not a new architecture decision.

### Cross-story dependencies

- Sibling [US-20.5](US-20.5-list-rendering-performance.md) — both own render performance and share the **render-cost benchmark** harness; US-20.5 targets the extension selection/collection screens, this story the WebApp/web-runner web surfaces.

### Performance budget

- WebApp Dapps/Mission-Pools lists paginated with animations (no all-at-once paint).
- web-runner shares one worker across tabs (not one runner per tab).
- Measured by the render-cost benchmark + shared-worker count assertion (TASK-20.6.5).
- Story PR description must explicitly confirm these budgets are met.

### What we explicitly did NOT do

- No change to *what* the Dapps/Mission-Pools surfaces show or what the web-runner runs — this story optimizes the web surfaces' render + worker model, not their content or feature behaviour.
- No extension-screen list work — those screens are [US-20.5](US-20.5-list-rendering-performance.md); this story is scoped to the WebApp + web-runner surfaces.

### Points justification

5 pts — §3a-bis "production deliverable": two web-surface optimizations (WebApp
list animation/pagination + web-runner shared worker) sharing one concern, plus a
benchmark. Bounded to the web surfaces — a 5, below the multi-system 8. P2: a
polish/perf improvement, not a blocking defect.
