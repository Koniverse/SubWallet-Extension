## Goal

Make the client talk to the backend economically: stop issuing requests another
in-flight request already covers, cap the number of requests fired in a window,
and make sure no single request class can flood out the others. The headline
defect this closes is the notification fetch that today fans out so hard it
suspends every other request — to the point a user cannot even open the extension.
When this lands, the wallet's request volume is predictable, the backend infra
sees less redundant load, and one noisy feature can never starve the rest.

## Background

Multi-chain data is meant to be aggregated through the SubWallet Services SDK
backend ([AD-24](../../ARCHITECTURE.md#architecture-decisions)) and market /
metadata / media through the cache/CDN proxy layer
([AD-25](../../ARCHITECTURE.md#architecture-decisions)) precisely so the client
does *not* fan a request out per chain. In practice the request economy still
leaks in two ways this story fixes:

- **[#4447](https://github.com/Koniverse/SubWallet-Extension/issues/4447)** — redundant API fan-out: the client re-issues requests that overlap with ones already in flight and fires more per window than the backend should have to absorb, raising infra pressure without adding information.
- **[#4021](https://github.com/Koniverse/SubWallet-Extension/issues/4021)** — the notification-fetch flood: the notification feature fetches so aggressively that it *suspends other requests*, and in the worst case the extension cannot be opened at all while it is mid-flood. This is the concrete, user-visible symptom of an unbounded, un-prioritized request class.

The fix is request-economy plumbing, not new behaviour: deduplicate in-flight
requests, cap per-window request count, and bound/prioritize request classes so a
single class (notifications) can never monopolize the channel. This story defends
NFR-20 (Services SDK aggregation, reduce per-chain RPC fan-out) and NFR-21
(cache/CDN proxy layer) on the request path; it owns no FR (cf. the FR-less
defending-story pattern of
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)).

## Acceptance criteria

- [ ] **AC-1** — **Given** two callers requesting the same resource, **When** the second request is made while the first is still in flight, **Then** they share one request and the duplicate is not sent (no duplicate in-flight requests) (#4447, NFR-20).
- [ ] **AC-2** — **Given** a burst of requests in a refresh window, **When** they are issued, **Then** the total is held under a per-window cap and excess is coalesced/queued rather than fired immediately (#4447).
- [ ] **AC-3** — **Given** the notification fetch running, **When** it is active, **Then** it is bounded/prioritized so it cannot suspend other requests, and the extension remains openable while it runs (#4021).
- [ ] **AC-4** — **Given** aggregated multi-chain data and market/metadata/media, **When** they are fetched, **Then** they route through the Services SDK (AD-24) and the cache/CDN proxies (AD-25) respectively, not direct per-chain fan-out (NFR-20, NFR-21).
- [ ] **AC-5** — **Given** the backend or proxy is briefly unavailable, **When** a request fails, **Then** retries are bounded (no retry storm) and a stale/cached value is served where available, never an unbounded re-fire loop.

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — aggregated balance/fee/swap/XCM/NFT data rides the Services SDK backend; this story enforces that the client routes through it rather than fanning out per chain. The SDK itself is owned by [EPIC-2](../epics/EPIC-2.md).
- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — market/metadata/media goes through the `api-cache` / `static-data` / `ipfs-files` proxies; this story enforces routing, not the proxy build.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Sibling [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) — the idle-quiet lifecycle invariant and this no-redundant-fan-out invariant must agree on when the worker may talk to the network.
- Required by [US-20.4](US-20.4-many-account-submit-performance.md) — that story reuses the **request-count budget harness** set up here for the many-account submit path.

### Performance budget

- Zero duplicate in-flight requests; a per-window request cap holds.
- No request class may starve the others — the notification fetch must not suspend other requests, and the extension stays openable while it runs (#4021).
- Measured by the request-count budget harness (TASK-20.2.6).
- Story PR description must explicitly confirm these budgets are met.

### What we explicitly did NOT do

- No change to what the data *means* — correctness of balances/fees/routing stays with the engine/feature epics; this story only changes *how many* requests fetch it and *in what order*.
- No new backend — the Services SDK (AD-24) and proxies (AD-25) are consumed, not built (owned by [EPIC-2](../epics/EPIC-2.md)).

### Points justification

5 pts — §3a-bis "production deliverable / one external system integration":
request-dedup + per-window cap + class prioritization across the request layer,
plus a standing regression for the #4021 flood. Bounded, single-layer
optimization touching the request/SDK/proxy seam — squarely a 5, below the
multi-system 8.

## Implementation notes

**Audited 2026-07-13 (US-21.2 follow-up). Two of five ACs are already met by shipped
work that no story cited — and the CHANGELOG had the wrong issue number on the biggest
piece of it.**

**The story's own two issues never produced code**: `#4021` and `#4447` have **zero
commits**. But the request-economy work below *did* ship:

| Shipped | Release | What it actually did |
| --- | --- | --- |
| **`#4448`** (PR #4470) | **1.3.47** | The real "Optimize Subscan API request": `strategy/api-request-strategy-v2` — md5-keyed 60 s response cache, group request cancellation, a cache for `getMultiChainBalance`, and skipping reward-history fetches for addresses with no stake. **docs/CHANGELOG.md attributed this bullet to `#4458`, which is wrong** — #4458's commits are chain-list v0.2.108 bumps. Corrected in the changelog. |
| `#4465` | 1.3.52 | Deleted the in-repo `packages/subwallet-api-sdk` and routed aggregated data through the external `@subwallet-monorepos/subwallet-services-sdk` (18 call sites). **This is what satisfies AC-4 / NFR-20 / AD-24**, though its changelog bullet ("Apply sw version when request to Backend") badly undersells it. |
| `#4434` | 1.3.42 | "Optimize Request Handling in SendFund Form" — **only the 300 ms UI debounce shipped**. The Bitcoin-API-cache half of the issue is contained in **no tag** and sits unmerged on `koni/dev/issue-4434-v2`. |
| `#4623` | 1.3.54 | dTAO validator data moved behind a static cache — a request-flood fix. |
| `#4965` | 1.3.75 | User-configurable Subscan API key (raises the rate ceiling the #4448 backoff halves against). |

**AC-by-AC, against the v1.3.83 tree:**

- **AC-1 in-flight dedup — not met.** `api-request-strategy-v2`'s `cacheMap` is written *after* `run()` resolves, so two identical requests in the same tick both miss and both fire. It is a post-hoc response cache, not single-flight — and it is Subscan-only.
- **AC-2 per-window cap — met, pre-existing.** `BaseApiRequestContext` sets `callRate = 2` per `intervalCheck = 1000 ms`; the strategy slices to `callRate` each tick and queues the rest. Scope is the six strategy consumers (Subscan, Runes, Hiro, Bitcoin mainnet/testnet) — there is **no app-wide cap**; notification, price, SDK and RPC traffic bypass it.
- **AC-3 notification-fetch flood (#4021) — not met, and the flood is intact.** `fetchAllAvailBridgeClaimable` is an **unbounded `while (isContinue)` pagination loop** with no page cap and no rate limit, fired 4× per address across every Substrate and EVM address on a 30-minute cron with bare `.then().catch(console.error)`.
- **AC-4 aggregated data via the SDK + cache/CDN — met (via #4465).** Caveat: "no direct per-chain fan-out" is not fully true — `SubscanService` still calls `https://{chain}.api.subscan.io` directly, as does the bridge-indexer path.
- **AC-5 bounded retry + stale-on-error — half met.** Bounded retry is real (`maxRetry = 9`, `reduceLimitRate()` halves `callRate` for 30 s on a rate-limit error — a genuine adaptive backoff). Fallback exists only for *static* data (`fetchStaticData` → bundled JSON, `fetchStaticCache` → default behind a 9 s timeout). There is no stale-value fallback on the SDK request path, and the notification/bridge-indexer path has **neither** retry bounding nor fallback.
- **TASK-20.2.6 harness — does not exist.** No spec asserts request counts, in-flight duplicates or a per-window cap. US-20.4 depends on this harness.
