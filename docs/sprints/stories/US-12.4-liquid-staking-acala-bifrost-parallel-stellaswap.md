## Goal

A user can liquid-stake into LST protocols (Acala LDOT, Bifrost vToken — vDOT
plus the Bifrost-Manta variant, Parallel sDOT and StellaSwap stDOT) and receive
a liquid derivative token they can hold or use elsewhere while still earning
staking yield, so that staking no longer locks the underlying for the full
unbonding period.

## Background

Liquid staking mints a derivative (e.g. a vToken) against staked principal,
keeping the position liquid. Unlike native staking, the protocol — not the native
pallet — owns the bond/redeem semantics, so the engine models it through the
`BaseSpecialStakingPoolHandler` → liquid-staking branch
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)), which integrates each LST
protocol's mint/redeem calls. The shipped liquid-staking handlers are **Acala**
(`acala.ts`), **Bifrost** (`bifrost.ts` + the **Bifrost-Manta** variant
`bifrost-manta.ts`), **Parallel** (`parallel.ts`) and **StellaSwap** (stDOT on
Moonbeam, `stella-swap.ts`) — all under
`services/earning-service/handlers/liquid-staking/`. This story is the
liquid-staking *screen*: it subscribes to the special-pool handler subjects
(position, derivative balance, exchange rate, APY) and submits the protocol
mint/redeem extrinsics, reusing the earning-list surface from
[US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md).

**Note**: Interlay is *lending*, not liquid staking — its handler lives under
`handlers/lending/interlay.ts` and is owned by
[US-12.7](US-12.7-lending-interlay.md); it is deliberately not part of this story.

Materializes [FR-117](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** a supported LST protocol, **When** the user stakes an
  amount, **Then** the mint extrinsic submits through the special-pool handler and
  the user's derivative-token balance and position (with current exchange rate +
  APY) appear in the earning list.
- [x] **AC-2** — **Given** a derivative-token holding, **When** the user redeems
  (instant or via the protocol's unbonding queue), **Then** the redeem extrinsic
  submits and the position / claimable-principal state updates from the subject.
- [x] **AC-3** — **Given** the protocol exchange rate is stale or unavailable,
  **When** the user opens the stake flow, **Then** the estimated derivative amount
  is shown as unavailable rather than a wrong figure, and submit is blocked until a
  fresh rate is available.

### Architecture constraints

- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — liquid staking is the `BaseSpecialStakingPoolHandler` → liquid-staking branch; protocol-specific mint/redeem semantics live in the handler subclass, this screen consumes the subjects.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the special-pool (liquid-staking) handler subjects.

- Sibling [US-12.7](US-12.7-lending-interlay.md) — both ride the `BaseSpecialStakingPoolHandler` branch; coordinate review on shared special-pool screen code.

### Points justification

5 pts — §3a-bis "1 external-system integration": integrating the external LST
protocol family (Acala / Bifrost (+ Bifrost-Manta) / Parallel / StellaSwap) —
each with its own mint/redeem and exchange-rate semantics behind one shared
special-pool screen — plus the derivative-token position surface and the
unbonding-queue redeem path. The handlers share the `BaseSpecialStakingPoolHandler`
→ liquid-staking seam (one external protocol family, not one integration per
provider), so the family sizes at 5.
