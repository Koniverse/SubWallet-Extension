## Goal

A user can move assets between Ethereum and Polkadot Asset Hub directly in the
wallet via Snowbridge — seeing the bridge fee and estimated arrival time before
signing — so that ETH-ecosystem value can flow into Polkadot (and back) without
a separate bridge dApp.

## Background

Snowbridge is the first external (non-XCM) bridge in the epic: a trustless
ETH ↔ Polkadot Asset Hub bridge. It hangs off the bridge-hop interface
established by [US-13.1](US-13.1-xcm-parachain-transfers.md), adding a
Snowbridge-specific provider handler behind the shared quote → submit → track
lifecycle. Bridge construction still follows the delegate-don't-build posture
([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the Snowbridge provider
is reached through the SubWallet-hosted proxy so no provider key ships in the
bundle ([AD-19](../../ARCHITECTURE.md#architecture-decisions), NFR-16).

Because the bridge spans Ethereum finality and Polkadot consensus, transit is
asynchronous and can take meaningful wall-clock time; the story must surface
in-transit status rather than reporting funds as lost while the bridge settles.

Materializes [FR-129](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** an account with a Snowbridge-supported asset on
  Ethereum (or Asset Hub), **When** the user requests a bridge to the other side,
  **Then** the wallet shows the bridge fee, estimated destination amount, and ETA
  before any signature.
- [ ] **AC-2** — **Given** a confirmed Snowbridge transfer, **When** the user
  signs, **Then** the source-side transaction is submitted and tracked, and the
  destination balance updates on settlement.
- [ ] **AC-3** — **Given** a Snowbridge transfer that has left the source chain
  but not yet settled (or that stalls), **When** the user returns, **Then** it is
  shown as in-transit with recovery guidance — funds are never reported as lost.
- [ ] **AC-4** — **Given** the Snowbridge provider, **When** the wallet calls it,
  **Then** the call goes through the SubWallet-hosted proxy and no Snowbridge
  provider key is present in the extension bundle (AD-19 / NFR-16).

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction is delegated; the Snowbridge handler consumes the provider, it does not reimplement bridge logic.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — the keyed Snowbridge provider is reached via the SubWallet-hosted proxy; no key in the bundle (NFR-16).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) — uses the bridge-hop interface and quote/track surface; sets up the **bridge provider-handler fixture** the later external-bridge stories reuse.
- Sibling [US-13.3](US-13.3-avail-bridge.md), [US-13.4](US-13.4-polygon-unified-bridge.md), [US-13.5](US-13.5-across-protocol.md) — each adds a provider handler against the same interface; coordinate the shared fixture.

### Points justification

5 pts — §3a-bis "1 external system integration": one external bridge provider
(Snowbridge) integrated behind the existing bridge-hop interface, including its
async-settlement status handling and key-proxy wiring. The interface and quote
surface already exist (US-13.1), so this is a single-provider integration, not a
multi-system base.
