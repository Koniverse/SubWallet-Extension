## Goal

A user can cash out — sell crypto they hold for fiat — without leaving SubWallet,
by being handed off to Transak's hosted sell flow with a pre-built order URL for
the right asset, network, and address. This closes the loop opened by the
on-ramp: money in (buy) and money out (sell) ride the same provider hand-off and
the same backend-minted order-URL path, differing only by the `action`.

## Background

Selling is the symmetric half of the fiat-gateway: the same partner (Transak)
that serves the on-ramp also serves the off-ramp via the web app, so the
off-ramp does **not** introduce a new provider or a new integration surface — it
reuses the Transak adapter and the Services-SDK order-URL path from
[US-14.1](US-14.1-fiat-on-ramp-buy-crypto-with-card.md), calling
`subwalletApiSdk.transakApi.generateOrderUrl({ ..., action: 'SELL' })` instead of
`'BUY'`. The partner secret stays server-side
([AD-19](../../ARCHITECTURE.md#architecture-decisions),
[AD-24](../../ARCHITECTURE.md#architecture-decisions), NFR-16 / NFR-20); SubWallet
never receives the fiat — Transak handles KYC, payout, and settlement.

Materializes [FR-139](../../PRD.md#functional-requirements). This story is **Retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during version
reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** a token the user holds that Transak supports for sell,
  **When** the user starts a sell, **Then** Transak's hosted sell flow opens with
  an order URL carrying `action: 'SELL'` and the correct asset, network, and
  source address.
- [ ] **AC-2** — **Given** the sell order URL, **When** it is built, **Then** it is
  minted by the Services SDK (`transakApi.generateOrderUrl`) and **no** Transak
  secret credential is present in the shipped bundle (same invariant as the buy
  path).
- [ ] **AC-3** — **Given** a token that Transak does not support for off-ramp,
  **When** the user opens the sell surface, **Then** the option is unavailable/
  disabled with guidance rather than an errored screen (unhappy path: no off-ramp
  support for the asset).
- [ ] **AC-4** — **Given** the user's region is not serviceable for sell, or their
  KYC is rejected by Transak, **When** they reach the hosted flow, **Then** the
  rejection is handled by Transak and the user returns to the wallet without a
  partial or stuck order state (unhappy path: provider unavailable in region /
  KYC rejection).

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — the sell order URL is built by the Services SDK backend (`transakApi.generateOrderUrl`), reusing the buy path's substrate with `action: 'SELL'`.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) / NFR-16 — the Transak partner secret stays behind the backend; the off-ramp adds no new bundled credential.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-14.1](US-14.1-fiat-on-ramp-buy-crypto-with-card.md) — reuses the Transak `CreateBuyOrderFunction` adapter and the Services-SDK `generateOrderUrl` path; off-ramp is the same call with `action: 'SELL'`.

### What we explicitly did NOT do

- No second off-ramp provider — Transak only, per FR-139. Trigger to revisit: a second partner offering off-ramp coverage for SubWallet's assets/regions.
- No in-product payout/settlement — Transak owns KYC, fiat payout, and settlement; SubWallet never takes fiat custody.
