## Goal

Keep every reward and APY figure the user reads on the earning surface
*correct*. A displayed APY or a claimable-reward total that does not match what
the chain actually pays erodes trust the instant a user compares the number to
what lands in their wallet — and on staking, that comparison happens every
reward era. This story is one half of the earning epic's hardening cluster: it
owns **correctness** of the reward/APY numbers (the sibling
[US-12.14](US-12.14-earning-performance-and-cache-hardening.md) owns *speed* of
the earning surface). It ships no new feature — it defends the accuracy of the
reward and APY figures every other earning story renders.

## Background

The earning epic's bug backlog splits cleanly into two genuinely different
logics, and conflating them produced fixes that chase the wrong root cause.
This story owns the first logic: **reward / APY-accuracy** — a correctness
problem, where a figure is computed or shown wrong regardless of how fast the
surface loads. (The second logic — earning **performance & stale cache** — is
its own story, [US-12.14](US-12.14-earning-performance-and-cache-hardening.md);
this split corrects an earlier mislabel that filed the performance issue #2615
under an "APY-formula accuracy" banner. #2615 is a performance + stale-cache
issue and belongs to US-12.14, not here.)

The accuracy backlog gathers issues that all share one failure mode — the number
shown ≠ the number the chain pays — but across different pool types and reward
mechanics:

- [#3527](https://github.com/Koniverse/SubWallet-Extension/issues/3527) — relaychain
  *direct* (validator) staking APY computes wrong, distinct from nomination-pool
  APY; the validator-path APY needs its own correctness check.
- [#2708](https://github.com/Koniverse/SubWallet-Extension/issues/2708) — staking
  **rewards** figure is wrong (claimable / accrued reward total mismatched).
- [#4135](https://github.com/Koniverse/SubWallet-Extension/issues/4135) — rewards
  do not re-display correctly after a **Claim reward** on Mythos (collator path);
  post-claim reward state is stale/incorrect.
- [#2975](https://github.com/Koniverse/SubWallet-Extension/issues/2975) — rewards
  must be shown for **all** staking methods, not only the subset that currently
  surface them (coverage gap across pool types).
- [#2755](https://github.com/Koniverse/SubWallet-Extension/issues/2755) — the
  **earning preview** (pre-stake projected APY / reward) shows wrong figures
  before the user commits.

The fix lives at the seam the engine publishes: reward and APY figures come from
the `BasePoolHandler` tree ([AD-22](../../ARCHITECTURE.md#architecture-decisions))
and accuracy is verified *at that seam* against each pool type's real reward
formula — the screen subscribes to the handler subjects, it does not re-derive
the math. This story owns **no FR** — it is the reward/APY-accuracy half of the
earning hardening cluster (cf.
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md),
[US-11.8](US-11.8-cross-chain-swap-routing.md) pattern of an FR-less defending
story).

## Acceptance criteria

- [ ] **AC-1** — **Given** relaychain *direct* (validator) staking, **When** APY
  is displayed, **Then** the validator-path APY matches the chain's reward
  formula for direct staking (distinct from the nomination-pool APY path),
  asserted by a validator-APY accuracy test
  ([#3527](https://github.com/Koniverse/SubWallet-Extension/issues/3527)).
- [ ] **AC-2** — **Given** an account with accrued/claimable staking rewards,
  **When** the reward total is displayed, **Then** the figure matches the chain's
  reward state within tolerance, across staking methods
  ([#2708](https://github.com/Koniverse/SubWallet-Extension/issues/2708),
  [#2975](https://github.com/Koniverse/SubWallet-Extension/issues/2975)).
- [ ] **AC-3** — **Given** a Mythos collator position, **When** the user claims a
  reward, **Then** the displayed rewards re-render to the correct post-claim
  state (no stale/duplicated claimable figure)
  ([#4135](https://github.com/Koniverse/SubWallet-Extension/issues/4135)).
- [ ] **AC-4** — **Given** every supported pool type (native / pool / collator /
  dApp / liquid / lending / dTAO), **When** the earning preview projects APY and
  reward before a stake, **Then** the projected figure matches the per-pool-type
  reward formula within a defined tolerance
  ([#2755](https://github.com/Koniverse/SubWallet-Extension/issues/2755)).
- [ ] **AC-5** *(unhappy path)* — **Given** a pool whose reward data is
  unavailable for an account (no reward feed for that method), **When** the
  earning surface renders, **Then** rewards are shown as `unavailable` for that
  position rather than silently `0` (a wrong-but-confident figure), and the
  other positions still show correct rewards.

## Dev notes

### Architecture constraints

- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — reward/APY figures come from the `BasePoolHandler` tree; accuracy is verified **at that seam** against each pool type's reward formula, not re-derived per screen. The screen subscribes to the handler subjects.
- This story does NOT introduce new AD entries.
- This story does NOT touch the static-data cache or Services-SDK aggregation (AD-23 / AD-24) — those govern *speed* of the earning surface and are the sibling [US-12.14](US-12.14-earning-performance-and-cache-hardening.md)'s concern.

### Cross-story dependencies

- **Builds on** [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — verifies reward/APY accuracy at the `EarningService` handler seam every earning story consumes; reuses its mock pool-handler subject fixture for known-reward cases.
- **Builds on** [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md) — the relaychain direct/validator staking path whose APY #3527 reports wrong.
- **Builds on** [US-12.8](US-12.8-earning-path-simulation-and-step-tracking.md) — the earning-preview projection (#2755) rides the simulation path; projected reward/APY accuracy is part of the figures users trust pre-stake.
- **Hardens** [US-12.1](US-12.1-native-nomination-staking-relay-and-parachains.md)–[US-12.7](US-12.7-lending-interlay.md) — the reward/APY display every pool-type story renders.
- **Sibling** [US-12.14](US-12.14-earning-performance-and-cache-hardening.md) — the *speed/cache* half of the earning hardening cluster; coordinate so accuracy assertions are not masked by stale-cache reads.

### What we explicitly did NOT do

- No change to the reward **formulas** themselves — the formulas live in the engine handlers (US-2.3); this story verifies the *displayed figure matches them*, it does not redefine them.
- No performance/cache work — earning-load speed, cold-start paint, static-cache reads and removed-account stale data are the sibling [US-12.14](US-12.14-earning-performance-and-cache-hardening.md)'s scope.
- No new earning protocol or pool type — hardening only.

### Points justification

5 pts — §3a-bis "production deliverable / 1 integration surface, 3-4 days". This
is a single-logic hardening cluster: reward/APY **correctness** verified across
the seven pool types at one engine seam (AD-22), absorbing a focused five-issue
bug cluster (#3527, #2708, #4135, #2975, #2755). It does **not** span the
cache/aggregation/load systems (that was the other half, now
[US-12.14](US-12.14-earning-performance-and-cache-hardening.md)), so it sizes
below the original combined 8-pt story: the split removes the multi-system
cache+SDK+memory surface, leaving one seam + per-pool-type accuracy tests, which
calibrates at 5 alongside the US-7.7-class single-concern defending story.
