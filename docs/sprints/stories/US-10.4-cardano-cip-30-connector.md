## Goal

A Cardano dApp connects to SubWallet through the standard **CIP-30** wallet
connector — `connect`, `signData`, `signTx`, `submitTx`, `getUtxo` — so Cardano
users can interact with the Cardano dApp ecosystem from the same wallet that holds
the rest of their assets.

## Background

CIP-30 is the Cardano dApp-connector standard: the wallet injects an object under
`window.cardano[<name>]` exposing `enable()`, and the enabled API provides
`getUsedAddresses` / `getUtxos` / `getBalance` reads plus `signData`, `signTx`, and
`submitTx`. SubWallet's Cardano model ([AD-14](../../ARCHITECTURE.md#architecture-decisions))
serves chain data through Blockfrost (via the backend proxy) and exposes a
standards-based CIP-30 connector so Cardano dApps treat it like any compliant
wallet. The Cardano serialization needs `@emurgo/cardano-serialization-lib-browser`
(wired into the koni/web bundles).

`enable()` is authorization-gated (US-10.7); `signData` / `signTx` are routed to the
Cardano request handler in `RequestService` ([AD-21](../../ARCHITECTURE.md#architecture-decisions))
behind the shared `PopupHandler`, and signing happens in the background keyring
(AD-04). `submitTx` broadcasts the signed transaction.

Materializes [FR-97](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a Cardano dApp, **When** it calls `window.cardano.<name>.enable()`
  from an unauthorized origin, **Then** the connect request is surfaced for approval;
  the API is returned only on approval.
- [x] **AC-2** — **Given** an enabled API, **When** the dApp calls `getUtxos` /
  `getUsedAddresses` / `getBalance`, **Then** it receives the authorized account's
  UTxOs / addresses / balance.
- [x] **AC-3** — **Given** an enabled API, **When** the dApp calls `signData` or
  `signTx`, **Then** the request routes to the Cardano request handler, is surfaced
  for approval, and is signed in the background keyring.
- [x] **AC-4** — **Given** a signed transaction, **When** the dApp calls `submitTx`,
  **Then** it is broadcast and the resulting tx id is returned.
- [x] **AC-5** — **Given** an unauthorized origin or a user rejection, **When** the
  request resolves, **Then** it rejects with the CIP-30 error shape (e.g. `APIError`
  / user-declined) and no address, UTxO, or signature leaks.

## Dev notes

### Architecture constraints

- [AD-14](../../ARCHITECTURE.md#architecture-decisions) — Cardano data via Blockfrost (backend proxy), connector follows CIP-30; staking/delegation deferred.
- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — `signData` / `signTx` route to the Cardano request handler behind the shared `PopupHandler`.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — signing runs in the background keyring.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-10.7](US-10.7-dapp-authorization-ui-per-origin.md) — `enable()` honors per-origin authorization.
- Builds on EPIC-2 US-2.7 — Cardano sign requests enqueue through `RequestService`.
- Sibling [US-10.8](US-10.8-arbitrary-message-signing.md) — CIP-30 `signData` is the Cardano arm of arbitrary message signing.

### Sizing

Sized 5: one external standard surface (CIP-30) for one ecosystem — inject
namespace + read methods + `signData`/`signTx`/`submitTx` + serialization-lib
wiring + Blockfrost-backed reads.
