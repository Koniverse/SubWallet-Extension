## Goal

A user can put their idle DOT/KSB (and parachain-native tokens) to work by
nominating validators directly from the wallet — bond, nominate, bond-extra,
unbond, rebond, withdraw-unbonded and claim — and see the resulting position,
APY and unbonding timers, so that native staking happens in-wallet without a
separate staking UI. This is the first earning screen and it sets up the
pool-handler subject fixture every later staking story reuses.

## Background

Native nomination staking is the canonical yield mechanism on Polkadot, Kusama
and Substrate parachains: a holder nominates a validator set and the staking
pallet rewards the bonded stake. The mechanism itself is modeled by the engine:
`EarningService`'s `BaseNativeStakingPoolHandler` → relay/para subclasses expose
the position, targets (validator list) and reward data over RxJS subjects
([AD-22](../../ARCHITECTURE.md#architecture-decisions)), shipped by
[EPIC-2](../epics/EPIC-2.md)
([US-2.3](US-2.3-earningservice-pool-handler-engine.md)). This story owns the
*screen and the write flows* — it subscribes to the handler subjects and submits
the bond / nominate / unbond / withdraw / claim extrinsics through them. Heavy
validator-target lists are read from the static-data cache
([AD-23](../../ARCHITECTURE.md#architecture-decisions)) rather than polled live.

The native-nomination handler tree (FR-114) also covers additional Substrate
parachains already shipped — **Mythos, Tanssi, Amplitude and Energy** (handlers
under `services/earning-service/handlers/native-staking/`) — so they are
documented as shipped here under FR-114 rather than in
[US-12.12](US-12.12-staking-for-additional-networks-enjin-phala-xx.md) (FR-125,
which scopes the *planned* Enjin / Phala / xx batch).

Materializes [FR-114](../../PRD.md#functional-requirements). **Retroactive** — already shipped;
`commit` / `version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a relay-chain account with a free balance above the
  minimum bond, **When** the user bonds and nominates a validator set, **Then**
  the extrinsic submits through the native handler and the new staking position
  (bonded amount, nominated validators, APY) appears in the earning list.
- [x] **AC-2** — **Given** an active nomination position, **When** the user
  bond-extra / unbond / rebond / withdraw-unbonded / claims rewards, **Then**
  each action submits the corresponding native extrinsic and the position and
  unbonding timer update from the handler subject.
- [x] **AC-3** — **Given** the validator-target list, **When** the staking screen
  loads, **Then** targets are read from the static-data cache (AD-23), not a live
  per-render RPC sweep.
- [x] **AC-4** — **Given** a bond amount below the chain minimum or one that would
  drop the account below the existential deposit, **When** the user tries to
  submit, **Then** the flow blocks with a clear validation error and submits
  nothing.

## Dev notes

### Architecture constraints

- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — staking is modeled by the `BasePoolHandler` tree; this screen consumes the native handler subjects and submits through them, it does not add per-chain conditionals to shared logic.
- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — validator/collator target lists and pool info are served from the static-content cache via `fetchStaticData`.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the native-staking handler subjects and the `EarningService` RxJS wiring.
- Required by [US-12.2](US-12.2-nomination-pool-staking.md), [US-12.3](US-12.3-collator-parachain-staking.md), [US-12.5](US-12.5-dapp-staking-astar.md) — they reuse the pool-handler subject fixture and the earning-list screen set up here.

### Points justification

5 pts — §3a-bis "production deliverable / 1 external-system integration": this is
the first end-to-end earning surface (list + position detail + five native write
flows + validator selection) wiring the screen to the native pallet via the
handler tree, plus it stands up the shared pool-handler subject fixture. First
full staking flow sizes at 5.
