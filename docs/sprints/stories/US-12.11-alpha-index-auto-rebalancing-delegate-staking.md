## Goal

A user can delegate-stake into an auto-rebalancing alpha-index strategy — a single
position that spreads stake across multiple Bittensor subnets and periodically
rebalances to track an index — so that a holder gets diversified subnet exposure
without manually staking and re-weighting each subnet themselves.

## Background

This is a *strategy* product, not a single-subnet stake: the position tracks an
alpha index across several subnets and rebalances over time (move stake between
subnets to maintain target weights). It builds on the dTAO subnet-staking
foundation ([US-12.6](US-12.6-bittensor-dtao-subnet-staking.md)) but adds index
composition, target-weight tracking and the rebalancing engine — multiple subnet
positions managed as one, with the rebalance schedule driven from index data.
Per [AD-15](../../ARCHITECTURE.md#architecture-decisions) all staking runs the
native Subtensor path; index composition/target data is assembled through the
Services SDK ([AD-24](../../ARCHITECTURE.md#architecture-decisions)) and the
position is modeled on the handler tree
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)).

Materializes [FR-124](../../PRD.md#functional-requirements). 📋 **Planned / forward** — not yet started;
this story scopes the capability ahead of any work. Tracked by
[#4946](https://github.com/Koniverse/SubWallet-Extension/issues/4946) — Support
Auto-Rebalancing Index Strategies (Delegate staking) for Bittensor.

## Acceptance criteria

- [ ] **AC-1** — **Given** an alpha-index definition (subnets + target weights),
  **When** the user delegate-stakes an amount, **Then** stake is distributed across
  the index's subnets per target weights via the native path and a single index
  position appears in the earning list.
- [ ] **AC-2** — **Given** an index position whose subnet weights have drifted,
  **When** a rebalance is triggered (scheduled or user-initiated), **Then** stake
  is moved between subnets to restore target weights and the position reflects the
  new allocation.
- [ ] **AC-3** — **Given** an index position, **When** the user unstakes, **Then**
  the position is unwound across its subnets and the aggregate alpha is returned.
- [ ] **AC-4** — **Given** a rebalance that partially fails (one subnet move
  reverts), **When** the user returns, **Then** the index position is consistent
  and recoverable — no subnet is left in a silently wrong-weight state.

## Dev notes

### Architecture constraints

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — all subnet staking and rebalancing run the native Subtensor path.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — the index position is modeled as a handler-backed aggregate over per-subnet stakes; the screen consumes its subjects.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — index composition / target-weight data is assembled through the Services SDK.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.6](US-12.6-bittensor-dtao-subnet-staking.md) — composes per-subnet stakes into an index strategy.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the (planned) index handler subjects.

### Points justification

8 pts — §3a-bis "multi-system integration": this is more than one external
integration — it composes per-subnet native staking + an index-data/Services-SDK
feed + a rebalancing engine with partial-failure consistency, all managed as one
position. Multi-system strategy with non-trivial failure semantics sizes at 8 (and
is at the split threshold — keep scope to one index, defer additional strategies).
