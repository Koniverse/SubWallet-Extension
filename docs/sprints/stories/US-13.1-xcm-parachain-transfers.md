## Goal

A user holding a token on one Polkadot/Kusama parachain can move it to another
parachain (or the Relay Chain / Asset Hub) directly inside the wallet — seeing
the fee and estimated destination amount before they sign — so that they can
rebalance across the ecosystem without leaving SubWallet or touching a separate
bridge UI. This story is the **XCM base** of the epic: it establishes the
ParaSpell-delegated transfer construction, the quote/fee surface, and the
per-route runtime toggle that every other bridge story is sized against.

## Background

This is the cross-chain primitive at the heart of the bridge-xcm epic. XCM
transfer construction is delegated to the ParaSpell API
([AD-18](../../ARCHITECTURE.md#architecture-decisions)) and upgraded through its
versions (v4 → v5 → v1 API) rather than built or forked in-house — forking would
mean tracking every Polkadot runtime upgrade ourselves. XCM transfers are
coordinated in `extension-base` using ParaSpell-compatible `MultiLocation` v3
formatting, and the route quote/fee assembly rides the Services SDK backend
([AD-24](../../ARCHITECTURE.md#architecture-decisions)).

The defining safety property is the **per-route runtime toggle**
([AD-09](../../ARCHITECTURE.md#architecture-decisions), NFR-15): individual XCM
routes are configurable per chain pair and can be disabled at runtime via the
online chain-list update, without an extension release. This exists so that a
partner-chain security incident (the canonical example being the Acala 2022
incident, issue #667) can be answered in minutes by disabling the affected route
rather than waiting on a full release cycle. A disabled route must vanish from
the quote and selection surfaces entirely.

Materializes [FR-126](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during version
reconciliation. It is the base that [US-13.2](US-13.2-snowbridge-eth-asset-hub.md)
through [US-13.7](US-13.7-bittensor-tao-subtensor-evm-bridge.md) build the
external-bridge providers on top of, and the engine that
[EPIC-11](../epics/EPIC-11.md) ([US-11.8](US-11.8-cross-chain-swap-routing.md))
and [EPIC-12](../epics/EPIC-12.md)
([US-12.9](US-12.9-xcm-deposit-routing-into-earning.md)) consume.

## Acceptance criteria

- [x] **AC-1** — **Given** an account holding a token on a source parachain and a
  reachable destination chain, **When** the user requests an XCM transfer, **Then**
  the wallet builds the hop via ParaSpell (AD-18) and shows the fee, estimated
  destination amount, and an ETA before any signature is requested.
- [x] **AC-2** — **Given** a confirmed XCM transfer, **When** the user signs and
  submits, **Then** the source-chain extrinsic is submitted, the transaction is
  tracked through the shared status machine, and the destination balance updates
  on arrival.
- [x] **AC-3** — **Given** an XCM route disabled at runtime via the online
  chain-list (AD-09 / NFR-15), **When** the user opens the bridge surface, **Then**
  that route is not quotable, not selectable, and not submittable — while every
  other route stays available, and no extension release was required.
- [x] **AC-4** — **Given** a destination chain whose data source is unreachable at
  quote time, **When** the user requests a transfer, **Then** the quote fails
  closed with a clear error (no silent best-effort submit) rather than building an
  unpriced hop.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — XCM construction is delegated to ParaSpell; this story consumes the ParaSpell API, it never reimplements XCM logic. Forecloses any hand-maintained `MultiLocation` encoder.
- [AD-09](../../ARCHITECTURE.md#architecture-decisions) — per-chain XCM route toggle; route availability is data-driven from the online chain-list, disable-able at runtime with no release.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — route quote / multi-chain assembly flows through the Services SDK backend.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Required by [US-13.2](US-13.2-snowbridge-eth-asset-hub.md), [US-13.3](US-13.3-avail-bridge.md), [US-13.4](US-13.4-polygon-unified-bridge.md), [US-13.5](US-13.5-across-protocol.md), [US-13.7](US-13.7-bittensor-tao-subtensor-evm-bridge.md) — each external-bridge provider hangs off the bridge-hop interface and quote surface established here.
- Required by [US-11.8](US-11.8-cross-chain-swap-routing.md) (EPIC-11) — cross-chain swap routing consumes this XCM hop as its bridge step.
- Required by [US-12.9](US-12.9-xcm-deposit-routing-into-earning.md) (EPIC-12) — XCM deposit into earning consumes this hop.
- Sibling [US-13.11](US-13.11-xcm-runtime-upgrade-paraspell-version-hardening.md) — regression-guards the per-route toggle and ParaSpell version drift introduced here.

### Performance budget

- Bridge quote: fee + ETA returned in ≤ 1 round-trip through the proxied provider per quote request (epic §Performance budgets).
- Route-toggle propagation: a route disabled in the online chain-list is hidden on next chain-list refresh, no extension release.

### Points justification

8 pts — §3a-bis "multi-system integration". This is not a single external
provider: it is the XCM base that integrates ParaSpell construction (AD-18), the
Services-SDK quote backend (AD-24), the shared transaction status machine, AND
the runtime-driven per-route toggle (AD-09) — spanning the chain-list data layer,
the bridge handler, and the transaction lifecycle. It is also the interface every
other bridge story depends on, so its contract surface is larger than a
single-provider integration (which sizes at 5). One step below the 13 split
threshold; the toggle + multi-route base is the reason it is 8 rather than 5.

> *"display incorrect transferable balance in the Send Fund/Donate/XCM screen"* — is 0.4.4 as well.
> A cross-chain transfer has to reserve destination fees out of the sending balance, so the number
> the screen shows and the number the extrinsic can spend diverge on day one. The whole of
> [US-13.14](US-13.14-xcm-fee-dry-run-and-validation.md) descends from that.
>
> **#552 is XCM meeting hardware signing.** Kusama, Polkadot, Astar and Shiden XCM transfers failed
> for QR-signer accounts (0.5.7) — an XCM payload is larger and differently shaped than a plain
> transfer, so the multi-frame QR path in
> [US-16.2](US-16.2-qr-signing-keystone-and-polkadot-vault.md) had to grow to carry it.
>
> **#1505 is the browser, not the chain.** *"Error page when performing XCM on Firefox"* (1.0.8).
> Nothing about the route was wrong.
