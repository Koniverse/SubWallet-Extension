## Goal

The MV3 background wakes **proportionally to the request that woke it**: a public
`pub(…)` message starts the worker's core only, while a privileged `pri(…)` /
`mobile(…)` message additionally starts the 11 data services. A dApp asking for an
account list no longer boots the entire wallet runtime.

## Background

Carved out of [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) on 2026-07-14.
US-20.1 was one story covering the whole **#4427 "Improve SubWallet Lifecycle"** program
— P1 (shipped), P2 and P3 (never started). One story cannot carry two truths: its status
was `in-progress`, which was false in both directions. **P1 is a shipped capability and
gets the retroactive story every other shipped capability in this repo has**; the
unstarted phases stay in US-20.1, honestly at `backlog`. See [CONTEXT D100](../../CONTEXT.md).

This is the **P1 phase — [#4428](https://github.com/Koniverse/SubWallet-Extension/issues/4428)**,
"Optimize Lifecycle Management - P1", shipped **1.3.42**. `ActionHandler._onPortMessage`
derives `requireFullActive` from the message prefix (`pri(` / `mobile(` → `wakeup(true)`,
`pub(` → `wakeup(false)`), and only a full wake runs `_startFull()` — the 11 data services
(cron, subscription, history, price, balance, nft, earning, swap, notification, openGov,
multisig). This is the runtime half of [AD-20](../../ARCHITECTURE.md#architecture-decisions),
riding on the MV3 substrate ([AD-08](../../ARCHITECTURE.md#architecture-decisions), EPIC-1).

**It shipped with a regression, and the fix is part of this story.** #4428 set
`isFullActive = true` on a `pri()` wake but never reset it on last-port-disconnect — so
after the first sleep, the next `pri()` skipped `_startFull()` and **Home / Earning failed
to load**. Fixed by `a0bbe26a98` (**#4478**, release **1.3.43**), a one-line reset. No
lifecycle test existed to catch it — which is why TASK-20.1.7 is now forward work in
US-20.1.

Materializes [NFR-8](../../PRD.md#non-functional-requirements) (MV3 shutdown/wake state
handling) on the wake path. **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** the MV3 worker is asleep, **When** a `pub(…)` message arrives,
  **Then** it wakes with `wakeup(false)` and `_startFull()` does **not** run — the 11 data
  services stay stopped.
- [x] **AC-2** — **Given** the MV3 worker is asleep, **When** a `pri(…)` or `mobile(…)`
  message arrives, **Then** it wakes with `wakeup(true)` and `_startFull()` starts the data
  services.
- [x] **AC-3** — **Given** the worker has completed a full wake and then slept, **When** a
  second `pri(…)` message arrives, **Then** `_startFull()` runs again — `isFullActive` is
  reset on last-port-disconnect, so the wake is **idempotent across sleep cycles** (#4478,
  1.3.43).

## Tasks

## Dev notes

### Architecture constraints

- [AD-20](../../ARCHITECTURE.md#architecture-decisions) — the four-state lifecycle (Init → Start-Partially → Start-Fully → Sleep). This story materializes the **wake-depth** half of it at runtime.
- [AD-08](../../ARCHITECTURE.md#architecture-decisions) — the MV3 service-worker substrate this rides on; owned by [EPIC-1](../epics/EPIC-1.md), not re-decided here.
- This story does NOT introduce new AD entries.

### What shipped is narrower than AD-20 promises

**The split is by *service*, not by *chain*.** `_start()` calls `resumeAllNetworks()` →
`chainService.resumeAllChainApis()`, which wakes **every active chain** on a partial wake
just as on a full one. AD-20's "a request that needs only a few chains never starts all of
them" is **not true in the shipped code** — chain-level granularity was never built. That
gap is forward work and stays in
[US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) AC-1.

State this plainly rather than letting the AD imply otherwise: a doc-stated invariant is a
claim, not evidence ([LESSONS §64](../../LESSONS.md)).

### Cross-story dependencies

- Builds on [US-1.1](US-1.1-mv3-service-worker-background.md) — the MV3 service-worker substrate. Sleep-on-last-disconnect and `SLEEP_TIMEOUT` (60 s) pre-date this story: they came from `c2046714a6` "Optimize MV3 lifecycle", shipped **1.2.1**, not from #4428.
- Remaining lifecycle phases: [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) (P2 #4445, P3 #4446 — `backlog`, unstarted).

### Points justification

5 pts — a state-machine change across the background's entry path (`ActionHandler`,
`connectionMap`, service start gating), plus the idempotency fix. No new external system,
no user-facing surface.
