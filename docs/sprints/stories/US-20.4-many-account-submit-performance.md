## Goal

Stop the screen freezing when a user with many accounts submits a transaction and
then closes the history popup. Per-account work on the submit/close path must be
bounded and kept off the critical UI thread, so the wallet stays responsive at any
account count instead of grinding to a halt for power users. When this lands, the
account count is no longer a cliff the submit flow falls off.

## Background

This is the many-account facet of the umbrella performance program
([#4197](https://github.com/Koniverse/SubWallet-Extension/issues/4197)). The
concrete defect is
**[#4984](https://github.com/Koniverse/SubWallet-Extension/issues/4984)**: with
many accounts, submitting a transaction and then *closing the history popup*
freezes the screen. The shape of the bug is main-thread blocking — work that
scales with account count (subscription teardown, history reconciliation, per-
account recomputation triggered by the popup lifecycle) runs synchronously on the
UI path, so the more accounts a user has, the longer the freeze, up to an
unresponsive screen.

The fix is to bound and offload that per-account work: nothing on the
submit/close path should be O(accounts) on the main thread. This rides the same
lightweight read path the rest of the wallet uses
([AD-07](../../ARCHITECTURE.md#architecture-decisions)) — the point is that the
*work*, not just the *memory*, must stay bounded as accounts multiply. This story
owns no FR (cf. the FR-less defending-story pattern of
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)); it defends the main-thread responsiveness of the submit/close path — a property the PRD does **not** state (NFR-11 is memory-by-chain-count, not main-thread-by-account-count; recorded as a PRD gap in US-21.2)
(performance under scale) on the many-account submit path specifically.

## Acceptance criteria

- [ ] **AC-1** — **Given** a wallet with many accounts (≥ a large fixture count), **When** the user submits a transaction and then closes the history popup, **Then** the screen does not freeze and stays interactive (#4984).
- [ ] **AC-2** — **Given** the submit/close path, **When** it runs, **Then** per-account work is bounded and does not run synchronously on the main thread (no O(accounts) blocking on the UI path).
- [ ] **AC-3** — **Given** the history popup closes, **When** subscriptions/listeners are torn down, **Then** teardown is incremental/deferred so it cannot block the close animation or the next interaction.
- [ ] **AC-4** — **Given** an interrupted or rapid close-reopen of the popup, **When** it happens under many accounts, **Then** no duplicated or leaked per-account work accumulates (no thundering re-subscribe).

## Dev notes

### Architecture constraints

- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — the submit/close path rides the lightweight read path; this story keeps the *work* bounded as accounts multiply, complementing the *memory* bound in [US-20.3](US-20.3-read-path-memory-budget.md).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-20.2](US-20.2-api-call-optimization.md) — reuses its **request-count budget harness** for the many-account submit benchmark, and relies on its request-dedup so popup close-reopen does not re-fire redundant per-account fetches. (Hence `depends_on: [US-20.2]`.)
- Sibling [US-20.3](US-20.3-read-path-memory-budget.md) — US-20.3 owns NFR-11 (memory by chain count); this story owns main-thread work by account count, which no NFR covers. US-20.3 bounds memory by chain count, this story bounds main-thread work by account count. Coordinate fixtures.

### Performance budget

- Submit + history-popup close never blocks the main thread; per-account work is bounded (not O(accounts) on the UI path).
- Measured by the many-account submit/close benchmark (TASK-20.4.4).
- Story PR description must explicitly confirm this budget is met.

### What we explicitly did NOT do

- No change to transaction-submit *behaviour* or history *content* — those stay with the send/history surface ([EPIC-8](../epics/EPIC-8.md)); this story only removes the main-thread blocking under many accounts.
- No read-path memory work — the ≤72 MB budget is [US-20.3](US-20.3-read-path-memory-budget.md), kept separate so each is a single concern.

### Points justification

5 pts — §3a-bis "production deliverable / one integration": profile and unblock a
single concrete main-thread hotspot (#4984) on the submit/close path plus a
standing benchmark. Priority sits at P1/P2 — it is a real freeze for power users
(P1) but narrower than the lifecycle/request programs; sized as a focused 5, below
the multi-system 8.
