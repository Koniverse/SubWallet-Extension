## Goal

A user can bridge assets via Hyperbridge directly in the wallet — seeing the fee
and ETA before signing — so that the chains Hyperbridge connects become reachable
cross-chain destinations without an external tool.

## Background

This is a **planned** (forward) external bridge provider. Hyperbridge is a
cross-chain interoperability protocol; integrating it adds one more provider
handler behind the bridge-hop interface from
[US-13.1](US-13.1-xcm-parachain-transfers.md) and the provider-handler fixture
from [US-13.2](US-13.2-snowbridge-eth-asset-hub.md). Bridge construction follows
the delegate posture ([AD-18](../../ARCHITECTURE.md#architecture-decisions)) and
the provider is reached through the SubWallet proxy
([AD-19](../../ARCHITECTURE.md#architecture-decisions), NFR-16).

Materializes [FR-136](../../PRD.md#functional-requirements), which is `📋 planned` in the PRD — this story
is **forward / planned**, not retroactive. `commit` / `version_shipped` are set
when the work lands. Tracked by
[#4605](https://github.com/Koniverse/SubWallet-Extension/issues/4605) — Integrate
Hyperbridge in-app, and
[#4940](https://github.com/Koniverse/SubWallet-Extension/issues/4940) — [Feature]
Integrate Hyperbridge on SubWallet.

## Acceptance criteria

- [ ] **AC-1** — **Given** a Hyperbridge-supported asset and route, **When** the
  user requests a bridge, **Then** the wallet shows the fee, estimated destination
  amount, and ETA before any signature.
- [ ] **AC-2** — **Given** a confirmed Hyperbridge transfer, **When** the user
  signs, **Then** the source-chain transaction is submitted and tracked, and the
  destination balance updates on settlement.
- [ ] **AC-3** — **Given** a Hyperbridge transfer in transit (or that stalls),
  **When** the user returns, **Then** it is shown as in-transit with recovery
  guidance — funds are never reported as lost.
- [ ] **AC-4** — **Given** the Hyperbridge provider, **When** the wallet calls it,
  **Then** the call goes through the SubWallet proxy and no provider key is in the
  bundle (AD-19 / NFR-16).

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction delegated; the Hyperbridge handler consumes the provider.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — provider proxied; no key in the bundle (NFR-16).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) (bridge-hop interface) and [US-13.2](US-13.2-snowbridge-eth-asset-hub.md) (provider-handler fixture).

### What we explicitly did NOT do

- Forward story — not yet shipped. No multi-hop routing via Hyperbridge here; that, if added, composes in EPIC-11. This story owns the Hyperbridge hop only.

### Points justification

5 pts — §3a-bis "1 external system integration": one external bridge provider
(Hyperbridge) behind the existing interface, including its in-transit recovery
state and key-proxy wiring. Single-provider integration → 5.
