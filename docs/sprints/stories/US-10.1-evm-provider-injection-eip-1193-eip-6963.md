## Goal

An EVM dApp (Uniswap, OpenSea, any MetaMask-targeting site) sees SubWallet as a
standard injected wallet: `window.ethereum` speaks EIP-1193, and the wallet
announces itself through EIP-6963 multi-provider discovery so it coexists with
other extensions in the picker. The user can connect and act on EVM dApps without
ever leaving for MetaMask.

## Status

> **✅ done — shipped in 1.1.25.** All acceptance criteria are ticked and the 13 rows below are
> settled: 12 delivered, 1 closed without shipping. The multi-wallet coexistence work this surface

## Background

EVM dApps expect an injected provider object at `window.ethereum` implementing the
EIP-1193 `request({ method, params })` interface plus the legacy events
(`accountsChanged`, `chainChanged`, `connect`, `disconnect`). SubWallet injects a
MetaMask-compatible shim from `@subwallet/extension-compat-metamask` via the
content script into every tab. EIP-6963 adds *multi-provider discovery*: instead of
a single global that wallets fight over, each wallet dispatches an
`eip6963:announceProvider` event so the dApp can show a picker — this is what lets
SubWallet and MetaMask coexist on the same page.

Requests that need user consent (`eth_requestAccounts`, `personal_sign`,
`eth_sendTransaction`, `wallet_switchEthereumChain`, …) are routed to the EVM
request handler in `RequestService` ([AD-21](../../ARCHITECTURE.md#architecture-decisions)),
which enqueues them behind the shared `PopupHandler`. Account access is gated by the
per-origin authorization store (US-10.7): an unauthorized origin sees the provider
but its privileged calls are rejected until approved.

Materializes [FR-94](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** any web page, **When** it loads, **Then** `window.ethereum`
  is present, reports SubWallet via `isSubWallet`/EIP-6963 metadata, and responds to
  EIP-1193 `request(...)` for read methods (`eth_chainId`, `eth_accounts`).
- [x] **AC-2** — **Given** a dApp calling `eth_requestAccounts` from an
  unauthorized origin, **When** the call is made, **Then** it is routed to the EVM
  request handler and surfaced in the approval popup; only on approval are the
  selected accounts returned.
- [x] **AC-3** — **Given** multiple injected EVM wallets on the same page, **When**
  the dApp listens for `eip6963:requestProvider`, **Then** SubWallet dispatches an
  `eip6963:announceProvider` event with its info + provider, so it appears in the
  dApp's wallet picker alongside others.
- [x] **AC-4** — **Given** the active account or network changes in the wallet,
  **When** the change occurs, **Then** the provider emits `accountsChanged` /
  `chainChanged` to every connected origin.
- [x] **AC-5** — **Given** a privileged request from an unauthorized origin or one
  the user rejects, **When** it resolves, **Then** the provider returns the standard
  EIP-1193 error (4001 user-rejected / 4100 unauthorized) and no account or signature
  leaks.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — privileged EVM requests route to the dedicated EVM handler behind the shared `PopupHandler`; no bespoke approval popup.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — the inject script holds no key bytes; signing happens only in the background keyring.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-10.7](US-10.7-dapp-authorization-ui-per-origin.md) — uses the per-origin authorization store to gate `eth_requestAccounts`.
- Builds on EPIC-2 US-2.7 — enqueues through `RequestService` / `PopupHandler` rather than building its own queue.
- Sibling [US-10.10](US-10.10-multi-wallet-coexistence-hardening.md) — EIP-6963 coexistence is hardened there against other injected wallets.

### Sizing

Sized 5: one external standard surface (EVM provider injection for one ecosystem)
— EIP-1193 + EIP-6963 + the event model + error mapping, all coexisting with other
extensions on arbitrary pages.
