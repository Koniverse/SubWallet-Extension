## Goal

A multisig user gets the Phase-2 quality-of-life layer: activated multisig accounts
are **auto-detected** via an on-chain scan (no manual re-entry of signatories), and
pending-transaction detail and history are **enriched via an indexer** — call data,
confirmations, full history — so that users can audit and manage their multisig
without piecing state together by hand. This builds on the indexer-free core delivered
by [US-18.1](US-18.1-multisig-account-creation-and-management.md) and
[US-18.2](US-18.2-pending-transaction-detection-and-approval.md).

## Background

This is the **Phase-2 enrichment** pillar of [EPIC-18](../epics/EPIC-18.md) and the one
forward (not-yet-shipped) story in the epic. Phase 1 deliberately shipped without an
indexer: creation is off-chain (US-18.1) and pending-tx detection reads on-chain pallet
state directly (US-18.2). That keeps the spend path infrastructure-free, but it leaves
two gaps this story closes:

1. **Auto-detection of activated multisig accounts** — once a multisig has executed at
   least once it is "activated" on-chain; an on-chain scan can surface it directly so a
   user does not have to re-enter the full signatory set + threshold to rediscover an
   account they already use.
2. **Indexer-enriched history & pending-tx detail** — richer transaction history and
   pending-tx detail (call data, confirmation counts) come from an indexer rather than
   from raw pallet-state reads, via the backend Services SDK
   ([AD-24](../../ARCHITECTURE.md#architecture-decisions)).

This story is the **cluster that absorbs** the multisig Phase-2 work tracked in
[#4839](https://github.com/Koniverse/SubWallet-Extension/issues/4839) and
[#4845](https://github.com/Koniverse/SubWallet-Extension/issues/4845)
(auto-detection + indexer history + Phase-2 optimization) — sized to that scope, the
same way [US-11.8](US-11.8-cross-chain-swap-routing.md) and
[US-4.21](US-4.21-asset-hub-migration-hardening.md) anchor a cluster
of issues to one story. The core approve/execute path stays indexer-free
(AD-17, owned by US-18.2); the indexer dependency (AD-24) is confined to this story's
*enrichment*, never the spend gate.

Materializes [FR-151](../../PRD.md#functional-requirements). **Forward / planned (Phase 2)** — not yet shipped.

## Acceptance criteria

- [ ] **AC-1** — **Given** a multisig account that has been activated on-chain (has
  executed at least once), **When** the wallet runs the on-chain scan, **Then** the
  account is auto-detected and surfaced without the user re-entering its signatory set +
  threshold.
- [ ] **AC-2** — **Given** an auto-detection scan finds no activated multisig for the
  user's signatory membership, **When** it completes, **Then** it returns cleanly with
  no spurious/phantom accounts added.
- [ ] **AC-3** — **Given** a multisig with prior and pending transactions, **When** the
  user opens history, **Then** indexer-enriched history and pending-tx detail (call
  data, confirmation counts) are shown via the Services SDK (AD-24).
- [ ] **AC-4** — **Given** the indexer is unavailable or returns stale data, **When**
  the user views the multisig, **Then** the **core** pending-tx detection and
  approve/execute path (US-18.2, on-chain) still works — the indexer enriches but is
  **never** a hard dependency of the spend flow.
- [ ] **AC-5** — **Given** the Phase-2 optimization scope of #4839 / #4845, **When** the
  regression suite runs, **Then** the issues' reported symptoms no longer reproduce.

## Dev notes

### Architecture constraints

- [AD-17](../../ARCHITECTURE.md#architecture-decisions) — auto-detection is an on-chain
  scan of activated multisig accounts; the core approve/execute path stays indexer-free.
- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — indexer-enriched history /
  pending-tx detail flows through the backend Services SDK; this is an **enrichment**
  layer, not a dependency of the Phase-1 spend gate. This story introduces no new AD
  entries.

### Cross-story dependencies

- Builds on [US-18.1](US-18.1-multisig-account-creation-and-management.md) — auto-detection
  rediscovers accounts whose deterministic derivation that story defines.
- Builds on [US-18.2](US-18.2-pending-transaction-detection-and-approval.md) — enriches
  the on-chain pending-tx detection that story owns; the spend gate stays in US-18.2.
- Relates to the data-aggregation epics that own the Services SDK / indexer path
  (AD-24) — this story consumes it, it does not re-implement it.

### What we explicitly did NOT do

- No moving the spend gate onto the indexer — the threshold/approve/execute path stays
  on-chain (US-18.2); the indexer is enrichment only (AC-4).
- No expansion beyond Substrate multisig — Phase 2 deepens the existing model, it does
  not add new chains.

### Dev notes — points

5 pts — a forward Phase-2 cluster combining an **external-system integration** (the
indexer / Services SDK enrichment path, AD-24) with an on-chain auto-detection scan and
graceful-degradation handling, plus regression coverage for the #4839 / #4845 cluster.
Per [SKILL §3a-bis](../../../.agents/skills/koni-docs/SKILL.md), one external-system
integration sits at 5; it is not multi-system (the spend path is reused from US-18.2,
not rebuilt), so it does not reach 8.
