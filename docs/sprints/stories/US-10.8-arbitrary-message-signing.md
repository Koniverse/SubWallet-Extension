## Goal

A dApp can ask the user to sign an arbitrary message — to prove address ownership,
authenticate a session, or accept terms — in every ecosystem SubWallet supports:
EVM `personal_sign` / `eth_signTypedData`, Polkadot `signMessage` / `signRaw`,
Cardano `signData`, Bitcoin `signMessage`, and TON sign. Message signing is how
most dApps log a user in, so it must work consistently across chains.

## Background

Message signing is distinct from transaction signing: it produces a signature over
bytes (or typed structured data) without moving funds, and is the basis of
"Sign-In With Ethereum"-style auth, gasless ownership proofs, and off-chain
attestations. Each ecosystem has its own method and payload shape — EVM
`personal_sign` (raw) and `eth_signTypedData` (EIP-712 structured), Polkadot
`signRaw` (already surfaced by the inject `signer` in US-10.2), Cardano CIP-30
`signData`, Bitcoin `signMessage`, and TON's sign request.

Each message-sign request routes to its ecosystem's request handler in
`RequestService` ([AD-21](../../ARCHITECTURE.md#architecture-decisions)) behind the
shared `PopupHandler`, which renders the message for the user to inspect before
approving; signing runs in the background keyring (AD-04). EIP-712 typed data must
be displayed in a human-readable, structured form so the user is not approving an
opaque blob. Authorization is required: only an approved origin acting on a granted
account may request a signature (US-10.7).

Materializes [FR-101](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [ ] **AC-1** — **Given** an authorized EVM dApp, **When** it calls `personal_sign`
  or `eth_signTypedData`, **Then** the message (raw text, or EIP-712 structured
  fields) is rendered in the approval popup and, on approval, a valid signature is
  returned.
- [ ] **AC-2** — **Given** an authorized Substrate dApp, **When** it calls
  `signer.signRaw`, **Then** the message is surfaced for approval and signed in the
  background keyring.
- [ ] **AC-3** — **Given** an authorized Cardano / Bitcoin / TON dApp, **When** it
  requests a message signature (CIP-30 `signData` / Bitcoin `signMessage` / TON
  sign), **Then** each routes to its ecosystem handler, is surfaced for approval, and
  returns a valid signature in that ecosystem's format.
- [ ] **AC-4** — **Given** EIP-712 typed data, **When** it is shown for approval,
  **Then** the domain and message fields are presented structurally (not as an opaque
  hex blob) so the user can inspect what they sign.
- [ ] **AC-5** — **Given** a message-sign request from an unauthorized origin or one
  the user rejects, **When** it resolves, **Then** it rejects with the ecosystem's
  standard error (e.g. EVM 4001) and no signature is produced.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — each message-sign method routes to its ecosystem's request handler behind the shared `PopupHandler`.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — signatures are produced in the background keyring; the popup/inject bridge never holds key bytes.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-10.7](US-10.7-dapp-authorization-ui-per-origin.md) — only approved origins on granted accounts may request a signature.
- Builds on the per-ecosystem connectors [US-10.1](US-10.1-evm-provider-injection-eip-1193-eip-6963.md), [US-10.2](US-10.2-substrate-inject-api-injectedweb3.md), [US-10.4](US-10.4-cardano-cip-30-connector.md), [US-10.5](US-10.5-bitcoin-dapp-connection-psbt.md), [US-10.6](US-10.6-ton-dapp-connection.md) — each exposes the message-sign method this story wires to its handler.
- Sibling EPIC-2 US-2.7 — enqueues through `RequestService`.

### Sizing

Sized 3: one cross-ecosystem feature riding on connectors and handlers that already
exist — the new work is the per-method message-sign approval surface (notably the
structured EIP-712 render), not a new integration.
