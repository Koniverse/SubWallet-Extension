## Goal

When the asset a user wants to earn on lives on a different chain than where they
hold it, the wallet inserts an XCM deposit step ahead of the stake — routing the
funds cross-chain so they land on the earning chain — and tracks it as part of the
earning path, so that a user can stake without manually bridging first.

## Background

Some earning positions require the principal on a specific chain (e.g. stake DOT
that currently sits on Asset Hub, or supply an asset that lives on another
parachain). Rather than make the user perform a separate bridge and then return to
stake, the earning flow composes an **XCM deposit step** as the first hop of the
path. The XCM transfer construction itself is **not** built here — it is delegated
to the ParaSpell-backed XCM engine
([AD-18](../../ARCHITECTURE.md#architecture-decisions)) owned by
[EPIC-13](../epics/EPIC-13.md); this story *invokes* that engine as a deposit
pre-step and threads it through the multi-step path harness from
[US-12.8](US-12.8-earning-path-simulation-and-step-tracking.md), so the deposit is
simulated, tracked and resumable like any other step.

Materializes [FR-122](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [ ] **AC-1** — **Given** an earning target on a chain other than where the funds
  sit, **When** the user opens the stake flow, **Then** the simulated path begins
  with an XCM deposit step (source chain → earning chain) with its fee and
  estimated time shown before any signature.
- [ ] **AC-2** — **Given** a confirmed path with a deposit step, **When** the user
  executes it, **Then** the XCM deposit is submitted and tracked first, and the
  bond/supply step proceeds only after the deposit is confirmed on the earning
  chain.
- [ ] **AC-3** — **Given** the deposit step needs a cross-chain transfer, **When**
  it is constructed, **Then** it reuses the ParaSpell XCM engine (AD-18 / EPIC-13),
  not a bespoke per-route transfer built inside earning.
- [ ] **AC-4** — **Given** a path that completes the deposit but stalls before the
  stake, **When** the user returns, **Then** the deposited funds are visible on the
  earning chain and the path is resumable from the stake step — funds are never
  reported as lost mid-route.

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — XCM transfer construction is delegated to ParaSpell; this story consumes the bridge engine, it does not reimplement XCM `MultiLocation` construction.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — the stake step still goes through the handler tree; XCM only routes the principal to the right chain first.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.8](US-12.8-earning-path-simulation-and-step-tracking.md) — uses the multi-step path harness (simulate / track / resume) for the deposit + stake sequence.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — the stake step uses the handler tree.
- Sibling [EPIC-13](../epics/EPIC-13.md) — owns the ParaSpell XCM engine this deposit step consumes; coordinate on the engine interface.

### Points justification

5 pts — §3a-bis "1 external-system integration": wiring the earning path to the
ParaSpell-backed XCM engine as a confirmed-before-stake deposit step, with
cross-chain confirmation gating and mid-route resumability. The XCM construction
is consumed (not built), so this is one bounded cross-engine integration — sizes
at 5.
