## Goal

New chains and updated token metadata reach users **without shipping a new
extension version**, so the wallet can track the fast-moving multi-chain
ecosystem in near-real-time instead of on the extension-store release cadence.

## Background

Hard-coding chain specs and token lists into the bundle would mean every new
parachain or token needs an extension release and store review. Instead, chain
list and token/asset metadata are served from the SubWallet static-data cache
and CDN proxy ([AD-25](../../ARCHITECTURE.md#architecture-decisions)), with the
heavy slow-changing data regenerated by a headless `web-runner-cron` build that
runs the real client and commits refreshed data
([AD-23](../../ARCHITECTURE.md#architecture-decisions)). The client consumes it
via `fetchStaticData` with a bundled JSON fallback when the proxy is
unreachable.

This is the invariant the whole epic leans on: a new chain or token is a
data-cache update, never a code change. It is what lets every ecosystem story
add networks online.

Materializes [FR-34](../../PRD.md#epic-4--chain-management). **Retroactive** —
already shipped.

## Acceptance criteria

- [x] **AC-2** — **Given** token metadata changes upstream, **When** the asset

  client loads, **Then** it falls back to the bundled chain-list/token JSON and
  still functions (no blank registry).

## Dev notes

### Architecture constraints

- [AD-23](../../ARCHITECTURE.md#architecture-decisions) — static data regenerated by the headless web-runner cron, not reimplemented server-side.
- [AD-25](../../ARCHITECTURE.md#architecture-decisions) — chain-list / token metadata fronted by `static-data` / `static-cache` proxies with JSON fallback.
- This story introduces no new AD entries.

### Cross-story dependencies

- Required by every ecosystem story ([US-4.5](US-4.5-evm-network-support.md) onward) — they register chains/tokens through this online-update mechanism rather than the bundle.

### Performance budget

- Chain-list refresh served from cache / CDN proxy; never blocks first paint, never live-polls RPCs (AD-23/AD-25).
- Bundled-JSON fallback guarantees a usable registry offline.

### Dev notes — points

3 pts — a registry/data feature (cache-proxy consumption + fallback), one
internal integration, no external system to build, per SKILL §3a-bis.
