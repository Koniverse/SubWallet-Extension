## Goal

Find out what the background's read path actually costs in memory, and put that number
in the docs — so the decision to rewrite the substrate read path, or to leave it alone,
rests on a measurement instead of a claim nobody has checked since 2022.

## Background

**Rescoped 2026-07-13.** This story used to say *"every balance read must ride the
lightweight WsProvider path so resident memory stays ≤72 MB"*, and its job was to guard
that invariant against regression. An audit found the invariant **does not exist**:

- the substrate read path awaits the **full `ApiPromise`** (`substrateApiMap[slug].isReady`)
  and reads off it — `substrateApi.api.query.balances.locks.multi(…)`, `.holds.multi`,
  `.freezes.multi`, `system.account.multi` — and one read site even reaches
  `substrateApi.api.tx.nominationPools`;
- `SubstrateApi`'s constructor builds `new ApiPromise` **eagerly for every enabled
  chain** and never tears it down after an extrinsic;
- the string `lightweight` has **zero hits** in `packages/*/src`.

Same shape at **v0.4.1** (2022), at **v1.1.64** — the release
[AD-07](../../ARCHITECTURE.md#architecture-decisions) claimed it shipped in — and at
**v1.3.83**. It was never built, so there is nothing to regress. AD-07's citation was
wrong too: PR #3024 merged `koni/dev/3020`, and 1.1.64 shipped only *"Update chain-list
(#3020)"*.

**And the premise has moved.** The 137 MB / 264 MB / 72 MB figures were measured in 2022
on the **MV2 always-on background page**. Under MV3 the worker sleeps after 60 s idle
and `sleep()` stops every chain API ([AD-08](../../ARCHITECTURE.md#architecture-decisions);
shipped 1.2.1, see [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md)), so
memory no longer accumulates for the life of the browser session. **Nobody has
re-measured under MV3.**

So this story no longer asks anyone to refactor. It asks for a **number**, then a
recorded decision. Rewriting every balance read across 200+ chains is the
highest-blast-radius change available in this codebase — a wrong balance costs trust,
not milliseconds — and it must be justified by data. If the measurement shows the budget
holds, the honest outcome is to rewrite AD-07 / NFR-11 to describe what the code does,
and close this story. See [CONTEXT D95](../../CONTEXT.md) (revision of D2).

## Acceptance criteria

- [ ] **AC-1** — **Given** a fixture with ≥20 chains enabled and balances to fetch,
  **When** the memory probe runs against the background, **Then** it reports peak and
  steady-state heap for the read path as a reproducible **number**.
- [ ] **AC-2** — **Given** that fixture, **When** memory is sampled with the MV3 worker
  awake and again after `sleep()`, **Then** both figures are reported — so the budget is
  stated for the state the user actually pays for, not for a hypothetical always-on
  background.
- [ ] **AC-3** — **Given** the measured baseline, **When** AD-07 and NFR-11 are read,
  **Then** they state the **measured** figure and the **actual** mechanism (a full
  `ApiPromise` per enabled chain) — no claim the code does not support.
- [ ] **AC-4** — **Given** the baseline against NFR-11's ≤72 MB target, **When** the gap
  is judged, **Then** a decision is appended to CONTEXT.md: either *the budget holds → no
  refactor, NFR-11 confirmed*, or *the budget is blown by X MB → open a scoped read-path
  refactor story carrying that number in its Goal*.
- [ ] **AC-5** — **Given** the probe exists, **When** it runs in CI against the
  ≥20-chain fixture, **Then** a change that pushes heap past the recorded baseline fails
  the build — the guard the old AC-4 wanted, but anchored to a real number.

## Dev notes

### Architecture constraints

- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — ⚠️ decided 2022, **never implemented**. This story owns that gap: measure, then either fix the code or fix the row.
- [AD-08](../../ARCHITECTURE.md#architecture-decisions) — the MV3 service worker sleeps after 60 s idle and `sleep()` stops every chain API. That is *why* the 2022 numbers no longer transfer — measure both states.
- This story does NOT introduce new AD entries. Its output may **revise** AD-07.

### Cross-story dependencies

- Related to [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) — the MV3 sleep/wake behaviour that changed this problem's premise (its P1 shipped in 1.3.42).
- Related to [US-20.4](US-20.4-many-account-submit-performance.md) — the unmerged `koni/dev/issue-4984` branch batches per-account EVM balance reads through multicall3. If it lands, re-measure before judging the budget.

### Performance budget

- NFR-11's ≤72 MB is a **target, not a verified property**. Treat it as the hypothesis under test, not as a constraint the code already satisfies.
- The number to beat is whatever TASK-20.3.1 measures. Until then, no PR should claim "the budget is met" — there is no budget in evidence.

### What we explicitly did NOT do

- No read-path refactor. That was the old scope, and it was scheduled on the strength of a doc claim that turned out to be false.
- No change to *what* a balance read returns — correctness stays with the balance engine ([EPIC-2](../epics/EPIC-2.md)) and the balance surface ([EPIC-7](../epics/EPIC-7.md)).
- No many-account main-thread work — that is [US-20.4](US-20.4-many-account-submit-performance.md).

### Points justification

3 pts — down from 5. A measurement harness plus a recorded decision: no product surface
changes, no engine rewrite. The refactor that used to live inside this story is
explicitly no longer in it, and would be sized on its own once there is a number.
