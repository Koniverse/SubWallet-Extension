## Goal

Keep the balance/read path memory-bounded no matter how many chains a user has
enabled. Every balance and token read must ride the lightweight WsProvider path so
the background's resident memory stays at or below 72 MB, instead of ballooning the
way a full `@polkadot/api` ApiPromise per chain would. When this holds, the read
epics (balance, NFT, earning) can fan their reads across 200+ networks without the
worker getting evicted for memory pressure or dragging the host down.

## Background

This is the memory facet of the umbrella performance program
([#4197](https://github.com/Koniverse/SubWallet-Extension/issues/4197), "Evaluate
and update the performance of the extension"). The architecture decision that
bounds it is [AD-07](../../ARCHITECTURE.md#architecture-decisions): use a custom
lightweight WsProvider connector for balance/token *reads*, and instantiate the
full `@polkadot/api` ApiPromise only when an *extrinsic* must be constructed.
The measured stakes are explicit in AD-07 and NFR-11 — a full ApiPromise consumed
~137 MB for 4 chains and ~264 MB for 20 chains, whereas the WsProvider-only mode
needs only ~72 MB regardless of chain count. The risk this story guards against is
regression: any read path that quietly forces a full ApiPromise (for convenience,
or via a shared helper) blows the budget the moment a user enables many chains.

This story owns no FR (cf. the FR-less defending-story pattern of
[US-12.15](US-12.15-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md)); it defends NFR-11
(memory performance: ≤72 MB read path) on the cross-cutting read path, and rides
the MV3 substrate (AD-08) whose eviction pressure makes the budget load-bearing.

## Acceptance criteria

- [ ] **AC-1** — **Given** a balance/token read across many enabled chains, **When** it executes, **Then** it uses the lightweight WsProvider connector and never instantiates a full `@polkadot/api` ApiPromise for the read (AD-07).
- [ ] **AC-2** — **Given** the read path under a many-chain fixture (≥20 chains), **When** balances are aggregated, **Then** background resident memory stays ≤ 72 MB regardless of chain count (NFR-11), asserted by a memory-budget probe.
- [ ] **AC-3** — **Given** an extrinsic must be constructed, **When** the full ApiPromise is instantiated, **Then** it is created lazily and torn down/scoped so it does not persist on the read path after the extrinsic completes (AD-07).
- [ ] **AC-4** — **Given** a new or refactored read site, **When** the code is built, **Then** a guard (test/lint) flags any read-path call that would force a full ApiPromise, preventing silent budget regressions.

## Dev notes

### Architecture constraints

- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — lightweight WsProvider for reads; full ApiPromise deferred to extrinsic construction. This story enforces the budget AD-07 promises (~72 MB vs ~264 MB across 20 chains).
- [AD-08](../../ARCHITECTURE.md#architecture-decisions) — the MV3 service worker is evicted under memory pressure, which is why the read-path budget is load-bearing, not cosmetic.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-20.1](US-20.1-core-structure-and-lifecycle-refactor.md) — a partial wake that starts only the needed chains is what keeps the read path from instantiating more than it must.
- Sibling [US-20.4](US-20.4-many-account-submit-performance.md) — both defend NFR-11; US-20.3 bounds memory by *chain* count, US-20.4 bounds main-thread work by *account* count. Coordinate the read-path fixtures.

### Performance budget

- Read-path memory: ≤ 72 MB regardless of chain count (NFR-11, AD-07).
- A full `@polkadot/api` ApiPromise is never instantiated for a read — only for extrinsic construction.
- Measured by the memory-budget probe (TASK-20.3.3).
- Story PR description must explicitly confirm this budget is met.

### What we explicitly did NOT do

- No change to *what* a balance read returns — correctness/aggregation stays with the balance engine ([EPIC-2](EPIC-2.md)) and the balance surface ([EPIC-7](EPIC-7.md)); this story only bounds the memory of fetching it.
- No many-account submit/close work — that main-thread concern is [US-20.4](US-20.4-many-account-submit-performance.md), kept separate so each is a single concern.

### Points justification

5 pts — §3a-bis "production deliverable / one integration": auditing read sites
onto the WsProvider path, scoping ApiPromise to extrinsics, plus a memory probe
and a regression guard. Bounded single-path optimization with a clear measurable
target (≤72 MB) — a 5, below the multi-system 8.
