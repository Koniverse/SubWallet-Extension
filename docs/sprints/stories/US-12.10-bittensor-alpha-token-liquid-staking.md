## Goal

A user can liquid-stake Bittensor alpha tokens — staking subnet alpha while
receiving a liquid derivative they can use elsewhere — so that subnet stakers keep
their position liquid instead of locking alpha for the full unstake period.

## Background

This extends the dTAO subnet staking foundation
([US-12.6](US-12.6-bittensor-dtao-subnet-staking.md)) with a *liquid* variant: a
protocol mints a liquid derivative against staked alpha, combining the LST pattern
(FR-117) with the Bittensor native model. Per
[AD-15](../../ARCHITECTURE.md#architecture-decisions) all Bittensor staking runs
the native Subtensor path; the engine would add a liquid-alpha handler on the
special-pool / Tao branch of the tree
([AD-22](../../ARCHITECTURE.md#architecture-decisions),
[US-2.3](US-2.3-earningservice-pool-handler-engine.md)).

Materializes [FR-123](../../PRD.md#functional-requirements). 📋 **Planned / forward** — not yet shipped;
this story scopes the planned capability. Tracked by
[#4902](https://github.com/Koniverse/SubWallet-Extension/issues/4902) — Support
Bittensor Liquid staking, and
[#4878](https://github.com/Koniverse/SubWallet-Extension/issues/4878) —
Integration: Liquid Stake Alpha Bridge.

## Acceptance criteria

- [ ] **AC-1** — **Given** a subnet alpha holding, **When** the user liquid-stakes
  an amount, **Then** the mint submits via the native Subtensor path and the
  user's liquid-alpha derivative balance and position (exchange rate + APY) appear
  in the earning list.
- [ ] **AC-2** — **Given** a liquid-alpha position, **When** the user redeems back
  to alpha, **Then** the redeem submits and the position / claimable state updates
  from the subject.
- [ ] **AC-3** — **Given** any liquid-alpha action, **When** it is constructed,
  **Then** it uses the native Subtensor pallet path (AD-15), never the
  Subtensor-EVM.

## Dev notes

### Architecture constraints

- [AD-15](../../ARCHITECTURE.md#architecture-decisions) — native Subtensor path for all Bittensor staking, including the liquid variant.
- [AD-22](../../ARCHITECTURE.md#architecture-decisions) — liquid-alpha staking is a handler subclass on the special-pool / Tao branch; the screen consumes its subjects.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-12.6](US-12.6-bittensor-dtao-subnet-staking.md) — extends the dTAO subnet-staking foundation with a liquid variant.
- Builds on [US-2.3](US-2.3-earningservice-pool-handler-engine.md) — consumes the (planned) liquid-alpha handler subjects.

### What we explicitly did NOT do

- No alpha-token *bridge* — moving alpha across chains is owned by [EPIC-13](../epics/EPIC-13.md) (FR-133). This story only liquid-stakes alpha on its native path.

### Points justification

5 pts — §3a-bis "1 external-system integration": a new liquid-staking protocol
integration for subnet alpha on the native Subtensor path (mint/redeem,
exchange-rate estimate), distinct from the base dTAO staking flow. One external
integration sizes at 5.
