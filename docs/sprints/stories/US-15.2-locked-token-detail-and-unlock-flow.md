## Goal

After voting, a user can see exactly which tokens a conviction vote has locked —
the amount, the conviction multiplier, and the block at which the lock expires —
and, once the lock period has elapsed, release those tokens back to transferable
with a single unlock action. This closes the conviction-voting loop opened by
[US-15.1](US-15.1-opengov-referenda-and-conviction-voting.md): the user is never
left wondering why their balance is locked or how to get it back.

## Background

A conviction vote locks the staked amount for a multiple of the enactment period;
the stronger the conviction, the longer the lock. Without a clear view, users see
a reduced transferable balance with no explanation and no obvious release path.

This story shows the **vote-lock detail** — amount, conviction, unlock-at block /
estimated time — and provides the **unlock flow** (`convictionVoting.unlock` plus
any required `removeVote` for an already-decided referendum) once the lock has
expired. The lock figure shown is the same `locked` reservation owned by
[EPIC-7](../epics/EPIC-7.md)'s transferable/locked balance model — this story
*reads* that figure, it does not re-derive a second balance. The lock-status read
rides the lightweight WsProvider path
([AD-07](../../ARCHITECTURE.md#architecture-decisions)); the unlock extrinsic is
constructed and signed in the background and surfaced via the typed `pri()` bus
([AD-03](../../ARCHITECTURE.md#architecture-decisions)).

Materializes [FR-140](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped; `commit` / `version_shipped` are backfilled during version
reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** an account with one or more conviction-vote locks,
  **When** the user opens the locked-token detail view, **Then** each lock is
  listed with its amount, conviction multiplier, and unlock-at block / estimated
  unlock time.
- [x] **AC-2** — **Given** a lock whose period has elapsed, **When** the user
  triggers unlock, **Then** the unlock extrinsic (plus `removeVote` if required)
  is built and signed in the background and the released amount returns to the
  transferable balance.
- [x] **AC-3** — **Given** a lock whose period has NOT yet elapsed, **When** the
  user views it, **Then** the unlock action is disabled and the remaining
  lock time is shown, so no unlock extrinsic can be submitted prematurely.
- [x] **AC-4** — **Given** the locked figure displayed here, **When** it is read,
  **Then** it is the same `locked` reservation from the EPIC-7 balance model — no
  separate balance computation is performed in the governance code.

## Dev notes

### Architecture constraints

- [AD-07](../../ARCHITECTURE.md#architecture-decisions) — the lock-status read rides the lightweight WsProvider path; the full ApiPromise is instantiated only to construct the unlock extrinsic.
- [AD-03](../../ARCHITECTURE.md#architecture-decisions) — the unlock (+ removeVote) extrinsic is signed in the background and surfaced via `pri()`; no key material in the governance UI.
- This story does NOT introduce new AD entries. The conviction-lock figure is read from the [EPIC-7](../epics/EPIC-7.md) balance model (FR-69), not re-derived here.

### Cross-story dependencies

- Builds on [US-15.1](US-15.1-opengov-referenda-and-conviction-voting.md) — reads and releases the conviction lock the vote in US-15.1 creates.
- Consumes the `locked` reservation from [EPIC-7](../epics/EPIC-7.md)'s transferable/locked split ([US-7.2](US-7.2-transferable-vs-locked-balance-calculation.md)) — the home-screen display of that figure is EPIC-7's, not this story's.

### What we explicitly did NOT do

- No staking/nomination unlock — that is a *different* lock owned by [EPIC-12](../epics/EPIC-12.md) (earning). This view covers governance vote locks only.
- No home-screen rendering of the locked amount — owned by [EPIC-7](../epics/EPIC-7.md); this story renders the per-lock governance detail view.
