## Goal

A multisig signatory can see the transactions pending against their multisig and act
on them — the initiator initiates and approves, co-signers approve or reject — and the
transaction executes **only once the threshold number of approvals is collected**, all
detected **on-chain without an indexer**. This delivers the actual M-of-N spend flow
on top of the account created in
[US-18.1](US-18.1-multisig-account-creation-and-management.md).

## Background

This is the **approval** pillar of [EPIC-18](../epics/EPIC-18.md) and the heart of the
M-of-N model. Native Substrate multisig works through the pallet extrinsics `as_multi`
/ `approveAsMulti` ([AD-17](../../ARCHITECTURE.md#architecture-decisions)): the
**initiator** kicks off a call (initiate + first approval), and the **co-signers**
each add their approval until the threshold is met, at which point the final approval
executes the call. The roles are genuinely different — the initiator carries the call
data and the deposit; co-signers approve against the call hash — so the flow is
**role-differentiated**, not a single uniform "approve" button.

The defining constraint is that pending transactions are detected by reading on-chain
pallet state **directly, with no indexer** (AD-17). The wallet scans the multisig's
on-chain pending state to know what is awaiting approval and how many approvals it
already has. This keeps the spend flow infrastructure-free; indexer enrichment
(richer history, call-data detail) is a separate Phase-2 concern
([US-18.3](US-18.3-auto-detection-indexer-history-and-optimization.md)). This story
owns **when** a transaction may execute (the threshold gate); it does not build or
submit the extrinsic itself ([EPIC-8](../epics/EPIC-8.md)) and it plugs the approval
UI into the generic request/approval queue ([EPIC-2](../epics/EPIC-2.md)) rather than
owning that queue.

Materializes [FR-150](../../PRD.md#functional-requirements). **Retroactive** — already shipped.

## Acceptance criteria

- [x] **AC-1** — **Given** a multisig with pending transactions, **When** a signatory
  opens the multisig, **Then** the pending transactions are detected and listed by
  reading **on-chain** pallet state, with **no indexer** dependency (AD-17), each
  showing its current approval count vs the threshold.
- [x] **AC-2** — **Given** the **initiator** role, **When** they initiate a multisig
  call, **Then** the call is submitted via `as_multi` (carrying call data + deposit)
  and counts as the first approval.
- [x] **AC-3** — **Given** a **co-signer** role on a pending tx below threshold, **When**
  they approve, **Then** their approval is recorded via `approveAsMulti` and the
  approval count advances by one; **When** they reject, **Then** the tx is not advanced
  and the rejection is surfaced.
- [x] **AC-4** — **Given** a pending tx whose approvals have just reached the threshold,
  **When** the threshold-meeting approval is submitted, **Then** the call executes; and
  **Given** a tx still below threshold, **Then** it stays pending and **cannot** be
  unilaterally executed by any single signatory.
- [x] **AC-5** — **Given** a signatory who has **already approved** a pending tx, **When**
  they attempt to approve it again, **Then** the duplicate approval does not advance the
  count (one approval per signatory toward the threshold).
- [x] **AC-6** — **Given** a **stale** pending entry (the tx already executed or was
  cancelled on-chain since last scan), **When** the list refreshes from on-chain state,
  **Then** the stale entry is reconciled/removed and is not actionable.

## Dev notes

### Architecture constraints

- [AD-17](../../ARCHITECTURE.md#architecture-decisions) — approval uses the native
  pallet extrinsics `as_multi` / `approveAsMulti`; pending-tx detection reads on-chain
  state directly with **no indexer**. This story introduces no new AD entries.
- The threshold gate is the FR-150 invariant: execution is forbidden until the
  threshold count of distinct approvals exists; this story owns *when* a tx may
  execute, not how the extrinsic is built/submitted ([EPIC-8](../epics/EPIC-8.md)).

### Cross-story dependencies

- Builds on [US-18.1](US-18.1-multisig-account-creation-and-management.md) — approves
  transactions for the multisig account that story creates; consumes its signatory
  set + threshold.
- Plugs into [EPIC-2](../epics/EPIC-2.md) (RequestService) — the approval UI rides the
  generic request/approval queue; this story does not own the queue.
- Required by [US-18.3](US-18.3-auto-detection-indexer-history-and-optimization.md) —
  Phase-2 enrichment adds indexer-backed history/detail on top of this on-chain
  detection.

### What we explicitly did NOT do

- No indexer-backed history or rich call-data/confirmation detail — Phase-1 detection
  is on-chain only; enrichment is deferred to
  [US-18.3](US-18.3-auto-detection-indexer-history-and-optimization.md).
- No multi-chain beyond Polkadot Substrate — Phase-1 multisig is scoped to Substrate
  chains (AD-17).

- [Source: ARCHITECTURE AD-17](../../ARCHITECTURE.md#architecture-decisions) — multisig account model (`as_multi` / `approveAsMulti`, indexer-free detection)
