## Goal

A user can see every website they have connected to the wallet, control which of
their accounts each site may use, and revoke a site's access at any time — the
"Manage website access" surface. This is the consent backbone every connector in
this epic depends on: it decides which origins are trusted with which accounts.

## Background

dApp connection is meaningless without a permission model: a page must not get
account access just by injecting a provider. SubWallet stores authorization
**per dApp origin** — which accounts the origin may see, when access was granted —
and exposes a management UI ("Manage website access") where the user lists
connected sites, edits per-account access, and revokes. Unauthorized origins still
*see* the provider object (so dApps can feature-detect the wallet) but every
privileged request is rejected until the user explicitly approves the origin
([ARCHITECTURE §Security boundaries](../../ARCHITECTURE.md) — "dApp injection scope").

The connect approval itself is enqueued through `RequestService` (the Auth handler,
[AD-21](../../ARCHITECTURE.md#architecture-decisions)) behind the shared
`PopupHandler`; this story owns the persistent authorization store that the Auth
handler writes and the management UI that reads/edits it. Every connector
(US-10.1..10.6, US-10.9) consumes this gate; revoking here must immediately stop
further privileged requests from that origin.

Materializes [FR-100](../../PRD.md#functional-requirements). **Retroactive** — already shipped; `commit` /
`version_shipped` are backfilled during version reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a dApp requesting connection from a new origin, **When**
  the user approves, **Then** a per-origin authorization entry is created recording
  the granted accounts, and the origin's subsequent privileged requests are accepted.
- [x] **AC-2** — **Given** an unauthorized origin, **When** it issues a privileged
  request (account access / sign), **Then** the request is rejected (the provider is
  visible but unauthorized) until the user approves the origin.
- [x] **AC-3** — **Given** the "Manage website access" screen, **When** the user
  opens it, **Then** it lists every connected origin with the accounts each may use.
- [x] **AC-4** — **Given** a connected origin, **When** the user edits its per-account
  access or revokes it, **Then** the authorization store is updated and any further
  privileged request from that origin reflects the new scope immediately (revoked →
  rejected).
- [x] **AC-5** — **Given** an origin granted access to a subset of accounts, **When**
  the dApp reads accounts, **Then** only the granted accounts are returned — never the
  full wallet.

## Dev notes

### Architecture constraints

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — the Auth request handler enqueues connect approvals behind the shared `PopupHandler`; this story owns the persisted authorization the handler reads/writes.
- [ARCHITECTURE §Security boundaries — dApp injection scope](../../ARCHITECTURE.md) — content script injected into all tabs; authorization stored per origin; unauthorized dApps receive rejection until explicit approval.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Required by [US-10.1](US-10.1-evm-provider-injection-eip-1193-eip-6963.md), [US-10.2](US-10.2-substrate-inject-api-injectedweb3.md), [US-10.3](US-10.3-walletconnect-v2-substrate-and-evm.md), [US-10.4](US-10.4-cardano-cip-30-connector.md), [US-10.5](US-10.5-bitcoin-dapp-connection-psbt.md), [US-10.6](US-10.6-ton-dapp-connection.md), [US-10.9](US-10.9-dapp-createtransaction-api-rfc-6213.md) — every connector gates account access on this store.
- Sibling [EPIC-5](../epics/EPIC-5.md) US-5.1 (phishing-site protection) — that decides which origins are *dangerous*; this records which origins are *trusted*. They are distinct surfaces.

### Sizing

Sized 3: a single internal subsystem (per-origin authorization store + management
UI) on the existing Auth request handler — one ecosystem-agnostic consent model, no
external integration.
