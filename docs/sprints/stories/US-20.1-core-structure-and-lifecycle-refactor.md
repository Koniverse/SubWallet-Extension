## Goal

Give the MV3 background a clean, phased core structure so the wallet starts only
as much of itself as a request actually needs and does nothing avoidable while
idle. This single story covers the unified "Improve SubWallet Lifecycle" program
(#4427): a phased Init → Start → StartFull lifecycle plus background-idle, the
removal of deprecated mechanisms (the ZK Asset feature), the refactor of the cron
and subscription machinery into proper services, and a refinement of the
data-processing architecture. When this lands, every feature epic gets a runtime
that wakes fast, sleeps quiet, and processes data through one well-shaped path
instead of ad-hoc cron/subscription wiring.

## Background

The background runs as an MV3 service worker (AD-08), which Chrome evicts after
~5 minutes idle, so the runtime needs explicit wake/sleep phasing rather than the
old MV2 always-on model. The four-state lifecycle is the architecture decision
that answers this ([AD-20](../../ARCHITECTURE.md#architecture-decisions)): an
`ActionHandler` + `connectionMap` drives Init → Start-Partially → Start-Fully →
Sleep, where `pub(…)` wakes the worker partially and `pri(…)`/`mobile(…)` wakes it
fully, so a request that needs only a few chains never starts all of them. This
story materializes the *runtime behaviour* of that decision; the decision and the
MV3 substrate (AD-08) themselves are owned by [EPIC-1](../epics/EPIC-1.md).

This story intentionally covers the whole unified #4427 program rather than
splitting it, because its phases are one coherent refactor of the same core
structure:

- **Lifecycle tracker [#4427](https://github.com/Koniverse/SubWallet-Extension/issues/4427)** ("Improve SubWallet Lifecycle") — the umbrella for the program, indexing the phased sub-issues below.
- **[#4428](https://github.com/Koniverse/SubWallet-Extension/issues/4428) — P1 (shipped → retroactive)** — the first phase of the lifecycle refactor; already closed/shipped, so this story records it retroactively, not as forward work.
- **[#4445](https://github.com/Koniverse/SubWallet-Extension/issues/4445) — P2 (forward)** — the next phase: removing the deprecated ZK Asset feature and refactoring the cron/subscription machinery into services.
- **[#4446](https://github.com/Koniverse/SubWallet-Extension/issues/4446) — P3 (forward)** — refining the data-processing architecture on top of the phased lifecycle and the service-shaped cron/subscription layer.

They stay together as one story because they refactor the same core structure
along one arc — but because P2 (#4445) and P3 (#4446) are still forward work of
non-trivial size, the Dev notes flag that they MAY split into their own stories
when scheduled (see "Split watch"). This story defends NFR-8 (MV3
shutdown/wake state persistence) and NFR-12 (cached cold-start + progressive
refresh) on the background lifecycle specifically; it owns no FR (cf. the
[US-12.13](US-12.13-earning-reward-and-apy-accuracy-hardening.md) /
[US-7.7](US-7.7-balance-cache-invalidation-hardening.md) FR-less defending-story
pattern).

## Acceptance criteria

- [ ] **AC-1** — **Given** the MV3 background worker, **When** a `pub(…)` request arrives, **Then** the worker wakes *partially* (Start) and starts only the services/chains that request needs, and **When** a `pri(…)`/`mobile(…)` request arrives **Then** it wakes *fully* (StartFull) — a partial wake never starts all chains (#4427, AD-20).
- [ ] **AC-2** — **Given** the worker has no open ports, **When** the last connection disconnects and `SLEEP_TIMEOUT` elapses, **Then** the worker sleeps and performs no avoidable polling, chain connection, or API call while backgrounded (NFR-8, AD-20).
- [ ] **AC-3** — **Given** the cold-start path, **When** the popup opens, **Then** last-known cached state is served immediately (≤ 300 ms) and fresh data streams in progressively with visible skeletons, never a blank wait (NFR-12).
- [ ] **AC-4** — **Given** the deprecated ZK Asset feature, **When** the codebase is built after P2 (#4445), **Then** the ZK Asset feature and its dead wiring are removed and no longer referenced.
- [ ] **AC-5** — **Given** the cron and subscription machinery, **When** P2 (#4445) lands, **Then** they run as first-class services (lifecycle-aware start/stop) rather than ad-hoc background timers, and the data-processing architecture is refined onto that path (P3, #4446).
- [ ] **AC-6** — **Given** a service-worker eviction mid-flight, **When** the worker wakes again, **Then** state-persistence restores the last-known state and the lifecycle transitions remain idempotent (no double-start, no lost subscription) (NFR-8, AD-20).

## Dev notes

### Architecture constraints

- [AD-20](../../ARCHITECTURE.md#architecture-decisions) — the four-state lifecycle (Init → Start-Partially → Start-Fully → Sleep) with heartbeat is the contract this story materializes at runtime; transitions must stay idempotent across MV3 eviction.
- [AD-08](../../ARCHITECTURE.md#architecture-decisions) — the MV3 service-worker substrate is the runtime this lifecycle rides on; this story does NOT re-decide the substrate (owned by [EPIC-1](../epics/EPIC-1.md)).
- This story does NOT introduce new AD entries — it realizes AD-20 on top of AD-08.

### Cross-story dependencies

- Sibling [US-20.2](US-20.2-api-call-optimization.md) — both touch the background request/lifecycle layer; the idle-quiet invariant here and the no-redundant-fan-out invariant there must agree on when the worker is allowed to talk to the network.
- Required by [US-20.3](US-20.3-read-path-memory-budget.md) and [US-20.4](US-20.4-many-account-submit-performance.md) — a partial wake that starts only the needed chains is what keeps the read path memory-bounded and the submit path cheap.

### Performance budget

- Idle background: no avoidable polling / chain connect / API call while backgrounded; a partial wake never starts all chains.
- Cold-start first paint: cached state visible ≤ 300 ms on popup open (NFR-12).
- Measured by the lifecycle-phase assertion harness (TASK-20.1.7).
- Story PR description must explicitly confirm these budgets are met.

### What we explicitly did NOT do

- No change to the MV3 substrate decision (AD-08) — owned by [EPIC-1](../epics/EPIC-1.md); this story only phases the lifecycle on top of it.
- No new feature behaviour — the ZK Asset removal is a deletion, the cron/subscription refactor is structural; no user-facing capability is added.

### Split watch

P2 ([#4445](https://github.com/Koniverse/SubWallet-Extension/issues/4445)) and P3
([#4446](https://github.com/Koniverse/SubWallet-Extension/issues/4446)) are still
forward work of non-trivial size. They are kept inside this story for now because
they are one coherent refactor arc with P1, but **when scheduled they MAY split
into their own stories** (e.g. US-20.1a "remove ZK Asset + cron/subscription →
services" and US-20.1b "data-processing architecture refinement"). If they split,
the points here drop accordingly and the new stories carry their own AC/Tasks.

### Points justification

8 pts — §3a-bis "multi-system integration / multi-week, split-if-possible". This
is a multi-phase architectural refactor spanning the lifecycle state machine, a
deprecated-feature removal, a cron/subscription → services refactor, and a
data-processing architecture refinement, across the background runtime. It sits at
the 8/13 boundary: it is intentionally bundled as the unified #4427 program, so it
is sized at 8 with an explicit **13→split watch** — if P2/P3 are scheduled without
splitting, re-estimate to 13 and split per the §3a-bis splitting rule.

## Implementation notes

**Audited 2026-07-13 (US-21.2 follow-up). Phase 1 shipped 13 months ago and the docs
never recorded it; phases 2 and 3 have zero commits.** The story was `backlog`, which
was wrong in both directions.

**Shipped: P1 — `#4428` "Optimize Lifecycle Management - P1", release 1.3.42**
(`f29da4271d`, `b6b6e2a99b`, `3cdd16937c`, author `saltict`). It introduced the
wake-*depth* split that is live at v1.3.83: `ActionHandler._onPortMessage` computes
`requireFullActive` from the message prefix (`pri(` / `mobile(` → `wakeup(true)`,
`pub(` → `wakeup(false)`), and only a full wake runs `_startFull()` (the 11 data
services: cron, subscription, history, price, balance, nft, earning, swap,
notification, openGov, multisig).

**AC-by-AC, against the v1.3.83 tree — not against git:**

| AC | Verdict |
| --- | --- |
| AC-1 wake proportional to the request | **half false.** The `pub`/`pri` depth split is real, but the AC's clause "a partial wake never starts all chains" is **not true in the shipped code**: `_start()` calls `resumeAllNetworks()` → `chainService.resumeAllChainApis()`, which wakes every active chain on a partial wake as much as a full one. Chain-level granularity was never built. |
| AC-2 sleep on last disconnect | **met — but by older work.** `SLEEP_TIMEOUT` (60 s) and sleep-on-last-port-disconnect came from `c2046714a6` "Optimize MV3 lifecycle", shipped **1.2.1**, not from #4428. Caveat: `stopHeartbeat()` is wired only to keyring-lock, so the 20 s `chrome.storage.local` heartbeat keeps running after `sleep()` while the keyring stays unlocked. |
| AC-3 cached-first paint ≤300 ms + skeletons | **not met.** redux-persist `PersistGate` gives a cached-first paint (pre-existing, ~1.1.33), but there are only 9 `Skeleton` references in the whole UI and none on the Tokens home screen, and no ≤300 ms instrumentation exists. |
| AC-4 ZK-Asset/MantaPay removed after P2 | **not met.** `#4445` (P2) has **zero commits**. MantaPay is fully live: 334 hits across 30+ files, `initMantaPay()` / `enableMantaPay()` still in `State.ts`. The story's own verification command fails. |
| AC-5 cron/subscription as first-class services | **not met.** They are phase-driven now (a real #4428 gain) but still plain classes driving raw `setInterval`. `#4446` (P3) has **zero commits**. |
| AC-6 idempotent wake/sleep across eviction | **met, at 1.3.43 — and only after #4428's own regression was fixed.** #4428 set `isFullActive = true` on a `pri()` wake but never reset it on last-port-disconnect, so after the first sleep the next `pri()` skipped `_startFull()` and **Home / Earning failed to load**. Fixed by `a0bbe26a98` (`#4478`, release **1.3.43**) — a one-line reset. No lifecycle test exists (TASK-20.1.7 never built), which is exactly why it shipped. |

**Not `done`, not `backlog`:** P1 is in the product; P2 (#4445) and P3 (#4446) have no
code and nobody is working on them (last commit 2025-06). `in-progress` is the least
wrong status the enum offers — read it as **stalled after P1**.
