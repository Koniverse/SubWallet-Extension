## Goal

Finish the lifecycle refactor that stopped after P1. The MV3 background must wake only the
**chains** a request needs (not just the right *services*), go genuinely quiet when it
sleeps, paint from cache in ≤300 ms, and run its cron/subscription machinery as
lifecycle-aware services instead of raw `setInterval` timers — with a test harness that
fails the build when any of that regresses.

## Background

**Narrowed on 2026-07-14.** This story used to cover the whole **#4427 "Improve SubWallet
Lifecycle"** program — P1 (shipped 1.3.42), P2 and P3 (never started) — under a single
`in-progress` status that was false in both directions. **A story is the unit of status: if
two halves have different truths, it is not one story.** The shipped P1 wake-depth split is
now [US-20.7](US-20.7-mv3-wake-depth-split.md) (`done`, 1.3.42, `saltict`). What is left
here is **only work with no code behind it**. See [CONTEXT D100](../../CONTEXT.md).

Every issue this story now carries has **zero commits**:

- **[#4427](https://github.com/Koniverse/SubWallet-Extension/issues/4427)** — the program tracker. Its headline promise — *"a request that needs only a few chains never starts all of them"* — **was never built**. [AD-20](../../ARCHITECTURE.md#architecture-decisions) has been asserting it since 2025; the code calls `resumeAllNetworks()` on every wake.
- **[#4445](https://github.com/Koniverse/SubWallet-Extension/issues/4445) — P2** — remove the deprecated ZK Asset / MantaPay feature; refactor cron/subscription into services.
- **[#4446](https://github.com/Koniverse/SubWallet-Extension/issues/4446) — P3** — refine the data-processing architecture onto that path.

**`backlog`, not `in-progress`.** Nobody is working this: the last commit on any of these
issues is **none**, and the umbrella perf tracker `#4197` has sat at zero commits for about
a year. `assignee` is empty **on purpose** — an unowned story with a name on it is worse
than an unowned story, because it looks handled ([CONTEXT D100](../../CONTEXT.md)).

Defends [NFR-8](../../PRD.md#non-functional-requirements) (MV3 sleep/wake) and
[NFR-12](../../PRD.md#non-functional-requirements) (cached cold-start) on the background
lifecycle; owns no FR.

- [ ] **AC-1** — **Given** a partial (`pub(…)`) wake, **When** the worker starts, **Then** it starts **only the chains that request needs** — not every active chain. *(Today `_start()` calls `resumeAllNetworks()` → `resumeAllChainApis()`, waking every chain on a partial wake as much as a full one. The service-level split shipped; **chain-level granularity was never built**, and AD-20 has been claiming it since 2025.)* (#4427, AD-20)
- [ ] **AC-2** — **Given** the worker has slept, **When** the keyring is still unlocked, **Then** no avoidable timer keeps running. *(Today `stopHeartbeat()` is wired only to keyring-lock, so the 20 s `chrome.storage.local` heartbeat survives `sleep()`.)* (NFR-8, AD-20)
- [ ] **AC-3** — **Given** the cold-start path, **When** the popup opens, **Then** cached state is served **≤ 300 ms** with visible skeletons, then progressive refresh. *(Today redux-persist `PersistGate` gives a cached-first paint, but there are 9 `Skeleton` references in the whole UI — none on the Tokens home screen — and no ≤300 ms instrumentation exists.)* (NFR-12)
- [ ] **AC-4** — **Given** the deprecated ZK Asset / MantaPay feature, **When** the codebase is built after P2, **Then** it is removed and no longer referenced. *(Today: **334 hits across 30+ files**; `initMantaPay()` / `enableMantaPay()` still live in `State.ts`.)* (#4445)
- [ ] **AC-5** — **Given** the cron and subscription machinery, **When** P2/P3 land, **Then** they run as first-class lifecycle-aware services rather than plain classes driving raw `setInterval`. (#4445, #4446)
- [ ] **AC-6** — **Given** the lifecycle, **When** a change touches it, **Then** a phase-assertion harness (Init → Start → StartFull → Sleep) fails the build on a regression. *(**No lifecycle test exists** — which is precisely why #4428 shipped a regression that broke Home/Earning until #4478 fixed it.)*

## Dev notes

### Architecture constraints

- [AD-20](../../ARCHITECTURE.md#architecture-decisions) — the four-state lifecycle (Init → Start-Partially → Start-Fully → Sleep) with heartbeat is the contract this story materializes at runtime; transitions must stay idempotent across MV3 eviction.
- [AD-08](../../ARCHITECTURE.md#architecture-decisions) — the MV3 service-worker substrate is the runtime this lifecycle rides on; this story does NOT re-decide the substrate (owned by [EPIC-1](../epics/EPIC-1.md)).
- This story does NOT introduce new AD entries — it realizes AD-20 on top of AD-08.

### Cross-story dependencies

- Builds on [US-20.7](US-20.7-mv3-wake-depth-split.md) — the shipped P1. The *service*-level wake split is done; this story adds the *chain*-level one AD-20 has been claiming all along.
- Sibling [US-20.2](US-20.2-api-call-optimization.md) — the forward request-economy work, also `backlog`, also unowned.
- Required by [US-20.4](US-20.4-many-account-submit-performance.md) — a partial wake that starts only the needed chains is what keeps the submit path cheap. ([US-20.3](US-20.3-read-path-memory-budget.md) is **deprecated** — its requirement was retired, [D96](../../CONTEXT.md).)

### Performance budget

- Idle background: no avoidable polling / chain connect / API call while backgrounded; a partial wake never starts all chains.
- Cold-start first paint: cached state visible ≤ 300 ms on popup open (NFR-12).
- Measured by the lifecycle-phase assertion harness (TASK-20.1.7).
- Story PR description must explicitly confirm these budgets are met.

### What we explicitly did NOT do

- No change to the MV3 substrate decision (AD-08) — owned by [EPIC-1](../epics/EPIC-1.md); this story only phases the lifecycle on top of it.
- No new feature behaviour — the ZK Asset removal is a deletion, the cron/subscription refactor is structural; no user-facing capability is added.

### The split happened — and along a different line than this section predicted

This section used to say P2/P3 **may** split into their own stories when scheduled. What
actually forced the split was not scheduling — it was **status**. The story was carrying
shipped work (P1) and unstarted work (P2/P3) under one status, and no value of the enum was
true for both. `in-progress` was the least-wrong choice and it was still a lie: git shows
**nobody has touched this in a year**.

So the cut was made where the *truth* changed, not where the *phases* did:
[US-20.7](US-20.7-mv3-wake-depth-split.md) took the shipped P1, this story kept everything
with no code. **That is the general rule** — a story is the unit of status; if two halves
have different truths, it was never one story ([CONTEXT D100](../../CONTEXT.md), and the
same cut that produced [US-3.9](US-3.9-unified-to-solo-account-split.md) from US-3.5).

## Implementation notes

**Narrowed 2026-07-14 ([CONTEXT D100](../../CONTEXT.md)).** The audit below is the evidence
that forced the split; it is kept because it is *why* this story now looks the way it does.
The rows marked **met** describe work that has **left this story** for
[US-20.7](US-20.7-mv3-wake-depth-split.md); the rows marked **not met** are the ACs above.

**Audited 2026-07-13 (US-21.2 follow-up). Phase 1 shipped 13 months ago and the docs
never recorded it; phases 2 and 3 have zero commits.**

**Shipped: P1 — `#4428` "Optimize Lifecycle Management - P1", release 1.3.42**
(`f29da4271d`, `b6b6e2a99b`, `3cdd16937c`, author `saltict`). It introduced the
wake-*depth* split that is live at v1.3.83: `ActionHandler._onPortMessage` computes
`requireFullActive` from the message prefix (`pri(` / `mobile(` → `wakeup(true)`,
`pub(` → `wakeup(false)`), and only a full wake runs `_startFull()` (the 11 data
services: cron, subscription, history, price, balance, nft, earning, swap,
notification, openGov, multisig).

**AC-by-AC, against the v1.3.83 tree — not against git:**

| AC | Verdict |
| --- | --- |
| AC-1 wake proportional to the request | **half false.** The `pub`/`pri` depth split is real, but the AC's clause "a partial wake never starts all chains" is **not true in the shipped code**: `_start()` calls `resumeAllNetworks()` → `chainService.resumeAllChainApis()`, which wakes every active chain on a partial wake as much as a full one. Chain-level granularity was never built. |
| AC-2 sleep on last disconnect | **met — but by older work.** `SLEEP_TIMEOUT` (60 s) and sleep-on-last-port-disconnect came from `c2046714a6` "Optimize MV3 lifecycle", shipped **1.2.1**, not from #4428. Caveat: `stopHeartbeat()` is wired only to keyring-lock, so the 20 s `chrome.storage.local` heartbeat keeps running after `sleep()` while the keyring stays unlocked. |
| AC-3 cached-first paint ≤300 ms + skeletons | **not met.** redux-persist `PersistGate` gives a cached-first paint (pre-existing, ~1.1.33), but there are only 9 `Skeleton` references in the whole UI and none on the Tokens home screen, and no ≤300 ms instrumentation exists. |
| AC-4 ZK-Asset/MantaPay removed after P2 | **not met.** `#4445` (P2) has **zero commits**. MantaPay is fully live: 334 hits across 30+ files, `initMantaPay()` / `enableMantaPay()` still in `State.ts`. The story's own verification command fails. |
| AC-5 cron/subscription as first-class services | **not met.** They are phase-driven now (a real #4428 gain) but still plain classes driving raw `setInterval`. `#4446` (P3) has **zero commits**. |
| AC-6 idempotent wake/sleep across eviction | **met, at 1.3.43 — and only after #4428's own regression was fixed.** #4428 set `isFullActive = true` on a `pri()` wake but never reset it on last-port-disconnect, so after the first sleep the next `pri()` skipped `_startFull()` and **Home / Earning failed to load**. Fixed by `a0bbe26a98` (`#4478`, release **1.3.43**) — a one-line reset. No lifecycle test exists (TASK-20.1.7 never built), which is exactly why it shipped. |

**Resolution (2026-07-14):** the paragraph that used to sit here argued `in-progress` was
"the least wrong status the enum offers — read it as *stalled after P1*." That was a story
asking its readers to decode it. The enum was not the problem; **the story was drawn
wrong.** P1 left for [US-20.7](US-20.7-mv3-wake-depth-split.md) (`done`, 1.3.42, `saltict`),
this story is now **`backlog` and unowned** — which is simply true, and needs no decoding.
