## Goal

A TON dApp connects to SubWallet (via TON Connect), reads the user's TON address,
and requests signatures, so the TON ecosystem is reachable from the same wallet as
the rest of the user's chains.

## Background

> **Status: PARTIAL.** TON *account* support is shipped — SubWallet manages TON
> accounts, balances, transfers, and staking via the chain-service TON handlers
> (`@ton/core` + `@ton/ton` against TonCenter,
> `packages/extension-base/src/services/chain-service/handler/TonApi.ts`). What is
> **not yet implemented** is the **TON Connect dApp-connection page-provider**:
> `packages/extension-base/src/page/` ships `bitcoin/`, `cardano/`, `evm/`, and
> `substrate/` providers (registered in `page/index.ts`) but has **no `ton/`** —
> there is no injected TON Connect namespace, so a TON dApp cannot currently
> connect to SubWallet through the wallet. This story is therefore framed as
> **forward / in-progress**, not shipped. Tracking issue:
> [#3768 — Support connect dApp for TON account](https://github.com/Koniverse/SubWallet-Extension/issues/3768)
> (open).

TON dApps connect through a wallet provider injected as its own namespace
(TON Connect). The connector must expose connect/account reads and signature
requests for the selected wallet-contract account. SubWallet's TON model
([AD-13](../../ARCHITECTURE.md#architecture-decisions)) uses `@ton/core` + `@ton/ton`
against TonCenter, with a user-selectable wallet-contract version (v3r1 / v3r2 / v4 /
v5, default v5) — the connector should expose the account for the active contract
version so the dApp signs against the right address. The signing primitives this
relies on (TON keyring signing, TON confirmation handling via
`RequestConfirmationCompleteTon`) already exist from the account/transaction work;
the missing piece is the page-side injected provider that bridges a TON dApp to
them.

Connect is authorization-gated (US-10.7); signature requests route to the TON
request handler in `RequestService` ([AD-21](../../ARCHITECTURE.md#architecture-decisions))
behind the shared `PopupHandler`; signing runs in the background keyring (AD-04).

Materializes [FR-99](../../PRD.md#functional-requirements). The TON account/transaction layer is shipped;
the **TON Connect dApp-connection provider is forward work** (no `page/ton`,
issue [#3768](https://github.com/Koniverse/SubWallet-Extension/issues/3768)). Do
**not** mark this story retroactive/done on the basis of TON account support
alone.

## Acceptance criteria

- [ ] **AC-1** — **Given** a TON dApp, **When** it requests connection from an
  unauthorized origin, **Then** the request is surfaced for approval and the TON
  account is returned only on approval.
- [ ] **AC-2** — **Given** a connected dApp, **When** it reads the account, **Then**
  it receives the address for the user's active wallet-contract version (default v5).
- [ ] **AC-3** — **Given** a connected dApp, **When** it sends a signature request,
  **Then** the request routes to the TON request handler, is surfaced for approval,
  and is signed in the background keyring.
- [ ] **AC-4** — **Given** an unauthorized origin or a user rejection, **When** the
  request resolves, **Then** it rejects with a clear error and no address or
  signature leaks.

- [AD-21](../../ARCHITECTURE.md#architecture-decisions) — TON signature requests route to the TON request handler behind the shared `PopupHandler`.
- [AD-04](../../ARCHITECTURE.md#architecture-decisions) (owned by EPIC-3) — signing runs in the background keyring.

- Sibling [US-10.8](US-10.8-arbitrary-message-signing.md) — TON signing is the TON arm of arbitrary message signing.

account exposure, and signature routing for one chain family.
