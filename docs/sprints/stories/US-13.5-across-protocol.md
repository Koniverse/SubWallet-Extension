## Goal

A user can bridge assets across EVM chains via the Across protocol directly in
the wallet — seeing the fee and ETA before signing — so that fast intent-based
cross-chain transfers (including Across-backed Bridge-Swap-Bridge routing) are
available without leaving SubWallet.

## Background

Across is an intent-based ("optimistic") cross-chain bridge: a relayer fronts the
destination-chain funds and is reimbursed from the source escrow, giving fast
settlement relative to canonical bridges. It hangs off the bridge-hop interface
from [US-13.1](US-13.1-xcm-parachain-transfers.md) and the provider-handler
fixture from [US-13.2](US-13.2-snowbridge-eth-asset-hub.md), adding an
Across-specific handler behind the shared quote → submit → track lifecycle.
Across also backs the Bridge-Swap-Bridge routing pattern, but the *routing*
composition is owned by [EPIC-11](../epics/EPIC-11.md)
([US-11.8](US-11.8-cross-chain-swap-routing.md)) — this story owns the Across
**hop**, not the multi-hop route. Bridge construction follows the delegate
posture ([AD-18](../../ARCHITECTURE.md#architecture-decisions)); the provider is
reached through the SubWallet proxy
([AD-19](../../ARCHITECTURE.md#architecture-decisions), NFR-16).

Materializes [FR-130](../../PRD.md#functional-requirements). This story is **retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** an Across-supported asset and route, **When** the user
  requests a bridge, **Then** the wallet shows the relayer fee, estimated
  destination amount, and ETA before any signature.
- [x] **AC-2** — **Given** a confirmed Across bridge, **When** the user signs,
  **Then** the source-chain deposit is submitted and tracked, and the destination
  balance updates when the relayer fills the intent.
- [x] **AC-3** — **Given** an Across intent that the relayer has not yet filled
  (or that exceeds the expected fill window), **When** the user returns, **Then**
  it is shown as in-transit with recovery guidance — funds are never reported as
  lost.
- [x] **AC-4** — **Given** the Across provider, **When** the wallet calls it,
  **Then** the call goes through the SubWallet proxy and no provider key is in the
  bundle (AD-19 / NFR-16).

## Dev notes

### Architecture constraints

- [AD-18](../../ARCHITECTURE.md#architecture-decisions) — bridge construction delegated; the Across handler consumes the provider.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) — provider proxied; no key in the bundle (NFR-16).
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Builds on [US-13.1](US-13.1-xcm-parachain-transfers.md) (bridge-hop interface) and [US-13.2](US-13.2-snowbridge-eth-asset-hub.md) (provider-handler fixture).
- Required by [US-11.8](US-11.8-cross-chain-swap-routing.md) (EPIC-11) — Across-backed Bridge-Swap-Bridge routing composes this hop; the routing lives in EPIC-11, the hop here.

### Points justification

5 pts — §3a-bis "1 external system integration": one external bridge provider
(Across, intent/relayer model) behind the existing interface, including its
unfilled-intent recovery state. The Bridge-Swap-Bridge routing is out of scope
(EPIC-11), so this stays a single-provider integration at 5.
