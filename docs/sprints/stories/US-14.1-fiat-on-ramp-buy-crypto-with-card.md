## Goal

A user with an empty (or under-funded) wallet can buy crypto with a debit/credit
card without leaving SubWallet: they pick a token and the receiving account,
choose one of the aggregated regulated providers (Transak, Banxa, Coinbase Pay,
Meld), and are handed off to that provider's hosted KYC + payment flow with a
pre-built order URL carrying the right asset, network, and address. This is the
front door that turns "money in my bank" into on-chain assets, and it
establishes the provider-agnostic adapter contract every later provider reuses.

## Background

Funding the wallet is the first thing a user needs after onboarding, and doing
it in-product (vs "go buy on an exchange and transfer in") is the difference
between a wallet that gets used and one that sits empty. SubWallet does **not**
run KYC, capture the card payment, or settle — it aggregates several regulated
providers and hands the user to whichever one supports their token, region, and
payment method.

The integration is provider-agnostic by construction. Each provider is a
`CreateBuyOrderFunction` adapter (`packages/extension-koni-ui/src/utils/buy/`:
`transak.ts`, `banxa.ts`, `coinbase.ts`, `meld.ts`) keyed by `SupportService`,
and `BuyService`
(`packages/extension-base/src/services/buy-service/`) serves the available
provider/token roster from `buyServiceInfos` / `buyTokenConfigs` via
`fetchStaticData` ([AD-23](../../ARCHITECTURE.md#architecture-decisions),
[AD-25](../../ARCHITECTURE.md#architecture-decisions)). The most security-sensitive
adapter, Transak, mints its order URL **server-side** through the SubWallet
Services SDK (`subwalletApiSdk.transakApi.generateOrderUrl({ ..., action: 'BUY' })`,
[AD-24](../../ARCHITECTURE.md#architecture-decisions),
NFR-20) so the Transak partner secret never ships in the bundle
([AD-19](../../ARCHITECTURE.md#architecture-decisions), NFR-16). Meld, by
contrast, uses a provider-issued *publishable* wizard key (`publicKey`) that the
provider intends to live client-side — the documented exception to the
no-bundled-credential rule.

Materializes [FR-136](../../PRD.md#functional-requirements). This story is **Retroactive** — already
shipped. `commit` / `version_shipped` are backfilled during version
reconciliation.

## Acceptance criteria

- [x] **AC-1** — **Given** a supported token and a receiving account, **When** the
  user opens the Buy screen, selects a provider, and confirms, **Then** the
  provider's hosted flow opens with an order URL carrying the correct token,
  network, and receiving address.
- [x] **AC-2** — **Given** the Transak provider, **When** the order URL is built,
  **Then** it is minted by the Services SDK (`transakApi.generateOrderUrl`) with
  `action: 'BUY'` and **no** Transak secret credential is present anywhere in the
  shipped bundle.
- [x] **AC-3** — **Given** the provider/token roster served from `buyServiceInfos`
  / `buyTokenConfigs`, **When** a provider is marked `isSuspended` or a token is
  absent from a provider's config, **Then** that provider/token is not offered —
  driven by the static-data config, with no extension release required.
- [x] **AC-4** — **Given** a token the user holds is not supported by any active
  provider in their context, **When** they open the Buy screen, **Then** the
  unsupported token shows an empty/disabled state with guidance rather than an
  errored or blank screen (unhappy path: no eligible provider).
- [x] **AC-5** — **Given** the user's region or selected token is not serviceable
  by the chosen provider, **When** they reach the provider's hosted flow, **Then**
  region/KYC rejection is handled by the provider and the user is returned to the
  wallet without a partial or stuck order state (unhappy path: provider
  unavailable in region / KYC rejection).

## Dev notes

### Architecture constraints

- [AD-24](../../ARCHITECTURE.md#architecture-decisions) — the Transak order URL is built by the SubWallet Services SDK backend, not assembled in-client; this adapter consumes it.
- [AD-19](../../ARCHITECTURE.md#architecture-decisions) / NFR-16 — provider secret credentials are hidden behind the backend; only provider-issued *publishable* keys (Meld `publicKey`) may live client-side.
- [AD-23](../../ARCHITECTURE.md#architecture-decisions), [AD-25](../../ARCHITECTURE.md#architecture-decisions) — the provider/token roster is release-free config served via `fetchStaticData`.
- This story does NOT introduce new AD entries.

### Cross-story dependencies

- Required by [US-14.2](US-14.2-fiat-off-ramp-sell-crypto-for-fiat.md) — the off-ramp reuses the Transak adapter and the Services-SDK order-URL path established here (same call with `action: 'SELL'`).
- Required by [US-14.3](US-14.3-additional-on-ramp-providers.md) — new providers plug into the `CreateBuyOrderFunction` adapter map and `SupportService` roster set up here.

### What we explicitly did NOT do

- No in-product KYC, card capture, or settlement — SubWallet hands off to the provider; the wallet never takes fiat custody. Trigger to revisit: a licensed-provider/embedded-flow partnership.
- No swap or bridge path — crypto↔crypto is [EPIC-11](../epics/EPIC-11.md); cross-chain is [EPIC-13](../epics/EPIC-13.md).
