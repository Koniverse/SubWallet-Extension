## Goal

Finish the extension-side Ledger work that connect-and-discover
([US-16.5](US-16.5-ledger-connection-and-account-discovery.md)) and the Generic-app migration
([US-16.7](US-16.7-generic-ledger-app-migration-and-metadata.md)) left open: user-chosen derivation
paths, importing a Ledger account by JSON, automatic migration off the old apps, an honest message
when a feature is unavailable, and dropping the legacy `@polkadot/hw-ledger` connector.

## Status

> **📋 backlog — nothing here has shipped.** All five rows below are **open on the tracker**, four
> `In Backlog` and one `In Plan`, and no acceptance criterion is ticked. `assignee`, `commit`,
> `sprint` and `version_shipped` stay empty until the work starts.

## Scope

This is an **improvement story on shipped ones**. [US-16.5](US-16.5-ledger-connection-and-account-discovery.md)
and [US-16.7](US-16.7-generic-ledger-app-migration-and-metadata.md) are both `done`; these five
issues ask for things neither delivered. A `done` story may not carry an open row
([AGENTS.md](../../../AGENTS.md) rule 9), so they moved here on 2026-07-23 rather than sitting
under a settled table.

It materializes **no FR** — FR-144 is owned by
[US-16.1](US-16.1-ledger-hardware-wallet-signing.md).

**Split from [US-16.10](US-16.10-ledger-on-webapp-and-avail-space.md) by version space.** These five
ship in an extension release; the four there ship in the WebApp / Avail Space space, which has its
own numbering ([AGENTS.md](../../../AGENTS.md) rule 1b). Mixing them would put rows in one table
that can never share a `Shipped` value.

> **#3392 is the unfinished quarter of the Generic-app migration.** The app was integrated (#2453),
> the WebApp followed (#3256), and the Migration Polkadot app can attach accounts by hand (#3307,
> #3460) — all shipped in [US-16.7](US-16.7-generic-ledger-app-migration-and-metadata.md). What
> never shipped is doing it **for** the user. Every Ledger holder on an old per-chain app still
> migrates by hand.
>
> **#3866 is the prevention for the problem [US-16.8](US-16.8-ledger-asset-recovery-tool.md)
> repairs.** Assets strand on a derivative address the user cannot choose; a custom derivation path
> is how they would stop stranding. The repair tool shipped in 2024; the prevention is still `In
> Plan` — the only row here the board has moved past `In Backlog`.
>
> **#4647 is a dependency, not a feature**, and it is the one row a user would never notice. It
> follows the two `ledger-substrate-js` bumps in
> [US-16.7](US-16.7-generic-ledger-app-migration-and-metadata.md): once Zondax's library covers the
> surface, the legacy `@polkadot/hw-ledger` connector is dead weight that still has to be kept
> working.

## Acceptance criteria

- [ ] **AC-1** — A user can choose a derivation path when connecting a Ledger account (#3866), and can import a Ledger account by JSON exported from another wallet (#3319).
- [ ] **AC-2** — A Ledger account attached under an old per-chain app is migrated to the Generic app without manual re-attachment (#3392).
- [ ] **AC-3** — When a wallet feature is unavailable for a Ledger account, the UI says why rather than failing silently or showing a generic error (#1980).
- [ ] **AC-4** — `@polkadot/hw-ledger` is removed from the Ledger connector with no regression in the flows [US-16.5](US-16.5-ledger-connection-and-account-discovery.md) covers (#4647).
