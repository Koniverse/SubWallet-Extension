## A. Assignee backfill — name → GitHub-login identity resolution

**Current state.** `assignee` is required when a story ships (frontmatter-spec §3.1,
RULE-15) and must be a **GitHub login** (`saltict`), not a display name. Today all
169 stories carry an empty `assignee`; `STATUS.md` renders every row as `—`.

**The hard part — identity resolution.** The raw git history is *not* clean
name→login. `git shortlog -sne --all` shows the same human under multiple
identities, and bot accounts mixed in. The mapping table must resolve, at minimum:

| Same person, multiple identities (examples from `git shortlog`) | Resolution |
|---|---|
| `AnhMTV <maithachvietanh@gmail.com>` **and** `Peter Mai <maithachvietanh@gmail.com>` | one login (shared email) |
| `Nam Phạm <namphamc99@gmail.com>` **and** `nampc <namphamc99@gmail.com>` | one login |
| `lw` / `lw-cdm` / `leewong` across `cdmteck.com` / `konistudio.xyz` / `newayict.com` | one login, many work emails |
| `github-actions[bot]`, `Automation Bot <bot@subwallet.app>`, `Travis CI`, `Github Actions` | **exclude** — never an assignee |

**Recommendation.**

1. Produce a canonical **contributor map** — one row per person: `{ canonical GitHub
   login, display names[], git emails[] }`. Keep the sensitive/raw parts under
   `tmp/canonical/contributors-map.md` (gitignored); optionally publish a clean
   `docs/CONTRIBUTORS.md` with only login + role. Resolve logins via the GitHub API
   (PR authors, `users.noreply.github.com` handles already encode the login, e.g.
   `72647326+bluezdot`, `139972251+Thiendekaco`).
2. Define **assignee semantics** explicitly: the assignee is the **DRI / last
   substantive owner** of the user story — resolved from the story's `commit`(s) →
   PR author / last non-trivial committer of that surface — *not* every co-author and
   *never* a bot. `assignee` is a single scalar login; capture co-authors in the body
   (`## Implementation notes`) if useful.
3. Fill `assignee` in the **same backfill pass** as `commit`/`version_shipped` (§B),
   because they come from the same commit lookup.

## B. Backfill full history → complete, reconciled change logs

**Current state.** Every shipped story is a "Retroactive story — capability already
shipped; fill `commit` / `version_shipped` during version reconciliation" stub. The
body has an empty `## Changelog entry` and a dangling `**Commit**:` line. The root
`CHANGELOG.md` is the **authoritative main change log** (per-version, issue-numbered,
e.g. `#4987`).

**Gaps.**

- `version_shipped`, `commit` empty on shipped stories → violates RULE-16
  (`version_shipped` mandatory when `done`) and RULE-2 (`commit` never `pending`).
- No past **sprint files** — the Wave-6 I2/S6b "derive past sprints from git
  tags + CHANGELOG build dates" step was deferred and never done.
- **CHANGELOG lags reality by 3 releases**: HEAD is `Release 1.3.82` but CHANGELOG
  stops at `1.3.79`; `1.3.80 / 1.3.81 / 1.3.82` sections are missing. `VERSION` and
  `PRD.md` also still say `1.3.79`.

**Recommendation — one backfill pipeline per shipped story:**

1. **Trace**: story → FR(s) → shipped capability → the CHANGELOG version(s) + issue
   number(s) that delivered it → the git commit SHA(s) (`git log --grep`, tags).
2. **Fill frontmatter**: `version_shipped` (bare semver, no `v`), `commit` (SHA CSV
   for multi-commit stories), `assignee` (§A), flip `status: backlog → done`, set
   `sprint` to the reconstructed past sprint.
3. **Complete the body**: write the `## Changelog entry` (Added/Changed/Fixed) and the
   `**Commit**:` line, **worded consistently with the root CHANGELOG** so the two do
   not drift. The root CHANGELOG stays the single narrative; story entries are the
   per-story slice of it.
4. **Reconstruct past sprint files** (`sprint-YYYY-WNN.md`) from tag/build dates, then
   assign each done story to its sprint.
5. **Extend the main CHANGELOG** to cover `1.3.80–1.3.82`; bump `VERSION` and the PRD
   Version to `1.3.82`.
6. **Regenerate** `STATUS.md` via `npx koni-docs status` (never hand-edit — RULE-5).

**Reconciliation rule** (from the migration design): *codebase / git tag is reality* →
drives `done`/`version_shipped`; where CHANGELOG wording and code disagree, code wins.

## E. SubWallet-Mobile as a submodule?

**Goal.** Give the AI cross-repo context to develop Extension + Mobile + WebApp
together. `@subwallet/web-runner` is the **contract boundary** (mobile runs the
background in a WebView via web-runner).

**Options + trade-offs:**

| Option | Gives | Costs |
|---|---|---|
| **Git submodule** (SubWallet-Mobile pinned in this repo) | Full mobile source in-context | Pins a commit; heavier clone/CI; contributors must `--recurse-submodules`; repo-visibility/licensing must be verified |
| **Sibling checkout + `reference` pointer doc** | Lightweight cross-repo pointer | Context only when the sibling is checked out locally |
| **Docs-only contract mirror** (document web-runner ↔ Mobile bridge in ARCHITECTURE) | Most of the value at least weight | Not the live source |

**Recommendation.** Sequence it: **first** document the web-runner ↔ Mobile bridge
contract (the §D base epic) and add a `reference`-type pointer to the Mobile repo.
**Adopt the submodule only when** active cross-repo development starts *and* after
confirming the Mobile repo's visibility + CI can carry it. Don't lead with the
submodule — the contract doc unblocks the AI for less coupling.

## H. Skills / runbooks for recurring tasks

Recurring work should be captured as **installable skills** (same registry as the
koni-docs skill — `Koniverse/Koni-Skills`) or, at minimum, as `docs/` runbooks, so an
AI agent runs them deterministically instead of re-deriving the steps each time. Each
skill = a trigger + ordered steps + a **verification-commands** table (mirroring the
`## Verification commands` block already in every story). Two buckets:

**1. Fixed / mechanical tasks** (same steps every time):

- Run **unit tests** (per package / whole monorepo) and read results.
- Run **e2e tests** (Playwright — see §G) across extension / webapp surfaces.
- **Build & package** each surface: `yarn webpack:build:extension`, `yarn webapp:build`,
  `yarn web-runner:build`.
- **Release**: version bump + `CHANGELOG.md` section + `VERSION` + PRD Version, kept in
  sync (closes the §C-3 drift permanently).
- **koni-docs ops**: `backfill-fields`, `sync --dry-run`, `validate`, `status` — the
  §A/§B/§C loop, turned into a repeatable skill.
- **Chain-list / asset-metadata update** (online hot-update path, FR-3/FR-34).

**2. Domain / specialized tasks** (recurring but with judgment — the codebase already
has clean extension seams that make them skill-able):

- **Add a new chain** for an ecosystem (ChainService registry, FR-31/35, AD-02).
- **Add a swap provider** — new SwapService per-provider handler (FR-8).
- **Add an earning pool** — new `BasePoolHandler` subclass (FR-7, AD-22).
- **Add a dApp connector** — per-ecosystem inject + RequestService handler (FR-11, AD-21).
- **Add a hardware wallet / bridge route / i18n locale.**

**Recommendation.** Start with the two highest-frequency, highest-consistency-payoff
skills — **run-tests** (unit + e2e) and **release/version-sync** — then the koni-docs
backfill skill. Author the domain skills as the corresponding stories get worked, so
each skill is validated against a real run. A `docs/` skill/runbook index (trigger →
skill) makes them discoverable.

## I. Multi-repo standardization — governance model

**Current state.** The koni-docs canon lives **only** in SubWallet-Extension, but the
product is multi-repo. The scope splits into **two docs projects** by repo visibility:

**1. This project (public repos) — governed here, in this canon.**

| Repo | Role | Wallet link |
|---|---|---|
| **SubWallet-Extension** (this) | Extension + WebApp + web-runner surfaces | — |
| **SubWallet-Mobile** | Native shell hosting the background via web-runner | §E; NFR-17, FR-65 |
| **Chainlist** (`chain-list` / `chain-list-assets`) | Online chain-list + token/asset registry, hot-update source | FR-3, FR-34, NFR-21 |

These three are standardized **together** under one shared frontmatter-spec + ID
convention (the spec is already the *single source of truth for every koni-docs
consumer project*, so ID namespaces are shared for free). Cross-repo links use
`reference`-type pointers, not copied content. This confirms **SubWallet-Mobile is
in-scope here** — which strengthens §E: a first-class managed relationship (submodule
or a governed pointer), not an external afterthought.

**2. A separate docs project (private / auxiliary repos) — publishes docs to us.**

The load-bearing backend/support repos the wallet *depends on at runtime* —
`@subwallet-monorepos/subwallet-services-sdk` (NFR-20), the `api-cache` proxy
(NFR-21), SubWallet-Base, SubConnect V2, and similar side projects — are standardized
in **their own project** and **publish a documentation surface** that this project
**consumes** (again via `reference`-type pointers). They are *not* owned or copied into
this canon; we depend on their published docs, not their source.

**Recommendation.** Draw the boundary explicitly and keep it stable:

- **In this canon:** author/standardize SubWallet-Mobile + Chainlist docs alongside the
  Extension. Prioritize **Chainlist** first (it is the FR-3/FR-34 hot-update source the
  Extension already reads, so its schema is effectively undocumented product surface),
  then Mobile (§E).
- **Consumed, not owned:** add `reference` pointers to the private project's published
  docs for services-SDK / api-cache / SubWallet-Base / SubConnect V2 — so an AI agent
  can follow the link for context without those repos entering this project's scope.
- Confirm each repo's public/private classification before wiring the pointer (visibility
  decides which project owns it).

Treat §E and this §I-1 as one "public multi-repo" track; §I-2 is a dependency the other
project delivers.

## J. Research knowledge base — `docs/knowledge/`

**Motivation.** The koni-docs canon is deliberately **abstracted**: PRD holds business
requirements only (implementation banned by its own scope boundary), ARCHITECTURE holds
*decisions* not deep mechanics, and LESSONS holds incident learnings. That leaves **no
durable home for the deep research** an AI needs to author accurate FRs / ADs / stories —
how XCM v4 + ParaSpell routing actually works, Bittensor dTAO subnet mechanics, ERC-4337
account abstraction, CIP-30 / PSBT signing, the MV3 service-worker lifecycle quirks, the
web-runner WebView bridge, the Chainlist schema. Today that understanding is scattered,
trapped in `tmp/` (gitignored), or lost. Without it, an agent re-derives — and
hallucinates — the same background every time.

**Proposal.** Add a committed, public `docs/knowledge/` layer: curated, topic-organized
**research/reference** material that *feeds* the canon but is too detailed or exploratory
to live in it. It is the grounding an agent reads before writing an FR or an AD.

**How it differs from what exists:**

| Location | Holds | Lifecycle |
|---|---|---|
| `tmp/` | Raw sources (Notion, GH dumps) | Gitignored, throwaway |
| `docs/notes/` | Dated discussion + decisions (this file) | Point-in-time; graduates into canon |
| `docs/superpowers/` | Specs/plans for the migration *process* | Process artifacts |
| `LESSONS.md` | Incident learnings | Canon |
| **`docs/knowledge/`** | **Durable topic research that grounds the canon** | **Committed reference, kept fresh** |

**Convention.**

- Path: `docs/knowledge/<area>/<topic>.md`, organized by the taxonomy area (swap,
  earning, xcm-bridge, signing, mv3-lifecycle, web-surface, chain-list, …).
- Small frontmatter: `title`, `area`, `sources[]` (every claim cited — reuse the
  migration rule "no artifact without citation"), `last_verified: YYYY-MM-DD`, and
  `supports:` linking the FR / AD / US it grounds. The `last_verified` date matters —
  research goes stale (protocol upgrades, API changes); a reader must know how old it is.
- It is **`reference`-type** in koni-docs terms: it does **not** enter the 5-layer
  consistency check (it's not an FR/story), but FRs/ADs/stories may cite *into* it.
- Guard against a dumping ground: each doc must trace to a real need — it grounds a
  specific FR/AD/story, or answers a recurring question. No orphan research.

**Seed it with the highest-leverage topics** (hardest to re-derive, most hallucination-
prone): per-ecosystem address derivation + signing, XCM/ParaSpell routing, the
`BasePoolHandler` earning taxonomy, the swap-provider matrix, MV3 service-worker
lifecycle, the web-runner bridge (ties to §D), and the Chainlist schema (ties to §I-1).

**Tie-in.** Pairs with §H — a **"research-capture" skill** can standardize turning a
spike/investigation into a `docs/knowledge/` doc with citations and a `last_verified`
stamp, so research is captured consistently instead of ad hoc.
