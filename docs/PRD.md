## Executive Summary

SubWallet is a comprehensive non-custodial wallet for the Polkadot, Ethereum, TON, Cardano, and Bitcoin ecosystems, delivered as a browser extension, web application, and mobile app. It gives users a single interface to hold, transfer, swap, stake, earn, and govern assets across more than 200 networks without switching between separate tools per chain. The product is positioned as "a Web3 multiverse gateway through which users can enjoy multi-chain services with utmost ease and absolute security." A Unified Account model is a central architectural commitment: one seed phrase manages addresses across Substrate, EVM, Bitcoin, TON, and Cardano, eliminating the need to juggle separate wallets per ecosystem. The codebase is open-source and has undergone a security audit by Verichains. See [BRIEF.md](BRIEF.md) for the full product brief.

SubWallet's longer-term trajectory is to become the default multi-chain access layer for Web3 — a wallet that abstracts not just key management but the entire complexity of operating across divergent blockchain architectures. The chain-abstraction-as-a-service direction, if realized, would extend SubWallet's multi-chain logic beyond the wallet itself into a developer-facing SDK that other teams can use to build dApps or wallets with the same cross-chain breadth. Near-term priorities include completing the Unified Account model, maturing governance and multisig features, improving wallet performance, and expanding ecosystem coverage.

#### EPIC-2 — core-platform

- **Covers:** The distinctive engine modules that power the wallet and are reused across every feature — the Unified-Account keyring, ChainService, the EarningService pool-handler tree, the SwapService routing engine, the balance and fee engines, the RequestService approval queue and the transaction lifecycle engine. User-facing behaviour lives in the feature epics; this epic catalogues the shared engines underneath. (Generic plumbing and pure runtime mechanics — message bus, service-worker lifecycle, storage, events — are documented in ARCHITECTURE.)
- **Why it matters:** These engines are SubWallet's hardest-won, most-reused code; they rarely have a single user-facing issue yet decide multi-chain correctness, performance and the non-custodial security boundary, so listing them as FRs keeps them in scope for stories and reviews.

## Epics & User Stories

This project is **requirement-centric**: stories link back to the PRD through `prd_ref`, which holds `FR-N` **or** `NFR-N` — capability stories cite the FR they materialize, hardening / performance stories cite the NFR they defend (mapped 2026-07-13). The FR and NFR tables above are the authoritative requirement list. This index lists the 20 epics and points to each epic's Stories table (the authoritative US list) rather than enumerating the ~170 user-story rows here.

- **EPIC-1 — Build & Platform Infrastructure** — The runnable platform every other epic builds on: MV3 service-worker background, the Yarn 3 monorepo, and the online hot-update channel. — [stories](sprints/epics/EPIC-1.md#stories)
- **EPIC-2 — Core Platform Engines** — The distinctive engine modules that power every wallet feature (Unified-Account keyring, ChainService, EarningService, SwapService, balance/fee engines, RequestService, transaction lifecycle). — [stories](sprints/epics/EPIC-2.md#stories)
- **EPIC-3 — Account Creation & Management** — How keys enter (create/import), are organized & derived, and leave (export) the wallet, including the one-seed-many-chains Unified Account. — [stories](sprints/epics/EPIC-3.md#stories)
- **EPIC-4 — Network & Token Management** — Determines which chains and tokens the wallet can see: networks/RPCs, the 200+-network registry, per-ecosystem integration, plus the chain-abstraction roadmap. — [stories](sprints/epics/EPIC-4.md#stories)
- **EPIC-5 — Wallet Security** — Where the non-custodial promise is enforced: key isolation, master-password / lock policy, and the phishing / malicious-transaction / scam-address threat-defence surface. — [stories](sprints/epics/EPIC-5.md#stories)
- **EPIC-6 — UI & User Experience** — The one shared, dark-only responsive experience layer every feature renders through across popup, expand view, web app, and mobile web-runner, plus settings and display-currency controls. — [stories](sprints/epics/EPIC-6.md#stories)
- **EPIC-7 — Portfolio & Balances** — The wallet's daily-home screen: the aggregated portfolio, transferable-vs-locked breakdown, token auto-detection, and live price plus per-asset charts. — [stories](sprints/epics/EPIC-7.md#stories)
- **EPIC-8 — Transactions & Transfers** — The core money-movement surface: send/receive across all five ecosystems, fee/tip controls, on-chain history, the existential-deposit guard, and the signing experiences a user confirms. — [stories](sprints/epics/EPIC-8.md#stories)
- **EPIC-9 — NFT Management** — A single collectibles surface to see and send every NFT across Substrate, EVM and Bitcoin, however rendered (image / 3D / video / inscription), regardless of standard or chain. — [stories](sprints/epics/EPIC-9.md#stories)
- **EPIC-10 — dApp Connectivity** — Connect SubWallet to any dApp across every supported ecosystem and approve connections, signatures and transactions from one consistent connector + authorization surface. — [stories](sprints/epics/EPIC-10.md#stories)
- **EPIC-11 — In-Wallet Token Swap** — Let users exchange tokens without leaving the wallet by aggregating many swap providers behind one quote / route / execute surface. — [stories](sprints/epics/EPIC-11.md#stories)
- **EPIC-12 — Staking & Earning** — The wallet's yield surface: native/pool/collator/dApp staking, liquid staking and lending, the Bittensor dTAO model, earning-path simulation, and XCM-deposit routing. — [stories](sprints/epics/EPIC-12.md#stories)
- **EPIC-13 — Cross-Chain Bridge & XCM** — Moving a token from the chain it is on to the chain the user needs it on, one hop end to end: XCM, external bridges to/from Ethereum, the TAO↔Subtensor-EVM bridge, and the claim step. — [stories](sprints/epics/EPIC-13.md#stories)
- **EPIC-14 — Fiat On/Off-Ramp** — The fiat↔crypto bridge: how a user funds the wallet with a card (on-ramp) and cashes out (off-ramp) by routing to a regulated third-party provider through one buy/sell surface. — [stories](sprints/epics/EPIC-14.md#stories)
- **EPIC-15 — On-Chain Governance** — The wallet's on-chain voice: browse Polkadot OpenGov referenda, cast conviction votes that lock tokens, track and unlock them, and read legacy Governance V1 / Democracy referenda. — [stories](sprints/epics/EPIC-15.md#stories)
- **EPIC-16 — Hardware Wallet Signing** — Cold-storage signing for security-conscious users: the private key never enters the extension; the signing step completes on an external device. — [stories](sprints/epics/EPIC-16.md#stories)
- **EPIC-17 — Proxy Accounts** — Delegated signing authority on Substrate: a proxy account signs and submits with its own key on behalf of another account, scoped to a named Polkadot proxy-pallet type. — [stories](sprints/epics/EPIC-17.md#stories)
- **EPIC-18 — Multisig Accounts** — Multi-party custody on Substrate/Polkadot without a custodial backend: an M-of-N signatory set shares one deterministic address and a tx executes only at threshold on-chain approval. — [stories](sprints/epics/EPIC-18.md#stories)
- **EPIC-19 — Onboarding, Localization & Engagement** — The supporting surfaces wrapped around the core wallet: first-run onboarding, bundled multi-language UI, in-app growth campaigns, and the notification center. — [stories](sprints/epics/EPIC-19.md#stories)
- **EPIC-20 — Performance & Lifecycle** — The wallet's cross-cutting performance and lifecycle program: how fast the background wakes, how few API requests fan out, read-path memory, render performance, web-surface performance, and dependency hygiene. — [stories](sprints/epics/EPIC-20.md#stories)

## Non-Functional Requirements

| ID | Requirement | Category |
| ---- | ------------- | ---------- |
| NFR-1 | Non-custodial: private keys never leave the background service worker; no key material transmitted to any server | Security |
| NFR-2 | Key isolation: all signing and key operations are confined to the background process; UI and inject scripts hold no private key bytes | Security |
| NFR-3 | Keys encrypted at rest with AES-256-GCM (via browser-passworder) derived from the master password; raw key bytes never written to storage | Security |
| NFR-4 | Phishing protection auto-updates: @polkadot/phishing list and ChainPatrol API refreshed online; flagged sites show a blocking warning | Security |
| NFR-5 | dApp injection scope: unauthorized dApps see the provider object but receive rejection until the user explicitly approves per origin | Security |
| NFR-6 | Message-bus isolation: `pri(…)` privileged messages are only reachable from extension pages; `pub(…)` public messages are available to injected content scripts | Security |
| NFR-7 | Seed phrase never rendered in a textarea element (demonic-vulnerability mitigation for autocomplete/autofill leaks) | Security |
| NFR-8 | Manifest V3 compatibility: background runs as a MV3 service worker; WASM loaded via `wasm-unsafe-eval` CSP; state-persistence strategy handles service-worker shutdown/wake | Build / Compliance |
| NFR-9 | Firefox extension compatibility: Webpack output split into chunks below Firefox's ~4 MB per-file submission limit | Build / Compliance |
| NFR-10 | Open-source codebase: published on GitHub under an open-source license; community contributions accepted | Transparency |
| NFR-11 | Memory performance: the background must stay within a bounded RAM budget regardless of how many chains are enabled. **Target ≤72 MB — currently UNVERIFIED and UNMET-as-specified.** The figure comes from a 2022 measurement on the MV2 always-on background page, and the mechanism the old wording prescribed ("lightweight WsProvider-only connections for balance queries; full ApiPromise only for extrinsic construction") **is not implemented** — the substrate read path uses a full ApiPromise per chain (see AD-07). No memory probe exists in the repo. [US-20.3](sprints/stories/US-20.3-read-path-memory-budget.md) owns measuring the real number before anyone commits to a refactor. | Performance |
| NFR-12 | Cold-start and balance-load performance: background caches last-known state and serves it immediately on popup open; progressive refresh on reconnect with visible skeleton states | Performance |
| NFR-13 | English-canonical i18n: all user-facing strings authored in English first; translations are bundled per locale in the extension package (loaded from `locales/{lng}/translation.json`) — online runtime translation hot-update is planned (FR-4); background error messages are internationalized | Localization |
| NFR-14 | Offline / air-gap signing: hardware wallet flows (Ledger, Keystone, Polkadot Vault) complete without an internet connection for the signing step; signing payloads sent to device; private keys never enter the extension | Security / Availability |
| NFR-15 | Per-chain XCM route toggle: individual XCM routes can be disabled at runtime via the online chain-list update without requiring an extension release (rapid response to partner-chain incidents) | Availability |
| NFR-16 | Third-party API-key protection: keyed external providers — swap/bridge routing (Chainflip, KyberSwap, SimpleSwap, Uniswap), Bittensor (dTAO), Cardano/Blockfrost, ParaSpell XCM, and the Bitcoin mainnet indexer + Ordinals/Runes (btc-api) — are proxied through SubWallet-hosted backends (`external-services` via `fetchFromProxyService`; `btc-api` with a service-token header) so provider API keys are never shipped in the extension bundle (only Bitcoin testnet hits public Blockstream/mempool endpoints directly) | Security |
| NFR-17 | Cross-platform portability: the same `@subwallet/extension-base` background logic runs in the browser extension service worker, the standalone web app, and the mobile web-runner WebView context | Portability |
| NFR-18 | Deterministic account derivation: unified account addresses are reproduced from the same seed phrase and derivation path across fresh installations; no server dependency for key recovery | Account Model |
| NFR-19 | Dependency auditability: Yarn 3 lock file committed; all packages published to npm; no private registry dependencies for core wallet logic | Supply Chain |
| NFR-20 | SubWallet Services SDK backend: aggregated balance, fee, swap, and XCM/NFT data is fetched through the SubWallet Services SDK backend (`sw-services`, via `@subwallet-monorepos/subwallet-services-sdk`) instead of being computed entirely on-device — reducing per-chain RPC load and centralizing multi-chain data aggregation | Performance / Availability |
| NFR-21 | Data cache & CDN proxy layer: market data (token prices, exchange rates, EVM gas) is served through SubWallet cache proxies (`api-cache`, with `static-cache` / `chain-data` JSON fallback); online chain-list and token/asset metadata come from `static-data` / `chain-list-assets`; NFT media is fetched via the `ipfs-files` IPFS gateway — these front upstream providers to reduce rate-limit exposure, enable release-free data updates, and provide fallbacks when an upstream is unavailable | Performance / Availability |
